### 1. مفاهيم عامة (المدخل إلى البرمجة الغرضية التوجه)

#### النص الأصلي يقول:
> «ما هي مواصفات السيارة؟ اللون، الحجم، عدد المقاعد، سعة المحرك... كما تمتلك السيارة مجموعة من الوظائف: الوظيفة الأساسية النقل، وظائف أخرى: عند الضغط على الفرامل تتوقف السيارة...»
> «مجموعة المواصفات (الخصائص) والمهام معًا تشكل السيارة. البرمجة الغرضية التوجه هي مفهوم برمجي يعمل على إسقاط مفاهيم من الواقع على المفاهيم البرمجية. من الناحية التقنية تشابه الصفوف السجلات. تمتلك بالإضافة للحقول مهام ومحددات وصول. بواني وهوادم»

#### الشرح المبسّط:
تخيل سيارة حقيقية: لها **صفات** (اللون، الحجم، سعة المحرك) ولها **أفعال** يمكنها القيام بها (التوقف، التحرك للأمام). فكرة **البرمجة الغرضية التوجه (`OOP`)** تقول: بدل أن نتعامل مع البيانات والدوال بشكل منفصل كما في البرمجة الإجرائية، نجمعهما معًا في "قالب" واحد يُحاكي الأغراض الواقعية. هذا القالب هو **الصف (`class`)**.

**لماذا؟** لأن العقل البشري يفهم العالم على شكل أغراض لها صفات وأفعال، فبرمجة الحاسوب بنفس الطريقة تجعل الكود أقرب لتفكيرنا، وأسهل في القراءة والصيانة وإعادة الاستخدام.

#### 💡 التشبيه:
> السيارة تمتلك مواصفات (لون، حجم) ووظائف (تسير، تتوقف) مجتمعة في غرض واحد اسمه "سيارة".
> **وجه الشبه:** السيارة = غرض (`Object`) في البرمجة، مواصفاتها = الحقول (`Fields`)، وظائفها = التوابع (`Methods`).

#### نقطة مهمة ⚠️:
> الصف من الناحية التقنية **يشابه السجل (`struct`)** في احتوائه على حقول بيانات، لكنه يضيف عليه: **التوابع (Methods)**، و**محددات الوصول (Access Modifiers)**، و**البواني والهوادم (Constructors/Destructors)**.

### 2. الصفوف (`Classes`) — التعريف الأساسي

#### النص الأصلي يقول:
> «الصفوف هي البنية الأساسية في البرمجة الغرضية التوجه. في لغة `++C` يتم تعريف الصف بالشكل التالي... نلاحظ الاختلاف بين تعريف السجل وتعريف الغرض (قمنا بإضافة حقل يحتوي علامات الطالب في كل المواد في الكلية)»

#### الشرح المبسّط:
لنقارن بين طريقتين لتمثيل بيانات طالب:

```cpp
// The struct way: only data, no access control, no behavior
struct student{
    long id;           // university id
    string firstName;  // first name
    string lastName;    // last name
    float points;       // high school average
    int marks[62];       // marks array across all courses
};
```

```cpp
// The class way: data + behavior + access control
class student{
private:                        // hidden section: only the class itself can access
    long id;                    // university id
    string firstName;           // first name
    string lastName;            // last name
    float points;               // high school average
    int marks[62];              // marks array across all courses
public:                         // visible section: anyone can access
    void changeMarks(int i,int v); // method to change one mark safely
};
```

**لماذا؟** الفرق الجوهري هو أن `struct` مجرد "صندوق بيانات" مفتوح بالكامل، بينما `class` يضيف **حماية (`private`)** و**سلوك (`method`)** يسمح بالتحكم في كيفية تعديل البيانات.

#### شرح كل سطر:
1. `class student{` → بداية تعريف الصف — نعطيه اسمًا (`student`) يصبح نوع بيانات جديد.
2. `private:` → كل ما يلي هذا الوسم مخفي عن الخارج حتى إشعار آخر.
3. `long id;` ... `int marks[62];` → الحقول (الخصائص) الخاصة بالطالب.
4. `public:` → كل ما يلي هذا الوسم متاح للجميع.
5. `void changeMarks(int i,int v);` → **تصريح** فقط عن تابع (لا كود تنفيذي هنا)، سيُكتب جسمه لاحقًا خارج الصف.

#### 💡 التشبيه:
> الحقول الخاصة `private` أشبه بخزانة أدوية داخل المستشفى، لا يدخلها إلا الأطباء المسؤولون عنها (توابع الصف)، بينما الاستقبال العام (`public`) يدخله أي زائر.
> **وجه الشبه:** خزانة الأدوية = `private fields`، الاستقبال = `public methods`.

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** لماذا لا نجعل كل الحقول `public` مباشرة ونستغني عن التوابع؟
> **لماذا هذا مهم؟** لأن جعل الحقول `public` يفتح الباب أمام أي جزء من البرنامج لوضع قيم خاطئة (مثل علامة سالبة) دون أي تحقق، وهذا بالضبط ما يحله `private` مع توابع محكومة.

### 3. محددات الوصول (`Access Modifiers`)

#### النص الأصلي يقول:
> «تشجع البرمجة الغرضية التوجه على مبدأ التغليف وإخفاء التفاصيل. بمعنى آخر لا يهمنا كيف يتم يعمل الغرض على تخزين معلومات الطالب، المهم أننا نحصل على المعلومات التي نريدها وقت ما نريدها بالشكل الصحيح. المحدد `private` يعني أن التوابع والحقول الموجودة هنا خاصة، لا أحد قادر على الوصول إليها سوى التوابع ضمن الصف نفسه والأصدقاء (لاحقًا). المحدد `public` يعني أن كل الكرة الأرضية تمتلك القدرة على الوصول لهذه التوابع والحقول»

#### الشرح المبسّط:
هذا المفهوم يسمى **التغليف (`Encapsulation`)**، وهو أحد أهم أعمدة `OOP` الأربعة. الفكرة: أخفِ التفاصيل الداخلية (كيف تُخزَّن البيانات، كيف تُحسب) واعرض للمستخدم فقط "الواجهة" التي يحتاجها.

**لماذا؟** تخيل أنك تستخدم جهاز تلفاز: لا يهمك كيف تعمل الدوائر الكهربائية بالداخل، يهمك فقط زر التشغيل وزر تغيير القناة. لو تغيّرت الدوائر الداخلية (تحسين تقني) لن يتأثر استخدامك للتلفاز طالما الأزرار الخارجية بقيت كما هي. هذا بالضبط ما يوفره `private`/`public`: يمكنك تغيير طريقة تخزين البيانات داخليًا دون أن يتأثر أي كود يستخدم الصف من الخارج.

| محدد الوصول | من يستطيع الوصول | مثال |
| --- | --- | --- |
| `private` | التوابع ضمن الصف نفسه فقط (والأصدقاء لاحقًا) | الحقول `id`, `firstName`, `marks` |
| `public` | أي جزء من البرنامج (كل الكرة الأرضية) | التابع `changeMarks` |

#### ⚖️ المقايضة: `private` مقابل `public`
| | `private` | `public` |
| --- | --- | --- |
| المزايا | حماية البيانات من التعديل الخاطئ، إخفاء التفاصيل | سهولة الوصول المباشر بدون توابع وسيطة |
| العيوب | يتطلب كتابة توابع `getters`/`setters` إضافية | لا يوجد أي تحقق أو حماية على القيم |
| متى تختاره | الحقول التي تحتاج حماية أو تحقق من الصحة | التوابع التي يُفترض أن يستخدمها المستخدم مباشرة |

### 4. البناء والهدم (`Construction & Destruction`)

#### 4.1. مفهوم البناء والهدم

#### النص الأصلي يقول:
> «يمكننا النظر إلى الصف على أنه البنية التي تحتوي الحقول والتوابع `structure`. الغرض هو الصف عندما يتم بناؤه، أي البنية والقيم `structure + value`. عملية البناء تعنى بإنشاء الغرض وإسناد قيم أولية `initial values` للحقول الموجودة فيه. الهدم هي مجموعة العمليات التي يتم تنفيذها عند الانتهاء من الغرض وحذفه من الذاكرة»

#### الشرح المبسّط:
- **الصف (`class`)** = مجرد "مخطط" أو "قالب" (خريطة بناء منزل).
- **الغرض (`object`)** = المنزل الفعلي المبني حسب المخطط، وله قيم فعلية.
- **البناء (`Construction`)** = عملية "بناء" المنزل ووضع أثاث ابتدائي فيه (تهيئة القيم).
- **الهدم (`Destruction`)** = عملية "هدم" المنزل عند الانتهاء منه وتفريغه من الذاكرة.

**لماذا نحتاج بناءً صريحًا؟** لأن أي متحول جديد قد يحمل "قيمة قمامة" (Garbage Value) في الذاكرة إن لم يُهيَّأ، والبناء هو الطريقة الآمنة لضمان أن كل غرض يبدأ بقيم منطقية.

#### 💡 التشبيه:
> المخطط الهندسي لمنزل هو الصف، والمنزل الفعلي المبني هو الغرض.
> **وجه الشبه:** المخطط = `class` (تعريف نظري)، المنزل المبني = `object` (نسخة فعلية بقيم حقيقية).

#### 4.2. الباني (`Constructor`)

#### النص الأصلي يقول:
> «الباني هو تابع خاص ضمن الصف يمتلك نفس اسم الصف. في لغة `++C` لدينا 3 أنواع من البواني: الباني الافتراضي (الذي لا يحتاج وسطاء)، الباني ذو الوسطاء، الباني الناسخ»

#### الشرح المبسّط:
الباني هو تابع خاص جدًا: **اسمه مطابق تمامًا لاسم الصف**، ولا يمتلك نوع إرجاع (ولا حتى `void`). يُستدعى **تلقائيًا** عند إنشاء أي غرض جديد.

#### ⚙️ الخطوات / الخوارزمية: أنواع البواني في `++C`

> ما هدف هذا التصنيف؟ لتغطية كل الحالات الممكنة لإنشاء غرض: بدون بيانات، ببيانات جاهزة، أو نسخًا عن غرض موجود.

```algorithm
1 | الباني الافتراضي | Default Constructor | لا يحتاج وسطاء، يُعطي قيمًا ابتدائية ثابتة (0, "", ...)
2 | الباني ذو الوسطاء | Parameterized Constructor | يستقبل وسطاء ليبني الغرض بقيم محددة عند الإنشاء
3 | الباني الناسخ | Copy Constructor | ينسخ قيم غرض موجود مسبقًا إلى غرض جديد
```

##### 4.2.1. الباني الافتراضي (`Default Constructor`)

```cpp
class student{
    // ... fields omitted for brevity
public:
    student(); // constructor declaration — same name as class, no return type
};

// definition outside the class using the scope resolution operator ::
student::student(){
    id=0;                       // initialize id to 0
    firstName="";                // initialize first name to empty string
    lastName="";                 // initialize last name to empty string
    for(int i=0;i<62;i++)        // loop over all 62 marks
        marks[i]=0;               // initialize each mark to 0
}
```

**استدعاء الباني:**
```cpp
student x();   // WARNING: this is actually a function declaration, not an object! (Most Vexing Parse)
student y;     // Correct: calls the default constructor with no parentheses
```

#### مهم للامتحان ⚠️:
> عند عدم تمرير أي وسيط عند تعريف المتحول، نضع اسم المتحول فقط **بدون أي قوسين** لاستدعاء الباني الافتراضي (`student y;`). وضع أقواس فارغة (`student x();`) في سياق تعريف متحول محلي يُفسَّر أحيانًا كتصريح دالة، وهذا خطأ شائع يجب الانتباه له.

##### 4.2.2. الباني ذو الوسطاء (`Parameterized Constructor`)

```cpp
class student{
    // ... fields omitted for brevity
public:
    student(long,string,string,float); // declaration with parameter types
};

student::student(long id,string fisrtName,string lastname,float ponits){
    this->id=id;                      // assign parameter id to the field id
    this->firstName=firstName;        // assign parameter firstName to the field
    this->lastName=lastName;          // assign parameter lastName to the field
    this->points=points;              // assign parameter points to the field
    for(int i=0;i<62;i++)             // loop over marks array
        marks[i]=0;                    // initialize each mark to 0
}

student x(1,"a","b",232);   // calling the parameterized constructor
student y(2,"za","vb",432);
```

#### النص الأصلي يقول:
> «السؤال المهم: هل التعبير `id=id` صحيح؟ قواعديًا نعم، منطقيًا لا. لنتذكر الملاحظة أن مجال رؤية التابع العضو في صف يتضمن التابع والصف الذي هو عضو فيه. في حالتنا هذه لدينا متحولين باسم `id` الأول وسيط للتابع والثاني حقل ضمن الصف. أيهما أقرب للتابع؟ الوسيط أم الحقل؟ الوسيط طبعًا. لذلك يعمل المترجم على اعتبار `id` تشير إلى الوسيط وليس للحقل. ما الحل؟ لدينا حلان، الأول إعادة تسمية الوسيط والثاني استخدام المؤشر `this`»

#### الشرح المبسّط:
هنا تكمن نقطة دقيقة جدًا وشائعة الخطأ عند المبتدئين: عندما يتشابه اسم **الوسيط (Parameter)** مع اسم **الحقل (Field)**، فإن قاعدة "الأقرب أولًا" في مجال الرؤية تجعل المترجم يفسّر `id=id` على أنه **الوسيط يساوي نفسه** (لا فائدة!) وليس "الحقل = الوسيط".

**لماذا؟** لأن مجال رؤية التابع العضو يشمل: (1) المتحولات المعرّفة داخل التابع نفسه، و(2) حقول الصف. والقاعدة العامة في البرمجة هي أن الأقرب مكانيًا (الوسيط المحلي) يُفضَّل دومًا على الأبعد (حقل الصف) عند تشابه الأسماء.

**الحل باستخدام المؤشر `this`:**
```cpp
student::student(long id,string fisrtName,string lastname,float ponits){
    this->id=id;                 // this->id refers to the CLASS FIELD, id refers to the PARAMETER
    this->firstName=firstName;
    this->lastName=lastName;
    this->points=points;
    for(int i=0;i<62;i++)
        marks[i]=0;               // no ambiguity here since no local variable named "marks" exists
}
```

#### مهم للامتحان ⚠️:
> `this` هو **مؤشر خاص** ضمن لغة `++C` يشير إلى **الغرض الذي استدعى هذا التابع**. في الاستدعاء `student x(1,"a","b",232);` تكون قيمة `this` داخل الباني تساوي `&x`، وفي `student y(2,"za","vb",432);` تكون `this` تساوي `&y`.

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** لماذا لم نحتج لاستخدام `this` مع المصفوفة `marks` رغم أنها أيضًا حقل في الصف؟
> **لماذا هذا مهم؟** لأنه لا يوجد أي متحول محلي (وسيط) باسم `marks` ضمن هذا الباني، فلا يوجد أي تعارض في التسمية، وبالتالي لسنا ملزمين باستخدام `this` هنا (رغم أنه ما زال يعمل لو استُخدم).

##### 4.2.3. الباني الناسخ (`Copy Constructor`)

#### النص الأصلي يقول:
> «نستخدم هذا الباني لنسخ القيم من غرض لآخر. نقوم بنسخ القيم. مررنا الوسيط `c` بالمرجع لتجنب استنساخ الغرض في الذاكرة»

```cpp
class student{
    // ... fields omitted for brevity
public:
    student(student&); // copy constructor: takes a reference to another student
};

student::student(student& c){
    id=c.id;                      // copy id from c
    firstName=c.firstName;        // copy first name from c
    lastName=c.lastName;          // copy last name from c
    points=c.points;              // copy points from c
    for(int i=0;i<62;i++)         // loop over marks
        marks[i]=c.marks[i];       // copy each mark from c
}
```

#### نقطة مهمة ⚠️:
> الوسيط `c` يُمرَّر **بالمرجع (`&`)** وليس بالقيمة. لو مررناه بالقيمة، سيقوم `++C` باستنساخ الغرض `c` بالكامل لإنشاء نسخة مؤقتة منه فقط لتمريره للتابع — وهذا يستدعي الباني الناسخ نفسه من جديد! أي حلقة استدعاء لا نهائية (`Infinite Recursion`). المرجع يتجنب هذا لأنه لا ينسخ الغرض، بل يشير إليه مباشرة.

#### 4.3. الهادم (`Destructor`)

#### النص الأصلي يقول:
> «يُستخدم الهادم لتنظيم بعض العمليات أثناء حذف الغرض من الذاكرة. يسمح لنا ذلك بإغلاق الملفات وحذف المؤشرات والمصفوفات الديناميكية من الذاكرة في المراحل الأخيرة من عمر الغرض في الذاكرة. لا يمتلك الهادم أي وسيط، وله نفس اسم الصف يسبقه الرمز `~`. إذا لم يتم تعريف أي هادم يتم إضافة هادم فارغ. إذا لم يتم تعريف أي باني يتم إضافة باني فارغ بدون وسطاء»

#### الشرح المبسّط:
الهادم هو "الوداع الأخير" للغرض قبل أن يُمحى من الذاكرة. يُستخدم لتنظيف أي موارد استخدمها الغرض أثناء حياته (فتح ملف، حجز ذاكرة ديناميكية، إلخ).

```cpp
class student{
    // ... fields omitted for brevity
public:
    ~student(); // destructor: same name as class prefixed with ~, no parameters
};
```

#### نقطة مهمة ⚠️:
> إذا لم يعرّف المبرمج أي باني، يضيف المترجم **باني افتراضي فارغ تلقائيًا**. وبالمثل، إذا لم يعرّف أي هادم، يضيف المترجم **هادم فارغ تلقائيًا**. هذا يعني أن كل صف يمتلك دائمًا باني وهادم، سواء كتبهما المبرمج أم لا.

##### ترتيب استدعاء البناء والهدم

#### النص الأصلي يقول:
> «ترتيب الاستدعاء عند تعريف الغرض محليًا: يتم استدعاء الباني دومًا بصفة أول تابع في الصف. يستدعى الهادم من قبل المترجم في آخر سطر من مجال الرؤية الذي تم تعريف الغرض فيه. إذا لم يتم إنهاء البرنامج بشكل سليم لن يستدعى الهادم. ترتيب الاستدعاء للأغراض العامة `Global`: يتم بناء الأغراض المعرّفة بصفة متحولات عامة `Global` قبل استدعاء أي تابع في البرنامج... ترتيب الاستدعاء للأغراض الساكنة `static`: يتم استدعاء الباني لمرة واحدة فقط عند الوصول لسطر تعريف المتحول الساكن للمرة الأولى...»

#### ⚙️ الخطوات / الخوارزمية: ترتيب البناء والهدم حسب نوع الغرض

> ما هدف هذا الترتيب؟ لفهم متى بالضبط تُستدعى البواني والهوادم تلقائيًا في برنامج `++C` حقيقي، وهذا أساسي لتتبع تنفيذ البرنامج بشكل صحيح.

```algorithm
1 | الأغراض المحلية | Local Scope | الباني يُستدعى عند التعريف، الهادم يُستدعى في آخر سطر من مجال الرؤية (نهاية الكتلة { })
2 | الأغراض العامة Global | Global Scope | الباني يُستدعى قبل أي تابع في البرنامج (حتى قبل main)، الهادم يُستدعى في نهاية تنفيذ البرنامج
3 | الأغراض الساكنة static | Static Scope | الباني يُستدعى مرة واحدة فقط عند أول وصول لسطر التعريف، الهادم في نهاية تنفيذ البرنامج
```

#### نقاط التنفيذ:
- إذا لم ينتهِ البرنامج بشكل سليم (تعطّل، إغلاق قسري)، **لن يُستدعى الهادم**.
- الأغراض المحلية داخل حلقة `for` مثلًا تُبنى وتُهدم في **كل تكرار** من الحلقة، لأن مجال رؤيتها ينتهي في نهاية كل تكرار.

#### 🔍 تتبع التنفيذ: مثال البناء والهدم في `main`

**المدخل:**
```cpp
student gl(10,"first","First",22); // global object

int main(){
    cout<<"main starts"<<endl;
    student mn(11,"second","Second",33);   // local object in main
    cout<<"the loop starts"<<endl;
    for (int i=0;i<5;i++)
        student mm(12+i,"third","Thrid",44); // local object recreated each loop iteration
    cout<<"the loop ends"<<endl;
    cout<<"main ending"<<endl;
    return 0;
}
```

| الخطوة | العملية | الحالة |
| --- | --- | --- |
| 1 | بناء الغرض العام `gl` (قبل `main`) | Building student with Id: 10 |
| 2 | طباعة "main starts" | main starts |
| 3 | بناء `mn` داخل `main` | Building student with Id: 11 |
| 4 | طباعة "the loop starts" | the loop starts |
| 5 | كل تكرار من الحلقة: بناء `mm` ثم هدمه فورًا (نهاية جسم الحلقة) | Building/Destroying Id: 12، ثم 13، ثم 14، ثم 15، ثم 16 |
| 6 | طباعة "the loop ends" و "main ending" | the loop ends / main ending |
| 7 | نهاية `main`: هدم `mn` ثم `gl` (بترتيب عكسي عن البناء) | Destroying Id: 11 ثم Destroying Id: 10 |

**النتيجة:**
```
Building student with Id: 10
main starts
Building student with Id: 11
the loop starts
Building student with Id: 12
Destroying student with Id: 12
Building student with Id: 13
Destroying student with Id: 13
Building student with Id: 14
Destroying student with Id: 14
Building student with Id: 15
Destroying student with Id: 15
Building student with Id: 16
Destroying student with Id: 16
the loop ends
main ending
Destroying student with Id: 11
Destroying student with Id: 10
```

#### الدرس المستفاد:
> الأغراض تُهدم بترتيب **عكسي** عن ترتيب بنائها ضمن نفس مجال الرؤية (آخر من دخل، أول من يخرج — مثل المكدس `Stack`). الغرض العام `gl` يُبنى أولًا ويُهدم أخيرًا لأن حياته تغطي كامل عمر البرنامج.

### 5. `Getters` و`Setters`

#### النص الأصلي يقول:
> «كيف يمكن لنا أن نقرأ قيم الحقول بعد البناء في الغرض إذا كانت هذه الحقول `private`؟ الحل بكل بساطة باستخدام توابع تعرف باسم `getters`. مجموعة من التوابع تعطي القيم الخاصة بالحقول الموجودة ضمن الصف. سمية بهذا الاسم لأنها عادة تبدأ بالكلمة `get`»
> «كيف يمكن لنا أن نعدل قيم الحقول بعد البناء في الغرض إذا كانت هذه الحقول `private`؟ الحل بكل بساطة باستخدام توابع تعرف باسم `setters`»

#### الشرح المبسّط:
بما أن الحقول `private` مخفية عن الخارج، نحتاج "أبوابًا رسمية" للوصول إليها من الخارج بشكل آمن ومضبوط:
- **`Getter`**: تابع يُعيد (`return`) قيمة حقل معيّن — للقراءة فقط.
- **`Setter`**: تابع يستقبل قيمة جديدة ويضعها في الحقل — للتعديل، ويمكنه إضافة تحقق من الصحة قبل التعديل.

```cpp
class student{
private:
    long id;
    string firstName;
    string lastName;
    float points;
public:
    // Getters: read-only access to private fields
    long getId(){return id;}
    string getFirstName(){return firstName;}
    string getLastName(){return lastName;}
    float getPoints(){return points;}

    // Setters: controlled write access to private fields
    long setId(long v){id=v;}
    string setFirstName(string firstName){student::firstName=firstName;}
    string setLastName(string firstName){ this->lastName=lastName;}
    float setPoints(float x){ points=x;}
};
```

#### 💡 التشبيه:
> `Getter`/`Setter` أشبه بموظف الاستقبال في بنك: لا يمكنك أنت الدخول إلى خزنة الأموال مباشرة (الحقل `private`)، لكن تطلب من الموظف (`getter`) أن يخبرك برصيدك، أو تطلب منه (`setter`) أن يودع مبلغًا معينًا — وهو يتحقق من صحة العملية قبل تنفيذها.
> **وجه الشبه:** الخزنة = الحقل `private`، موظف الاستقبال = تابع `getter`/`setter`.

#### ملاحظة:
> اسم `getter`/`setter` عادة يبدأ بكلمة `get`/`set` متبوعة باسم الحقل بصيغة `PascalCase` (مثل `getFirstName`)، وهذا اصطلاح شائع (Convention) وليس قاعدة إلزامية في اللغة نفسها.

### 6. فوائد الصفوف

#### النص الأصلي يقول:
> «تبسيط البرمجة: من خلال دمج المنطق (التوابع) مع البيانات (الحقول) ضمن كتلة واحدة تصبح عملية البرمجة أسهل وأكثر وضوحًا. الوجهات: لا نقصد هنا الواجهات البرمجية، نقصد الواجهات البرمجية `Programming Interface`. ماذا يعني هذا؟ لا يهم كيف يعمل الغرض في الداخل، المهم أننا نحصل على النتيجة المطلوبة. إعادة استخدام البرمجيات `Software Reuse`: إمكانية إعادة استخدام الكود المكتوب من خلال: التركيب `Composition`: استخدام غرض ضمن غرض. الوراثة `Inheritance`: صف جديد يرث صف قديم»

#### الشرح المبسّط:
ثلاث فوائد جوهرية للصفوف:
1. **تبسيط البرمجة** — البيانات والسلوك معًا في مكان واحد بدل تشتتهما.
2. **الواجهات (`Interfaces`)** — يهمنا فقط "ماذا يفعل" الغرض، لا "كيف" ينفذه داخليًا.
3. **إعادة استخدام البرمجيات** — عبر **التركيب (`Composition`)** أو **الوراثة (`Inheritance`)** (ستُشرح تفصيليًا في محاضرة قادمة).

#### 💡 التشبيه:
> عند قيادة السيارة، لا يهمك كيف تعمل صمامات المحرك بالضبط، يهمك فقط أن الدواسة تُحرّك السيارة.
> **وجه الشبه:** الدواسة = الواجهة العامة (`public interface`)، صمامات المحرك = التفاصيل الداخلية المخفية (`private implementation`).

### 7. مجال الرؤية في الصف

#### النص الأصلي يقول:
> «أي تابع أو عنصر ضمن الصف قادر على رؤية محتويات الصف كاملًا. أي تابع ضمن الصف قادر على رؤية العناصر (المتحولات\الحقول، التوابع) ضمن هذا الصف. في حال وجد ضمن التابع متحول يحمل نفس اسم متحول\حقل ضمن الصف فوفقًا لقواعد مجال الرؤية (الأقرب أولًا) سيرى التابع المتحول الخاص به وليس حقل الصف. لحل هذه المشكلة نستخدم إما المتحول `this` أو `ClassName::VariableName`. للوصول لعناصر الصف نتعامل معه بالشكل التالي:»

#### الشرح المبسّط:
هذه إعادة تأكيد لنفس القاعدة التي ظهرت في الباني ذو الوسطاء (القسم 4.2.2)، لكن معممة على **أي تابع** ضمن أي صف: قاعدة **"الأقرب أولًا"** تعني أن أي متحول محلي (وسيط أو متحول داخل التابع) يُفضَّل دائمًا على حقل الصف عند تشابه الاسم.

**طرق الوصول لعناصر الصف حسب نوع المتحول:**
```cpp
Class1 c;                  // c is a regular object (not a pointer)
Class1 *c1=new Class1();   // c1 is a pointer to a dynamically allocated object

c.function1();       // dot operator: used with a regular object
c1->function1();     // arrow operator: used with a pointer to object
(*c1).function1();   // dereference then dot: equivalent to c1->function1()
```

| طريقة الوصول | متى تُستخدم | المعادل |
| --- | --- | --- |
| `objectName.method()` | عندما يكون المتحول غرضًا عاديًا (وليس مؤشرًا) | — |
| `pointerName->method()` | عندما يكون المتحول مؤشرًا على غرض | `(*pointerName).method()` |
| `(*pointerName).method()` | فك مرجعية المؤشر ثم الوصول بالنقطة | `pointerName->method()` |

#### مهم للامتحان ⚠️:
> `c1->function1()` و `(*c1).function1()` **متكافئان تمامًا**؛ الأول اختصار للثاني وهو الأكثر استخدامًا عمليًا.

### 8. فصل الكود عن الواجهة (`Header/Implementation Separation`)

#### النص الأصلي يقول:
> «ماذا نعنيه عندما نقوم بكتابة الكود `#include<iostream>`؟ أي أننا نلجأ لقراءة الملف `iostream.h` لمعرفة محتوياته. هذا الملف يسمى بالملف الرئيسي ويحتوي فقط التعاريف دون أي منطق برمجي. المنطق البرمجي يكون موجود ضمن ملف `cpp` مرافق له بنفس الاسم `iostream.cpp`. ما الغرض من هذه العملية؟ عندما ترغب بتوزيع الكود الذي تعبت وأنت تكتبه، كي تتجنب سرقته والتلاعب به تلجأ لتحويله إلى ملفات تنفيذية (صيغة `dll`). تقوم بإرسال الملفات الرئيسية `headers` مع الملفات التنفيذية `dll` إلى من يريد استخدامه وبالتالي تتضمن ألا يتم التلاعب بالكود الخاص بك»

#### الشرح المبسّط:
عندما تكتب `#include<iostream>`، أنت في الحقيقة تطلب من المترجم أن "يقرأ" ملف **رأس (`Header File` بامتداد `.h`)** يحتوي فقط على **تصريحات** (ماذا يوجد، وليس كيف يعمل). أما الكود الفعلي (المنطق البرمجي) فيكون في ملف منفصل بامتداد `.cpp` بنفس الاسم.

**لماذا نفصل بينهما؟** لسببين رئيسيين:
1. **الحماية من السرقة**: عند توزيع مكتبتك البرمجية لطرف آخر، ترسل له ملف `.h` (يعرف فيه "ماذا يوجد") مع ملف تنفيذي مُترجَم (`.dll` أو مكافئه)، **دون** أن ترسل له كود `.cpp` الأصلي، فلا يستطيع رؤية أو تعديل منطقك الداخلي.
2. **تنظيم الكود**: فصل الواجهة (ماذا) عن التنفيذ (كيف) يجعل الكود أكثر تنظيمًا وقابلية للصيانة.

#### ⚙️ الخطوات / الخوارزمية: بناء ملف رأس (`Header`) لصف `student`

> ما هدف هذه العملية؟ حماية الكود الأصلي `student.cpp` من التلاعب عند توزيعه، مع إتاحة معرفة "ماذا يحتوي" الصف عبر `student.h`.

```algorithm
1 | فتح ملف الرأس | Header Guard | نستخدم #ifndef/#define/#endif لمنع تضمين نفس الملف أكثر من مرة
2 | تعريف الصف والحقول | class + private | نضع تصريحات الحقول والتوابع فقط بدون كود تنفيذي
3 | كتابة ملف cpp منفصل | #include "student.h" | نكتب جسم كل تابع مسبوقًا بـ student:: في هذا الملف
```

**محتويات الملف `student.h`:**
```cpp
#ifndef STUDENT_H          // if STUDENT_H is not yet defined
#define STUDENT_H          // define it now (prevents double inclusion)
#include<string>           // needed for the string type
using namespace std;       // avoid writing std:: everywhere

class student{
private:
    long id;
    string firstName;
    string lastName;
    float points;
    int marks[62];
public:
    student();                              // default constructor
    student(student&);                      // copy constructor
    student(long,string,string,float);      // parameterized constructor
    ~student();                              // destructor
    void changeMark(int,int);                // change a single mark
    long getId();
    string getFirstName();
    string getLastName();
    float getPoint();
    int getMark(int);
    void setId(long);
    void setFirstName(string);
    void setLastName(string);
    void setPoints(float);
};
#endif                     // end of header guard
```

**محتويات الملف `student.cpp`:**
```cpp
#include "student.h"      // include our own header (note: quotes, not <>)
using namespace std;

student::student(){
    points=0;
    id=0;
    firstName="";
    lastName="";
    for(int i=0;i<62;i++)
        marks[i]=0;
}
// ... other method bodies follow in the same style
```

#### نقطة مهمة ⚠️:
> نلاحظ الفرق بين عمليتي التضمين `#include`: الملفات القياسية التي جاءت مع `++C` (مثل `iostream` و`string`) تُكتب ضمن أقواس زاويّة `< >`، أما ملفنا الخاص `student.h` فيُكتب ضمن علامتي اقتباس `" "`.

**الاستخدام ضمن `main`:**
```cpp
#include<iostream>
#include<string>
#include "student.h"
using namespace std;

int main(){
    student c;
    long id; string firstName,lastName; float points;
    cin>>id>>firstName>>lastName>>points;
    c.setId(id);
    c.setFirstName(firstName);
    c.setLastName(lastName);
    c.setPoints(points);
}
```

### 9. مشكلة المرجع (`Reference Problem`)

#### النص الأصلي يقول:
> «من المشاكل التي يمكن أن نعاني منها أن نعيد قيمة من أحد عناصر التابع الخاصة `private` بالمرجع. نلاحظ تعديل التابع `getId` ليعيد القيمة بالمرجع... تكون قيمة الخرج 10 مما يعني أن التعديل على القيمة المعادة بالمرجع أثر على القيمة المخزنة ضمن الغرض دون المرور بالتابع `setId` ضاربين بعرض الحائط قواعد محددات الوصول»

#### الشرح المبسّط:
هذا مثال عملي خطير يوضح كيف يمكن أن **يُكسَر مبدأ التغليف بالكامل** إذا لم ننتبه:

```cpp
class student{
private:
    // ... fields
public:
    long& getId();   // DANGEROUS: returns a REFERENCE to the private field
};

student c;
long& x = c.getId();  // x is now an alias (another name) for the private field id
x = 10;                // modifying x directly modifies the private field!
cout<<c.getId();       // prints 10 — modified WITHOUT going through setId()!
```

**لماذا هذا خطير؟** لأن `getId()` يُعيد **مرجعًا (`reference`)** إلى الحقل الخاص نفسه، وليس نسخة منه. أي متحول (`x`) يحصل على هذا المرجع يصبح فعليًا "اسمًا ثانيًا" لنفس الحقل، فالتعديل على `x` هو تعديل مباشر على الحقل الخاص `id` **دون المرور إطلاقًا بالتابع `setId`** الذي قد يحتوي على تحقق من صحة القيمة!

#### الفهم الخاطئ ❌:
> إعادة القيمة بالمرجع من `getter` أمر آمن دائمًا لأنه مجرد "قراءة" للقيمة.

#### الفهم الصحيح ✅:
> إعادة القيمة بالمرجع من حقل `private` تفتح ثغرة تسمح بتعديل الحقل مباشرة من خارج الصف، وهذا يُبطل الغرض الكامل من جعل الحقل `private`.

#### 💡 التشبيه:
> تخيل أن موظف البنك (الـ `getter`) بدل أن يخبرك برصيدك، أعطاك **المفتاح الفعلي لخزنة الأموال**! الآن يمكنك تعديل الرصيد مباشرة دون المرور بأي إجراء رسمي للإيداع أو السحب (الـ `setter`).
> **وجه الشبه:** المفتاح الفعلي = المرجع (`reference`) المُعاد من `getId()`.

#### الدرس المستفاد:
> في هذه الحالة **لا يجب استخدام الإعادة بالمرجع** (`return by reference`) للحقول الخاصة عبر `getters`. القاعدة الآمنة: `getter` يجب أن يُعيد **نسخة (`by value`)** من القيمة وليس مرجعًا لها، إلا في حالات خاصة جدًا ومقصودة (مثل توفير أداء عالٍ مع وعي كامل بالمخاطر).

## الجزء الثاني: ملخص منظم

### أهم التعاريف والمفاهيم
| المصطلح | التعريف | مثال/ملاحظة |
| --- | --- | --- |
| `class` | قالب برمجي يجمع بيانات (حقول) وسلوك (توابع) مع محددات وصول | `class student{...};` |
| `object` (غرض) | نسخة فعلية تم بناؤها من الصف، تحمل قيمًا حقيقية | `student x(1,"a","b",232);` |
| `Encapsulation` (تغليف) | إخفاء تفاصيل التنفيذ الداخلي، إظهار الواجهة العامة فقط | `private` + `public` |
| `Constructor` (باني) | تابع خاص يحمل اسم الصف، يُستدعى تلقائيًا عند إنشاء الغرض | `student();` |
| `Destructor` (هادم) | تابع خاص باسم الصف مسبوقًا بـ `~`، يُستدعى عند حذف الغرض | `~student();` |
| `this` | مؤشر خاص يشير إلى الغرض الذي استدعى التابع الحالي | `this->id=id;` |
| `getter`/`setter` | توابع للقراءة/التعديل الآمن للحقول `private` | `getId()` / `setId(long)` |
| `Header/Implementation` | فصل التصريحات (`.h`) عن التنفيذ (`.cpp`) | `student.h` و`student.cpp` |

### المكونات الرئيسية (مرجع سريع)
| الأداة | الوظيفة | ملاحظة |
| --- | --- | --- |
| `private:` | يخفي الحقول/التوابع عن الخارج | يمكن الوصول من داخل الصف نفسه فقط |
| `public:` | يفتح الحقول/التوابع للخارج | يُستخدم للواجهة التي يستخدمها المستخدم |
| `::` (عامل تحليل المجال) | يربط تعريف تابع خارج الصف باسم الصف | `student::student(){...}` |
| `&` (مرجع) في الوسيط | يمرر الغرض بالمرجع بدل النسخ (يُستخدم في الباني الناسخ) | `student(student& c)` |
| `#ifndef`/`#define`/`#endif` | حارس التضمين (`Include Guard`) لمنع تكرار تعريف الصف | يُستخدم في ملفات `.h` |

### جداول مقارنات سريعة
| المقارنة | الأول | الثاني | الفرق |
| --- | --- | --- | --- |
| `struct` مقابل `class` | `struct`: حقول فقط، لا محددات وصول افتراضية خاصة | `class`: حقول + توابع + محددات وصول (`private` افتراضيًا) | الصف يضيف السلوك والحماية |
| الباني الافتراضي مقابل الباني ذو الوسطاء | لا يحتاج وسطاء، يعطي قيمًا ثابتة | يستقبل وسطاء لبناء الغرض بقيم محددة | الفرق في مصدر القيم الابتدائية |
| `c.function1()` مقابل `c1->function1()` | يُستخدم مع غرض عادي (وليس مؤشرًا) | يُستخدم مع مؤشر على غرض | نوع المتحول (غرض أم مؤشر) |
| إعادة بالقيمة مقابل إعادة بالمرجع في `getter` | تُعيد نسخة آمنة من القيمة | تُعيد مرجعًا يسمح بالتعديل المباشر | الأمان مقابل الأداء |

### قاموس المصطلحات
| الفئة | المصطلحات |
| --- | --- |
| البناء والهدم | `Constructor`, `Destructor`, `Default Constructor`, `Parameterized Constructor`, `Copy Constructor` |
| الوصول والحماية | `private`, `public`, `Encapsulation`, `Access Modifier` |
| الأدوات الخاصة | `this`, `::` (`Scope Resolution Operator`) |
| الملفات | `Header File`, `Implementation File`, `Include Guard`, `dll` |
| التوابع المساعدة | `getter`, `setter` |

### أبرز النقاط الذهبية
1. الصف = بيانات + سلوك + محددات وصول، بينما السجل (`struct`) = بيانات فقط.
2. الباني يُستدعى تلقائيًا عند إنشاء أي غرض جديد، ولا يحتاج نوع إرجاع.
3. عند تشابه اسم الوسيط مع اسم الحقل، استخدم `this->` للإشارة الصريحة إلى حقل الصف.
4. الهادم يُستدعى تلقائيًا في نهاية عمر الغرض، وترتيب الهدم عكسي لترتيب البناء.
5. الباني الناسخ يجب أن يستقبل وسيطه بالمرجع (`&`) لتجنب استدعاء نفسه بشكل لا نهائي.
6. `Getters`/`Setters` هي البوابة الآمنة للوصول إلى الحقول `private` من خارج الصف.
7. فصل ملف `.h` عن `.cpp` يسمح بتوزيع الكود المُترجَم دون كشف المنطق الداخلي.
8. لا تُعِد حقلًا `private` بالمرجع من `getter`، لأن ذلك يُبطل حماية `private` بالكامل.

### الأخطاء الشائعة عند الطلاب ⚠️
| الخطأ | التصحيح |
| --- | --- |
| كتابة `student x();` لاستدعاء الباني الافتراضي | استخدم `student x;` بدون أقواس |
| نسيان استخدام `this->` عند تشابه اسم الحقل مع الوسيط | استخدم `this->fieldName=fieldName;` |
| تمرير الوسيط بالقيمة في الباني الناسخ | مرره بالمرجع: `student(student& c)` |
| إعادة حقل `private` بالمرجع من `getter` | أعده بالقيمة: `long getId(){return id;}` |
| كتابة كل المنطق داخل تعريف الصف في `.h` بدل فصله في `.cpp` | عرّف التوابع خارج الصف باستخدام `student::` في ملف `.cpp` منفصل |

### خطوات وإجراءات المحاضرة

#### ⚙️ الخطوات / الخوارزمية: إنشاء غرض جديد بالباني ذو الوسطاء

> ما هدفه؟ فهم تسلسل ما يحدث فعليًا عند كتابة `student x(1,"a","b",232);`

```algorithm
1 | حجز الذاكرة | Memory Allocation | يحجز المترجم مساحة كافية لكل حقول الصف
2 | استدعاء الباني | Constructor Call | يُستدعى الباني المطابق لعدد ونوع الوسطاء الممرَّرة
3 | تعيين this | this Pointer | يشير this داخل الباني إلى عنوان الغرض الجديد (x)
4 | تنفيذ جسم الباني | Constructor Body | تُنسَخ قيم الوسطاء إلى حقول الغرض عبر this-> أو مباشرة
```

#### نقاط التنفيذ:
- إذا لم يوجد باني مطابق لعدد/نوع الوسطاء الممرَّرة، يفشل الترجمة (خطأ ترجمة).
- ترتيب الوسطاء يجب أن يطابق تمامًا ترتيب المعاملات في تعريف الباني.

---

#### ⚙️ الخطوات / الخوارزمية: حذف غرض من الذاكرة (الهدم)

> ما هدفه؟ فهم اللحظة الدقيقة التي يُستدعى فيها الهادم تلقائيًا.

```algorithm
1 | تحديد نهاية مجال الرؤية | Scope End | يصل التنفيذ لآخر سطر } الخاص بالكتلة التي عُرِّف فيها الغرض
2 | استدعاء الهادم | Destructor Call | يستدعي المترجم تلقائيًا ~ClassName() على الغرض
3 | تحرير الذاكرة | Memory Deallocation | تُعاد المساحة المحجوزة للغرض إلى النظام
```

#### نقاط التنفيذ:
- إذا انتهى البرنامج بشكل غير سليم (تعطل مفاجئ)، لن يُستدعى الهادم، وقد تبقى موارد غير مُحرَّرة (مثل ملفات مفتوحة).
- الأغراض المحلية داخل حلقة تكرارية تُبنى وتُهدم في كل تكرار، وليس مرة واحدة فقط.

### أنماط الأكواد والبنى المتكررة
| النمط | البنية الأساسية | متى تستخدمه |
| --- | --- | --- |
| فصل تعريف/تنفيذ التابع | `ReturnType ClassName::methodName(params){...}` | عند كتابة جسم أي تابع خارج تعريف الصف |
| استخدام `this` لحل تعارض الأسماء | `this->field = parameterName;` | عندما يتشابه اسم الوسيط مع اسم الحقل |
| الباني الناسخ الآمن | `ClassName(ClassName& other){ field=other.field; ... }` | عند الحاجة لنسخ غرض موجود لغرض جديد |
| حارس ملف الرأس | `#ifndef X_H #define X_H ... #endif` | في بداية ونهاية كل ملف `.h` |

### أنماط التعامل والسلوك
| السيناريو | التعامل الصحيح | لماذا؟ |
| --- | --- | --- |
| إنشاء غرض بدون قيم أولية محددة | استخدام الباني الافتراضي | يمنع القيم العشوائية (Garbage Values) |
| نسخ غرض موجود إلى غرض جديد | استخدام الباني الناسخ بوسيط مُمرَّر بالمرجع | يتجنب النسخ الزائد واستدعاء الباني الناسخ لنفسه |
| قراءة حقل `private` من خارج الصف | استخدام `getter` يُعيد بالقيمة | يحافظ على التغليف ويمنع التعديل غير المصرح به |
| تعديل حقل `private` من خارج الصف | استخدام `setter` مع تحقق من صحة القيمة إن لزم | يضمن أن القيم المعدَّلة منطقية دومًا |

### الأفكار الرئيسية الشاملة
1. مبدأ **التغليف (`Encapsulation`)** هو الفكرة المحورية وراء كل هذه المحاضرة: إخفِ التفاصيل، اعرض الواجهة.
2. البواني والهوادم ليست اختيارية فعليًا — المترجم يضيفها تلقائيًا (فارغة) إذا لم يعرّفها المبرمج.
3. مشكلة إعادة المرجع من `getter` مثال حي على كيف يمكن أن يُنقَض مبدأ التغليف عن غير قصد رغم استخدام `private` بشكل ظاهري صحيح.

## الجزء الثالث: أسئلة اختيار من متعدد (MCQ)

> **16 سؤالاً** — مستوى: متوسط/صعب. التوزيع: مقارنات 25%، سيناريو كود 35%، تطبيق 30%، تتبع خوارزمية 10%.

### السؤال 1 (سهل)
ما الفرق الجوهري بين `struct` و`class` في `++C` كما ورد في المحاضرة؟
أ) لا يوجد أي فرق بينهما إطلاقًا
ب) الصف يضيف التوابع ومحددات الوصول والبواني/الهوادم فوق ما يوفره السجل
ج) السجل يدعم الوراثة بينما الصف لا يدعمها
د) الصف لا يمكن أن يحتوي على مصفوفات كحقول
**الإجابة الصحيحة: ب**
**التعليل:** (ب) صحيحة لأن المحاضرة أوضحت أن الصف "من الناحية التقنية يشابه السجل" لكنه "يمتلك بالإضافة للحقول مهام ومحددات وصول وبواني وهوادم". (أ) خاطئة لوجود فروق واضحة. (ج) خاطئة وعكس الحقيقة تقريبًا (الوراثة تُشرح لاحقًا للصفوف تحديدًا). (د) خاطئة، مثال `marks[62]` موجود في كلا التمثيلين في المحاضرة.

### السؤال 2 (متوسط)
ماذا تفعل الكلمة المحجوزة `private` ضمن تعريف الصف؟
أ) تجعل الحقول قابلة للوصول من أي مكان في البرنامج
ب) تحصر إمكانية الوصول للحقول/التوابع بتوابع الصف نفسه (والأصدقاء لاحقًا)
ج) تحذف الحقل من الذاكرة بعد الاستخدام
د) تجعل الحقل ثابتًا (`const`) تلقائيًا
**الإجابة الصحيحة: ب**
**التعليل:** (ب) مطابقة لنص المحاضرة حرفيًا. (أ) هذا تعريف `public` وليس `private`. (ج) لا علاقة لـ `private` بإدارة الذاكرة. (د) `private` لا علاقة له بـ `const`، وهما مفهومان مختلفان تمامًا.

### السؤال 3 (صعب)
انظر الكود التالي:
```cpp
class student{
private:
    long id;
public:
    student(long id){
        id=id; // line X
    }
};
```
ماذا يحدث في السطر المعلَّم `X`؟
أ) يُسند الوسيط `id` إلى الحقل `id` بنجاح
ب) خطأ ترجمة (Compilation Error)
ج) الوسيط المحلي `id` يُسند إلى نفسه، والحقل `id` يبقى دون تهيئة صحيحة
د) يُسند الحقل `id` إلى الوسيط `id`
**الإجابة الصحيحة: ج**
**التعليل:** حسب قاعدة "الأقرب أولًا" في مجال الرؤية، `id=id` يعني أن المترجم يفسّر كلا الطرفين على أنهما الوسيط المحلي، فلا يحدث أي تأثير فعلي على حقل الصف. (أ) و(د) خاطئتان لأنهما تفترضان تأثيرًا على الحقل. (ب) خاطئة، الكود سليم نحويًا (لا خطأ ترجمة) لكنه خاطئ منطقيًا.

### السؤال 4 (متوسط)
ما الحل الصحيح لمشكلة السؤال السابق؟
أ) إعادة تسمية الوسيط أو استخدام `this->id=id;`
ب) حذف الحقل `id` من الصف نهائيًا
ج) وضع الوسيط ضمن قسم `private`
د) استخدام `static` قبل الوسيط
**الإجابة الصحيحة: أ**
**التعليل:** هذان هما الحلان المذكوران صراحة في المحاضرة. (ب) يفقدنا البيانات المطلوب تخزينها. (ج) الوسطاء لا تُوضع ضمن أقسام `private`/`public` أصلًا. (د) `static` غير متعلق بحل هذه المشكلة إطلاقًا.

### السؤال 5 (متوسط)
ما هي قيمة `this` داخل الباني عند تنفيذ السطر `student y(2,"za","vb",432);`؟
أ) `&x` (عنوان غرض آخر اسمه x)
ب) `NULL`
ج) `&y`
د) لا قيمة محددة، تعتمد على النظام
**الإجابة الصحيحة: ج**
**التعليل:** المحاضرة نصت أن `this` يشير للغرض الذي استدعى الباني، وهنا الغرض المُستدعي هو `y`. (أ) خطأ لأنها لغرض مختلف. (ب) `this` لا يكون `NULL` أثناء بناء غرض فعلي صالح. (د) خاطئة، `this` محدد بدقة ومنطقي وليس عشوائيًا.

### السؤال 6 (صعب)
لماذا يجب تمرير الوسيط في الباني الناسخ بالمرجع (`&`) وليس بالقيمة؟
أ) لأن المرجع أسرع في التنفيذ فقط، لا علاقة أخرى
ب) لتجنب استدعاء الباني الناسخ نفسه بشكل لا نهائي عند تمرير الوسيط بالقيمة
ج) لأن اللغة تمنع تمرير الأغراض بالقيمة نهائيًا
د) لضمان أن الغرض الأصلي يتغير أيضًا
**الإجابة الصحيحة: ب**
**التعليل:** تمرير الغرض بالقيمة يتطلب نسخه أولًا، وهذا النسخ يستدعي الباني الناسخ نفسه، فينتج استدعاء لا نهائي. (أ) صحيحة جزئيًا لكنها ليست السبب الجوهري المذكور في المحاضرة. (ج) خاطئة، تمرير الأغراض بالقيمة ممكن لغويًا لكنه يسبب هذه المشكلة تحديدًا هنا. (د) خاطئة، الباني الناسخ لا يُفترض أن يُعدّل الغرض الأصلي.

### السؤال 7 (متوسط)
ما هي وظيفة الهادم (`Destructor`) في الصف؟
أ) إنشاء غرض جديد بقيم ابتدائية
ب) تنظيم عمليات التنظيف عند حذف الغرض من الذاكرة (إغلاق ملفات، تحرير ذاكرة ديناميكية)
ج) تعديل قيم الحقول الخاصة من الخارج
د) مقارنة غرضين مع بعضهما
**الإجابة الصحيحة: ب**
**التعليل:** مطابق لنص المحاضرة حرفيًا. (أ) هذه وظيفة الباني وليس الهادم. (ج) هذه وظيفة `setter`. (د) هذا يتطلب `operator overloading` وهو موضوع مختلف (سيُشرح لاحقًا في محاضرة السجلات).

### السؤال 8 (سهل)
ماذا يحدث إذا لم يُعرّف المبرمج أي هادم للصف؟
أ) يرفض المترجم ترجمة الكود
ب) يُضاف هادم فارغ تلقائيًا من قبل المترجم
ج) لا يمكن إنشاء أي غرض من هذا الصف
د) يُستخدم هادم الصف الأب تلقائيًا
**الإجابة الصحيحة: ب**
**التعليل:** نص المحاضرة: "إذا لم يتم تعريف أي هادم يتم إضافة هادم فارغ". (أ) خاطئة، لا خطأ ترجمة بسبب غياب الهادم. (ج) خاطئة، يمكن إنشاء الأغراض بشكل طبيعي. (د) خاطئة ومفهوم الوراثة ليس موضوع هذه المحاضرة.

### السؤال 9 (صعب)
انظر الكود:
```cpp
int main(){
    student a(1,"x","y",10);
    {
        student b(2,"m","n",20);
    } // end of inner block
    student c(3,"p","q",30);
}
```
ما هو الترتيب الصحيح لاستدعاء الهدم لهذه الأغراض الثلاثة؟
أ) a ثم b ثم c
ب) b (عند نهاية الكتلة الداخلية) ثم c ثم a (عند نهاية main)
ج) c ثم b ثم a
د) لا يمكن تحديد الترتيب
**الإجابة الصحيحة: ب**
**التعليل:** الهادم يُستدعى في آخر سطر من مجال الرؤية الذي عُرّف فيه الغرض. `b` مجال رؤيته ينتهي عند إغلاق الكتلة الداخلية `}` فيُهدم أولًا، ثم عند نهاية `main` يُهدم `c` ثم `a` بترتيب عكسي عن بنائهما (a بُني أولًا فيُهدم أخيرًا). (أ)، (ج) لا تحترمان قاعدة "الأقرب أولًا" في نهاية مجال الرؤية. (د) خاطئة، الترتيب محدد بدقة حسب قواعد اللغة.

### السؤال 10 (متوسط)
ما الفرق بين استدعاء `c.function1()` و`c1->function1()`؟
أ) لا يوجد فرق إطلاقًا وهما متطابقان في كل الحالات
ب) الأول يُستخدم مع غرض عادي، والثاني يُستخدم مع مؤشر على غرض
ج) الأول للتوابع `private`، والثاني للتوابع `public`
د) الثاني أسرع دائمًا من الأول
**الإجابة الصحيحة: ب**
**التعليل:** `c` غرض عادي (`Class1 c;`) يُستخدم معه عامل النقطة، بينما `c1` مؤشر (`Class1 *c1=new Class1();`) يُستخدم معه عامل السهم. (أ) خاطئة، صياغتهما مختلفة حسب نوع المتحول. (ج) خاطئة تمامًا، محددات الوصول لا علاقة لها بنوع عامل الاستدعاء. (د) لا أساس لهذا الادعاء في المحاضرة.

### السؤال 11 (متوسط)
ما الغرض من فصل الكود في ملفي `.h` و`.cpp`؟
أ) لتسريع تنفيذ البرنامج فقط
ب) لتمكين توزيع المكتبة البرمجية (ملف رأس + ملف تنفيذي) دون كشف منطق الكود الأصلي
ج) لأن اللغة تفرض هذا الفصل إجباريًا في كل الحالات
د) لتقليل حجم الذاكرة المستخدمة أثناء التنفيذ
**الإجابة الصحيحة: ب**
**التعليل:** مطابق لما ورد في المحاضرة عن حماية الكود من السرقة والتلاعب عبر توزيع `headers` مع ملفات `dll`. (أ) و(د) لا علاقة لهما بالسبب المذكور. (ج) خاطئة، هذا الفصل اصطلاح واصطلاح تنظيمي وليس إلزاميًا لغويًا (يمكن كتابة كل شيء في ملف واحد).

### السؤال 12 (صعب)
انظر الكود التالي:
```cpp
class point{
private:
    int x;
public:
    int& getX(){return x;}
};
point p;
int& r = p.getX();
r = 99;
```
ماذا يحدث بعد تنفيذ `r = 99;`؟
أ) خطأ ترجمة لأن `x` حقل `private`
ب) تتغير قيمة `x` داخل `p` إلى 99 مباشرة دون استخدام أي `setter`
ج) `r` متحول مستقل تمامًا عن `x` ولا يؤثر عليه
د) القيمة لا تتغير لأن `getX` تُعيد نسخة فقط
**الإجابة الصحيحة: ب**
**التعليل:** هذا بالضبط سيناريو "مشكلة المرجع" في المحاضرة: إعادة حقل `private` بالمرجع تجعل أي متحول خارجي يحصل عليه (`r`) اسمًا آخر لنفس الحقل، فتعديله يعدّل الحقل مباشرة. (أ) خاطئة، لا خطأ ترجمة، الكود مقبول لغويًا رغم خطورته المنطقية. (ج) و(د) تناقضان مباشرة مفهوم المرجع (`reference`).

### السؤال 13 (متوسط)
ما الحل الصحيح لمنع مشكلة السؤال السابق؟
أ) جعل `getX` يُعيد القيمة بالقيمة (`by value`) بدل المرجع
ب) حذف التابع `getX` نهائيًا
ج) جعل `x` عامًا (`public`)
د) استخدام `const` قبل اسم الصف فقط
**الإجابة الصحيحة: أ**
**التعليل:** هذا الحل المذكور صراحة في المحاضرة: "في هذه الحالة لا يجب استخدام الإعادة بالمرجع". (ب) يفقدنا إمكانية القراءة كليًا. (ج) يُبطل الحماية تمامًا وهو أسوأ من المشكلة الأصلية. (د) `const` وحدها بدون تغيير نوع الإرجاع لا تحل المشكلة الجوهرية بذات الطريقة الموصوفة.

### السؤال 14 (سهل)
أي مما يلي هو الشكل الصحيح لتعريف الباني الافتراضي ضمن الصف `student`؟
أ) `void student(){}`
ب) `student();`
ج) `int student(){}`
د) `student::void();`
**الإجابة الصحيحة: ب**
**التعليل:** الباني لا يمتلك أي نوع إرجاع، حتى `void`، ويحمل اسم الصف نفسه فقط متبوعًا بأقواس الوسطاء. (أ) و(ج) خاطئتان لأنهما تضعان نوع إرجاع، وهذا غير مسموح للبواني. (د) صياغة غير صحيحة نحويًا إطلاقًا.

### السؤال 15 (صعب)
ما هو ناتج الكود التالي (بافتراض وجود تعريفات `Building`/`Destroying` كما في المحاضرة)؟
```cpp

    student temp(99,"t","t",1);
}
int main(){
    cout<<"start"<<endl;
    f();
    cout<<"end"<<endl;
}
```
أ) start ثم end فقط، بدون أي بناء أو هدم
ب) start، ثم Building (99)، ثم Destroying (99) (قبل طباعة end)، ثم end
ج) Building (99) قبل start، ثم start و end
د) start، end، ثم Building (99)، ثم Destroying (99)
**الإجابة الصحيحة: ب**
**التعليل:** الغرض `temp` محلي داخل `f()`، فيُبنى عند استدعاء `f()` ويُهدم عند نهاية `f()` (قبل العودة إلى `main` ومتابعة طباعة "end"). (أ) خاطئة، الغرض `temp` سيُبنى ويُهدم فعليًا. (ج) خاطئة، لا يوجد سبب لبناء الغرض قبل `main` لأنه ليس غرضًا عامًا (`Global`). (د) خاطئة، الترتيب الزمني غير صحيح — الهدم يحدث قبل طباعة "end" وليس بعدها.

### السؤال 16 (متوسط)

## الجزء الرابع: أسئلة تصحيح الكود

### سؤال تصحيح 1 (النوع: `misconception`)

**الكود (يحتوي خطأ):**
```cpp
class student{
private:
    long id;
public:
    student(long id){
        id = id; // intended to assign parameter to field
    }
    long getId(){ return id; }
};
```
**اكتشف الخطأ:** السطر `id = id;` لا يُسند شيئًا فعليًا إلى الحقل، لأن المترجم يفسّر كلا الطرفين على أنهما الوسيط المحلي بسبب تشابه الأسماء (قاعدة الأقرب أولًا).

**التصحيح:**
```cpp
class student{
private:
    long id;
public:
    student(long id){
        this->id = id; // this->id refers to the class field explicitly
    }
    long getId(){ return id; }
};
```
**شرح الحل:**
1. استخدمنا `this->` لتمييز حقل الصف عن الوسيط المحلي.
2. الآن `this->id` تشير للحقل الخاص بالغرض، بينما `id` (بدون `this`) تبقى تشير للوسيط.
3. عند استدعاء `getId()` بعد البناء، ستعيد القيمة الصحيحة الممرَّرة عند الإنشاء بدل قيمة عشوائية غير مهيَّأة.

### سؤال تصحيح 2 (النوع: `logic`)

**الكود (يحتوي خطأ):**
```cpp
class student{
private:
    string firstName;
public:
    student(string firstName){
        firstName = firstName; // same naming collision bug pattern
    }
    string getFirstName(){ return firstName; }
};
```
**اكتشف الخطأ:** نفس نمط الخطأ السابق لكن مع حقل من نوع `string` بدلًا من `long` — لن يتم إسناد القيمة الممرَّرة إلى الحقل.

**التصحيح:**
```cpp
class student{
private:
    string firstName;
public:
    student(string firstName){
        this->firstName = firstName; // explicit field assignment via this
    }
    string getFirstName(){ return firstName; }
};
```
**شرح الحل:**
1. المشكلة تتكرر بغض النظر عن نوع البيانات (`long`, `string`, إلخ) لأن السبب هو تشابه الأسماء، لا نوع البيانات.
2. الحل العام: استخدم `this->` أو غيّر اسم الوسيط (مثل `fName`) لتفادي أي التباس.
3. من الجيد اعتماد اصطلاح ثابت في كل المشروع (مثل استخدام `this->` دومًا) لتقليل احتمال الخطأ.

### سؤال تصحيح 3 (النوع: `dead_code`)

**الكود (يحتوي خطأ):**
```cpp
class student{
private:
    int marks[62];
public:
    student(student& c){
        for(int i=0;i<62;i++)
            marks[i]=c.marks[i];
        return; // dead code follows
        cout<<"Copy done"<<endl; // this line will never execute
    }
};
```
**اكتشف الخطأ:** السطر `cout<<"Copy done"<<endl;` يقع بعد جملة `return;`، وبالتالي هو **كود ميت (`Dead Code`)** لن يُنفَّذ أبدًا لأن التابع ينتهي قبل الوصول إليه. كما أن الباني لا يُفترض أن يحتوي `return;` أصلًا لأنه لا يُعيد قيمة.

**التصحيح:**
```cpp
class student{
private:
    int marks[62];
public:
    student(student& c){
        for(int i=0;i<62;i++)
            marks[i]=c.marks[i];
        cout<<"Copy done"<<endl; // now this line will execute normally
    }
};
```
**شرح الحل:**
1. حذفنا جملة `return;` غير الضرورية والخاطئة السياق (البواني لا تُعيد قيمًا في `++C`).
2. بذلك أصبح السطر التالي (`cout`) قابلًا للتنفيذ الطبيعي.
3. درس عام: أي كود يقع بعد `return` مباشرة في نفس كتلة التنفيذ هو كود ميت يجب حذفه أو إعادة ترتيب المنطق.

---

### سؤال تصحيح 4 (النوع: `return_check`)

**الكود (يحتوي خطأ):**
```cpp
class student{
private:
    float points;
public:
    float setPoints(float x){
        points = x; // no return statement despite float return type
    }
};
```
**اكتشف الخطأ:** التابع مُعرَّف بنوع إرجاع `float` لكنه لا يحتوي على أي جملة `return`، وهذا سلوك غير معرَّف (`Undefined Behavior`) عند استخدام القيمة المُعادة.

**التصحيح:**
```cpp
class student{
private:
    float points;
public:
    void setPoints(float x){   // changed return type to void since no meaningful value is returned
        points = x;
    }
};
```
**شرح الحل:**
1. بما أن التابع لا يحتاج فعليًا لإعادة أي قيمة (هو `setter` بسيط)، الأصح تغيير نوع الإرجاع إلى `void`.
2. البديل الآخر: إبقاء `float` وإضافة `return points;` في نهاية التابع إذا أردنا فعلًا إعادة القيمة الجديدة.
3. درس عام: أي تابع بنوع إرجاع غير `void` يجب أن يحتوي جملة `return` تُطابق النوع المُعلَن في كل المسارات الممكنة للتنفيذ.

### سؤال تصحيح 5 (النوع: `misconception`)

**الكود (يحتوي خطأ):**
```cpp
class student{
private:
    long id;
public:
    long& getId(){ return id; } // returns by reference
};

int main(){
    student c;
    long x = c.getId(); // student thinks this is always "safe read-only access"
    x = 500;            // student believes this does NOT affect c
    cout<<c.getId();    // student expects the original value, not 500
}
```
**اكتشف الخطأ:** الطالب يعتقد خطأً أن `x = c.getId();` تُعيد دائمًا نسخة آمنة. لكن هنا فعليًا `x` هو متحول عادي (`long x`) وليس مرجعًا (`long& x`)، لذلك **لا توجد مشكلة فعلية في هذا الاستدعاء تحديدًا** — لكن لو استُخدم `long& x = c.getId();` بدلًا من `long x`، لكانت المشكلة حقيقية تمامًا كما شُرح في المحاضرة.

**التصحيح (لتوضيح الفرق):**
```cpp
long x = c.getId();     // SAFE: x is a copy, modifying x does NOT affect c.id
long& y = c.getId();    // DANGEROUS: y is an alias, modifying y DOES affect c.id
```
**شرح الحل:**
1. المشكلة الحقيقية لا تكمن في `getId()` نفسه (رغم أنه يُعيد بالمرجع)، بل في **كيف يستقبل المستدعي القيمة المُعادة**.
2. إذا استقبلها بمتحول عادي (`long x`)، تُنسَخ القيمة فورًا فتصبح آمنة.
3. إذا استقبلها بمتحول مرجعي (`long& y`)، يبقى الارتباط بالحقل الأصلي قائمًا وتُصبح المشكلة فعلية — لذا الحل الجذري الأسلم يبقى: لا تُعِد `getter` بالمرجع من الأساس، لتفادي الاعتماد على انتباه المستخدم.

| 4 | هدم `c` (نهاية الكتلة الداخلية) | ؟ |
| 5 | نهاية `main`: هدم `b` ثم `a` | ؟ |

**نموذج الحل:**
| الخطوة | العملية | الحالة (الناتج المطبوع حتى الآن) |
| --- | --- | --- |
| 1 | بناء `a` | B1 |
| 2 | بناء `b` | B1B2 |
| 3 | بناء `c` | B1B2B3 |
| 4 | هدم `c` | B1B2B3D3 |
| 5 | هدم `b` ثم `a` | B1B2B3D3D2D1 |

**النتيجة:** `B1B2B3D3D2D1`

### تمرين تتبع 3: تتبع مشكلة المرجع خطوة بخطوة

**المدخل:**
```cpp
class Box{
private:
    int size;
public:
    Box(int s){ size=s; }
    int& getSize(){ return size; }
};
Box b(5);
int& r = b.getSize();
r = 50;
```

**تتبّع خطوة بخطوة (أكمل الجدول):**
| الخطوة | العملية | قيمة `size` داخل `b` |
| --- | --- | --- |
| 1 | بناء `b(5)` | ؟ |
| 2 | `int& r = b.getSize();` | ؟ |
| 3 | `r = 50;` | ؟ |

**نموذج الحل:**
| الخطوة | العملية | قيمة `size` داخل `b` |
| --- | --- | --- |
| 1 | بناء `b(5)` | 5 |
| 2 | `int& r = b.getSize();` (r يصبح اسمًا بديلًا لـ size) | 5 (لم يتغير بعد) |
| 3 | `r = 50;` (تعديل r يعدّل size مباشرة) | 50 |

**النتيجة:** `50` — القيمة تغيرت دون المرور بأي `setter`، وهذا يوضح خطورة إعادة الحقول `private` بالمرجع.

## الجزء الرابع: بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ما هو الفرق الأساسي بين `struct` و`class` كما ورد في المحاضرة؟
A: `struct` يحتوي حقولًا فقط، بينما `class` يضيف توابع ومحددات وصول (`private`/`public`) وبواني وهوادم.

**Q2:** ما اسم التابع الذي يُستدعى تلقائيًا عند إنشاء غرض جديد؟
A: الباني (`Constructor`)، ويحمل نفس اسم الصف.

**Q3:** كم نوعًا من البواني ذُكر في المحاضرة؟
A: ثلاثة أنواع: الباني الافتراضي، الباني ذو الوسطاء، والباني الناسخ.

**Q4:** ماذا يشير إليه المؤشر `this` داخل تابع عضو في صف؟
A: يشير إلى الغرض الذي استدعى هذا التابع.

**Q5:** متى نحتاج لاستخدام `this->` بدل الاسم المباشر للحقل؟
A: عندما يتشابه اسم الحقل مع اسم وسيط محلي داخل التابع، لتمييز الحقل عن الوسيط.

**Q6:** لماذا يُمرَّر وسيط الباني الناسخ بالمرجع (`&`) وليس بالقيمة؟
A: لتجنب استدعاء الباني الناسخ نفسه بشكل لا نهائي عند محاولة نسخ الوسيط لتمريره بالقيمة.

**Q7:** ما اسم التابع الذي يُستدعى تلقائيًا عند حذف الغرض من الذاكرة؟
A: الهادم (`Destructor`)، ويحمل نفس اسم الصف مسبوقًا بالرمز `~`.

**Q8:** ماذا يحدث إذا لم يعرّف المبرمج أي باني أو هادم للصف؟
A: يضيف المترجم تلقائيًا باني وهادم فارغين.

**Q9:** ما هو الغرض من توابع `getters` و`setters`؟
A: توفير وصول آمن ومضبوط لقراءة/تعديل الحقول الخاصة (`private`) من خارج الصف.

**Q10:** ما الفرق بين استدعاء التابع عبر `.` وعبر `->`؟
A: `.` تُستخدم مع غرض عادي، بينما `->` تُستخدم مع مؤشر يشير إلى غرض.

**Q11:** لماذا نفصل الكود بين ملف `.h` وملف `.cpp`؟
A: لتمكين توزيع المكتبة البرمجية (ملف رأس + ملف تنفيذي مُترجَم) دون كشف الكود المصدري الأصلي.

**Q12:** ما هي المشكلة التي تنتج عن إعادة حقل `private` بالمرجع من `getter`؟
A: تصبح إمكانية تعديل الحقل مباشرة من خارج الصف متاحة، دون المرور بأي `setter`، مما يُبطل حماية `private`.

**Q13:** ما هو الترتيب الزمني لاستدعاء الهادم مقارنة بالباني ضمن نفس مجال الرؤية؟
A: يكون الترتيب عكسيًا: آخر غرض بُني هو أول غرض يُهدم (مثل مبدأ المكدس LIFO).

**Q14:** متى يُستدعى باني الغرض الساكن (`static`)؟
A: مرة واحدة فقط، عند الوصول لسطر تعريف المتحول الساكن للمرة الأولى.

## الجزء الخامس: كتابة الكود الكامل (مرجع شامل)

```cpp
// ===== student.h =====
#ifndef STUDENT_H
#define STUDENT_H
#include<string>
using namespace std;

class student{
private:
    long id;                       // university id
    string firstName;              // first name
    string lastName;               // last name
    float points;                  // high school average
    int marks[62];                 // marks across all courses
public:
    student();                                     // default constructor
    student(student& c);                           // copy constructor
    student(long id,string firstName,string lastName,float points); // parameterized constructor
    ~student();                                     // destructor

    void changeMark(int i,int v);                   // change a single mark safely

    long getId();
    string getFirstName();
    string getLastName();
    float getPoints();
    int getMark(int i);

    void setId(long v);
    void setFirstName(string v);
    void setLastName(string v);
    void setPoints(float v);
};
#endif

// ===== student.cpp =====
#include "student.h"
#include<iostream>
using namespace std;

// Default constructor: initializes all fields to safe empty/zero values
student::student(){
    id=0;
    firstName="";
    lastName="";
    points=0;
    for(int i=0;i<62;i++)
        marks[i]=0;
    cout<<"Building student with Id: "<<id<<endl;
}

// Copy constructor: parameter passed by reference to avoid infinite recursion
student::student(student& c){
    id=c.id;
    firstName=c.firstName;
    lastName=c.lastName;
    points=c.points;
    for(int i=0;i<62;i++)
        marks[i]=c.marks[i];
}

// Parameterized constructor: uses this-> to disambiguate parameters from fields
student::student(long id,string firstName,string lastName,float points){
    this->id=id;
    this->firstName=firstName;
    this->lastName=lastName;
    this->points=points;
    for(int i=0;i<62;i++)
        marks[i]=0;
    cout<<"Building student with Id: "<<id<<endl;
}

// Destructor: prints a message when the object is destroyed
student::~student(){
    cout<<"Destroying student with Id: "<<id<<endl;
}

// Member method: validates arguments before modifying a mark
void student::changeMark(int i,int v){
    if(i>=0 && i<62 && v>=0 && v<=100)  // validate index and value ranges
        marks[i]=v;
}

// Getters: return copies of values (safe, does not expose internal references)
long student::getId(){ return id; }
string student::getFirstName(){ return firstName; }
string student::getLastName(){ return lastName; }
float student::getPoints(){ return points; }
int student::getMark(int i){ return marks[i]; }

// Setters: controlled write access to private fields
void student::setId(long v){ id=v; }
void student::setFirstName(string v){ firstName=v; }
void student::setLastName(string v){ lastName=v; }
void student::setPoints(float v){ points=v; }

// ===== main.cpp =====
#include<iostream>
#include<string>
#include "student.h"
using namespace std;

student gl(10,"first","First",22); // global object, built before main

int main(){
    cout<<"main starts"<<endl;
    student mn(11,"second","Second",33); // local object in main

    cout<<"the loop starts"<<endl;
    for(int i=0;i<5;i++)
        student mm(12+i,"third","Third",44); // rebuilt and destroyed each iteration
    cout<<"the loop ends"<<endl;

    cout<<"main ending"<<endl;
    return 0;
} // mn and gl are destroyed here, in reverse order of construction
```

## الجزء الخامس: أسئلة نظرية متوقعة بالامتحان

### سؤال 1: ما هو الصف (`Class`) وما الفرق بينه وبين السجل (`Struct`)؟
**نموذج الإجابة:**
1. التعريف: `class` هو قالب برمجي يجمع بيانات (حقول) مع سلوك (توابع) ومحددات وصول ضمن كتلة واحدة.
2. المكونات/الشروط: يحتوي حقولًا خاصة/عامة، توابع، باني وهادم.
3. مثال: `class student{ private: long id; public: void changeMark(int,int); };`
4. متى نستخدم: عندما نحتاج تمثيل غرض واقعي له صفات وسلوك معًا، مع حماية بياناته.

### سؤال 2: ما هو الباني (`Constructor`) وما أنواعه؟
**نموذج الإجابة:**
1. التعريف: تابع خاص يحمل اسم الصف نفسه، يُستدعى تلقائيًا عند إنشاء غرض جديد.
2. المكونات/الشروط: لا يمتلك نوع إرجاع، يمكن أن يأخذ وسطاء أو لا.
3. مثال: `student(long id,string fn,string ln,float p){...}`
4. متى نستخدم: أنواعه ثلاثة — الافتراضي (بدون وسطاء)، ذو الوسطاء (لبناء الغرض بقيم محددة)، والناسخ (لنسخ غرض موجود).

### سؤال 3: ما هو الهادم (`Destructor`) ومتى يُستدعى؟
**نموذج الإجابة:**
1. التعريف: تابع خاص باسم الصف مسبوقًا بـ `~`، يُستدعى تلقائيًا عند حذف الغرض من الذاكرة.
2. المكونات/الشروط: لا يمتلك وسطاء ولا نوع إرجاع.
3. مثال: `~student(){ cout<<"Destroying..."; }`
4. متى نستخدم: يُستدعى في نهاية مجال رؤية الغرض المحلي، أو نهاية البرنامج للأغراض العامة/الساكنة، لتنظيف الموارد المستخدمة.

### سؤال 4: ما هو المؤشر `this` وما وظيفته؟
**نموذج الإجابة:**
1. التعريف: مؤشر خاص ضمن `++C` يشير إلى الغرض الذي استدعى التابع الحالي.
2. المكونات/الشروط: متاح تلقائيًا داخل أي تابع عضو غير ساكن (`non-static`).
3. مثال: `this->id = id;` لتمييز حقل الصف عن وسيط محلي بنفس الاسم.
4. متى نستخدم: عند تعارض الأسماء بين الوسطاء والحقول، أو للإشارة الصريحة للغرض الحالي.

### سؤال 5: ما هي `Encapsulation` (التغليف) ولماذا هي مهمة؟
**نموذج الإجابة:**
1. التعريف: مبدأ إخفاء تفاصيل التنفيذ الداخلي للغرض، وإظهار واجهة عامة محدودة فقط للتعامل معه.
2. المكونات/الشروط: يُطبَّق عبر `private` للحقول/التوابع الداخلية، و`public` للواجهة.
3. مثال: إخفاء طريقة تخزين بيانات الطالب، والسماح فقط بالوصول عبر `getters`/`setters`.
4. متى نستخدم: دائمًا في التصميم الجيد للصفوف، لحماية البيانات وتسهيل الصيانة المستقبلية دون كسر الكود المعتمد على الصف.

### سؤال 6: ما هي `Getters` و`Setters` وما الفرق بينهما؟
**نموذج الإجابة:**
1. التعريف: توابع عامة (`public`) تُستخدم للوصول الآمن إلى الحقول الخاصة (`private`).
2. المكونات/الشروط: `getter` يعيد قيمة الحقل (قراءة فقط)، `setter` يستقبل قيمة جديدة ويعدّل الحقل (مع إمكانية التحقق).
3. مثال: `long getId(){return id;}` و`void setId(long v){id=v;}`
4. متى نستخدم: كلما احتجنا لتمكين الوصول الخارجي المضبوط إلى حقل خاص دون كشفه مباشرة.

### سؤال 7: لماذا يجب تجنب إعادة الحقول الخاصة بالمرجع من `getter`؟
**نموذج الإجابة:**
1. التعريف: "مشكلة المرجع" تحدث عندما يعيد `getter` مرجعًا (`&`) لحقل `private` بدل نسخة منه.
2. المكونات/الشروط: أي متحول يستقبل هذا المرجع يصبح اسمًا بديلًا للحقل الأصلي.
3. مثال: `long& getId(){return id;}` ثم `long& x=c.getId(); x=10;` يعدّل الحقل مباشرة.
4. متى نستخدم: يجب تجنب هذا النمط دائمًا للحقول الحساسة، والاكتفاء بالإعادة بالقيمة للحفاظ على مبدأ التغليف.

### سؤال 8: كيف يتم فصل الكود عن الواجهة في `++C` ولماذا؟
**نموذج الإجابة:**
1. التعريف: تقسيم تعريف الصف إلى ملف رأس (`.h`) يحتوي التصريحات فقط، وملف تنفيذ (`.cpp`) يحتوي المنطق البرمجي الفعلي.
2. المكونات/الشروط: يُستخدم عامل تحليل المجال `::` لربط جسم التابع باسم الصف في ملف `.cpp`.
3. مثال: `student.h` يحتوي `void changeMark(int,int);` بينما `student.cpp` يحتوي `void student::changeMark(int i,int v){...}`.
4. متى نستخدم: عند الرغبة في توزيع مكتبة برمجية دون كشف الكود المصدري الأصلي (نرسل `.h` مع ملف تنفيذي مُترجَم).
