## الجزء الأول: الشرح التفصيلي (سطر بسطر / فقرة بفقرة)

### 1. ماهي السجلات

#### النص الأصلي يقول:
> "السجلات هي نمط بيانات معقد يعتمد على تعريف مجموعة من الحقول ضمن نفس المتحول"

#### الشرح المبسّط:
تخيل أنك تريد تمثيل عدد عقدي (`complex number`) له جزء حقيقي وجزء تخيلي. بدل أن تُعرّف متحولين منفصلين `double x` و`double y` وتبقى تربط بينهما يدوياً في كل مكان بالبرنامج، يعطيك `struct` طريقة لتغليف الاثنين معاً في نمط بيانات واحد جديد اسمه `Complex`. أي أن السجل هو "صندوق" يحوي عدة حقول (`fields`) من أنواع مختلفة أو متماثلة، وتتعامل مع الصندوق كوحدة واحدة.

**لماذا؟** لأن كثيراً من الأفكار في الواقع مركّبة من عدة قيم مرتبطة منطقياً (نقطة إحداثيات، طالب له اسم وعلامة، عدد عقدي له جزء حقيقي وتخيلي)، والسجل يسمح للمبرمج ببناء نمط بيانات مخصص يعكس هذا الترابط بدل التعامل مع متحولات مبعثرة.

#### 💡 التشبيه:
> فكّر ببطاقة هوية شخصية: هي وثيقة واحدة، لكنها تحوي عدة حقول (الاسم، الرقم الوطني، تاريخ الميلاد). لا أحد يحمل ثلاث بطاقات منفصلة لكل معلومة.
> **وجه الشبه:** بطاقة الهوية = السجل (`struct`) / كل معلومة على البطاقة = حقل (`field`) داخل السجل

```cpp
// Defining a record type named Complex
struct Complex
{
    double x; // real part
    double y; // imaginary part
};
```

#### شرح كل سطر:
1. `struct Complex` → بداية تعريف نمط بيانات جديد باسم `Complex` — الكلمة المفتاحية `struct` تُعلم المترجم أننا ننشئ نمطاً مركّباً
2. `{` → بداية جسم السجل الذي سيحوي الحقول
3. `double x;` → حقل من نوع `double` يمثل الجزء الحقيقي
4. `double y;` → حقل من نوع `double` يمثل الجزء التخيلي
5. `};` → نهاية تعريف السجل — لاحظ الفاصلة المنقوطة الإلزامية بعد `}`

**الناتج المتوقع:**
> لا يوجد ناتج تنفيذي هنا؛ هذا مجرد تعريف نمط بيانات جديد لم يُستخدم بعد

### 1.1. أجزاء تعريف السجل

#### النص الأصلي يقول:
> "نستخدم الكلمة المفتاحية `struct` لنعلم المترجم أننا نريد أن ننشئ سجلاً خاصاً بنا يليها اسم النمط... يبدأ تعريف محتوى السجل (حقوله) بالرمز `{` وينتهي بالرمز `}`... نعرف حقلان هما x (الجزء الحقيقي) و y (الجزء العقدي)... يتم انهاء التعريف بالرمز `;`"

#### الشرح المبسّط:
تعريف السجل له بنية ثابتة يجب اتباعها حرفياً:
1. الكلمة المفتاحية `struct`
2. اسم النمط الجديد (هنا `Complex`) — هذا الاسم أصبح من الآن فصاعداً نوع بيانات (`data type`) يمكن استخدامه مثل `int` أو `double`
3. قوس فتح `{` ثم قائمة الحقول، كل حقل له نوعه واسمه المنتهي بفاصلة منقوطة
4. قوس إغلاق `}` **متبوع إلزاماً بفاصلة منقوطة `;`** — وهذه نقطة يخطئ بها الكثير من الطلاب لأنها تختلف عن نهاية التابع أو الحلقة

**لماذا؟** الفاصلة المنقوطة بعد `}` ضرورية لأن تعريف السجل من الناحية النحوية هو جملة تعريف نوع (`type declaration statement`)، تماماً مثل `int x;` — والمترجم يحتاج علامة واضحة لنهاية الجملة.

#### مهم للامتحان ⚠️:
> نسيان الفاصلة المنقوطة بعد `};` هو من أشيع أخطاء الترجمة (`compile error`) عند تعريف السجلات لأول مرة.

### 1.2. استخدام السجل كنوع بيانات

#### النص الأصلي يقول:
> "تفيدنا السجلات بتجميع المتحولات المرتبطة بنفس الهدف في حزمة واحدة فقط. لاستخدام السجل نستخدمه كنوع بيانات مثلاً `Complex number`"

#### الشرح المبسّط:
بمجرد تعريف `struct Complex`، أصبح `Complex` نوع بيانات جديد بالضبط مثل `int` أو `double`. لذلك نستطيع تعريف متحولات منه: `Complex number;` تُنشئ متحولاً اسمه `number` من نوع `Complex`، ويحوي بداخله الحقلين `x` و `y` تلقائياً.

للوصول إلى حقل داخل السجل نستخدم عامل النقطة `.` (`dot operator`): `number.x` و `number.y`.

**لماذا نستخدم النقطة؟** لأن الحقل ليس متحولاً مستقلاً بذاته، بل هو "جزء من" الكائن `number`؛ فعامل `.` يعني حرفياً "اذهب داخل `number` وأحضر لي الحقل `x`".

```cpp
struct Complex{
    double x;
    double y;
};
int main(){
    Complex number; // define a variable of type Complex
    cin >> number.x >> number.y; // read each field separately
    cout << number.x <<
    ((number.y >= 0)? " + " : " - ")
    << number.y << "i" << endl; // print each field separately
    return 0;
}
```

#### شرح كل سطر:
1. `Complex number;` → تعريف متحول `number` من نوع `Complex`؛ الحقلان `x` و `y` غير مهيأين بعد
2. `cin >> number.x >> number.y;` → إدخال قيمتين متتاليتين من المستخدم، الأولى تُخزَّن في `number.x` والثانية في `number.y`
3. `cout << number.x << ...` → طباعة الجزء الحقيقي، ثم إشارة `+` أو `-` حسب إشارة الجزء التخيلي، ثم قيمته المطلقة (فكرياً) متبوعة بالحرف `i`

**المكتبات المطلوبة (Imports):**
> `#include <iostream>` و `using namespace std;`

**الناتج المتوقع (لقطة الشاشة):**
> عند إدخال `1 2` يطبع البرنامج: `1 + 2i`

### 2. تعريف السجلات (التهيئة بقيم ابتدائية)

#### النص الأصلي يقول:
> "تبين لنا كيفية تعريف سجل. كيف يمكننا تهيئته بقيم ابتدائية؟"

#### الشرح المبسّط:
مثلما نهيئ متحولاً بسيطاً بقيمة ابتدائية `int x = 5;`، يمكننا تهيئة سجل كامل مباشرة عند تعريفه باستخدام قائمة تهيئة `{}` (`initializer list`)، بدل استخدام `cin` أو الإسناد الحقل تلو الآخر.

```cpp
int main(){
    Complex number1={1,2}; // x=1, y=2 (both fields initialized)
    Complex number2={1};   // x=1, y=0 (missing field defaults to 0)
    cout << number1.x <<
    ((number1.y >= 0)? " + " : " - ")
    << number1.y << "i" << endl; // prints 1+2i
    cout << number2.x <<
    ((number2.y >= 0)? " + " : " - ")
    << number2.y << "i" << endl; // prints 1+0i
}
```

#### شرح كل سطر:
1. `Complex number1={1,2};` → إسناد القيمة `1` للحقل الأول `x` والقيمة `2` للحقل الثاني `y` بالترتيب الذي عُرّفت فيه الحقول داخل `struct`
2. `Complex number2={1};` → إسناد القيمة `1` للحقل الأول `x` فقط؛ الحقل الثاني `y` **لم يُذكر**
3. سطرا `cout` → طباعة كل سجل بصيغة العدد العقدي

#### مهم للامتحان ⚠️:
> "يتم المعالجة مثل المصفوفة، الحقول التي لا تُسند لها قيم تأخذ القيمة `0`" — تماماً كما في تهيئة المصفوفات `int arr[5]={1,2};` حيث تُهيَّأ بقية العناصر تلقائياً بالصفر.

#### 💡 التشبيه:
> املأ استمارة تسجيل فيها ثلاث خانات؛ إذا كتبت قيمة في الخانة الأولى فقط وتركت الباقي فارغاً، يفترض النظام أن قيمة الخانات الفارغة هي القيمة الافتراضية (صفر).
> **وجه الشبه:** الخانة الفارغة في الاستمارة = الحقل غير المذكور في `{}` → القيمة الافتراضية `0`

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** ماذا يطبع `Complex n3={};` بدون أي قيم بين القوسين؟
> **لماذا هذا مهم؟** لأنه يوضح أن التهيئة الفارغة تجعل كل الحقول تساوي القيمة الافتراضية لنوعها (`0` بالنسبة لـ `double`)، أي `0+0i`.

### 2.1. قيود تعريف الحقول داخل السجل

#### النص الأصلي يقول:
> "هنالك شروط محددة للتعريف الحقول ضمن سجل"

#### الشرح المبسّط:
لا يمكن أن يحوي السجل حقلاً من نوعه هو نفسه بشكل مباشر (`Complex xc;` داخل `struct Complex` **خطأ**)، لأن ذلك يعني أن حجم السجل يعتمد على حجم نفسه — تناقض منطقي لا نهائي (مثل صندوق يحوي نسخة كاملة من نفسه، والتي بدورها تحوي نسخة أخرى... إلى ما لا نهاية).

لكن يمكن تعريف **مؤشر** (`pointer`) على نفس نوع السجل (`Complex* xc;`)، لأن حجم المؤشر ثابت ومعروف (عنوان ذاكرة) بغض النظر عن حجم ما يشير إليه.

```cpp
// INVALID - cannot embed the struct inside itself
struct ComplexBad
{
    double x;
    double y;
    // Complex xc; // compile error: incomplete type
};

// VALID - a pointer to the same struct type is allowed
struct Complex
{
    double x;
    double y;
    Complex* xc; // pointer, not a full embedded object
};
```

#### الفهم الخاطئ الشائع ❌: 
تعريف حقل من نفس نوع السجل مباشرة (`Complex xc;` داخل `Complex`) يعمل مثل باقي الأنواع

#### الفهم الصحيح ✅:
فقط **مؤشر** لنفس نوع السجل مسموح (`Complex* xc;`)؛ وهذا هو الأساس الذي تُبنى عليه لاحقاً بنى مثل `Linked List`

### 2.2. طرق تعريف متحولات من نوع سجل

#### النص الأصلي يقول:
> "كل ما يلي صحيح"

#### الشرح المبسّط:
تماماً كباقي أنواع البيانات، يمكن تعريف متحول من نوع سجل بعدة أشكال: قيمة عادية، مصفوفة من السجلات، مؤشر يشير إلى سجل موجود، سجل مُنشأ ديناميكياً بـ `new`، مرجع (`reference`)، أو مصفوفة سجلات مُنشأة ديناميكياً.

```cpp
Complex a;                 // normal value
Complex array[10];         // array of 10 Complex records
Complex* ptr = &a;         // pointer to an existing record
Complex* ptrN = new Complex; // dynamically allocated record
Complex& ref = a;          // reference to an existing record
Complex* ptrArray = new Complex[10]; // dynamically allocated array
```

#### شرح كل سطر:
1. `Complex a;` → متحول عادي غير مهيأ من نوع `Complex`
2. `Complex array[10];` → مصفوفة ساكنة (`static array`) من 10 سجلات متتالية
3. `Complex* ptr = &a;` → مؤشر يخزّن عنوان المتحول `a` باستخدام عامل العنونة `&`
4. `Complex* ptrN = new Complex;` → تخصيص سجل جديد على الكومة (`heap`) باستخدام `new`، ويعيد مؤشراً إليه
5. `Complex& ref = a;` → مرجع، أي اسم بديل (`alias`) لنفس المتحول `a`، وليس نسخة منفصلة
6. `Complex* ptrArray = new Complex[10];` → تخصيص مصفوفة من 10 سجلات ديناميكياً على الكومة

#### جدول مقارنة سريع:
| الشكل | أين يُخزَّن؟ | يحتاج تحرير يدوي (`delete`)؟ |
| --- | --- | --- |
| `Complex a;` | `stack` | لا |
| `Complex array[10];` | `stack` | لا |
| `Complex* ptrN = new Complex;` | `heap` | نعم، `delete ptrN;` |
| `Complex* ptrArray = new Complex[10];` | `heap` | نعم، `delete[] ptrArray;` |

### 3. السجلات والتوابع

#### النص الأصلي يقول:
> "تعامل السجلات مع التوابع مثل أي نوع من أنواع البيانات"

#### الشرح المبسّط:
بما أن `Complex` أصبح نوع بيانات كامل، يمكن استخدامه كنوع لمعامل (`parameter`) في تابع، أو كنوع القيمة المُرجعة (`return type`)، تماماً مثل `int` أو `double`. يمكن تمرير السجل بالقيمة (`by value`)، بالمرجع (`by reference`)، أو بالمؤشر (`by pointer`) — ولكل طريقة أثر مختلف على الأداء والقدرة على التعديل.

```cpp
void enter(Complex& number){ // pass by reference: modifies the original
    cin >> number.x
        >> number.y;
}

void enter(Complex* number){ // pass by pointer: modifies via address
    cin >> number->x
        >> number->y;
}

void print(Complex number){ // pass by value: just reads a copy
    cout << number.x <<
    ((number.y >= 0)? " + " : " - ")
    << number.y << "i" << endl;
}

Complex create(){ // returns a new Complex record by value
    Complex c={0};
    return c;
}
```

#### شرح كل سطر:
1. `void enter(Complex& number)` → التابع يستقبل **مرجعاً** للسجل الأصلي؛ أي تعديل داخل التابع (`cin >>`) يؤثر مباشرة على المتحول المُمرَّر من `main`
2. `void enter(Complex* number)` → التابع يستقبل **مؤشراً**؛ للوصول إلى الحقول نستخدم السهم `->` بدل النقطة، لأننا نمر عبر عنوان أولاً
3. `void print(Complex number)` → التابع يستقبل **نسخة** كاملة من السجل (`by value`)؛ أي تعديل داخل التابع لا يؤثر على الأصل، لكنه أبطأ لأنه ينسخ كل الحقول
4. `Complex create()` → تابع يبني سجلاً محلياً ويعيده بالقيمة؛ عند `return c;` تُنسخ قيم `c` إلى المكان الذي استدعى التابع

#### ⚖️ المقايضة: تمرير بالقيمة مقابل تمرير بالمرجع/المؤشر

| | تمرير بالقيمة (`by value`) | تمرير بالمرجع/المؤشر (`by reference/pointer`) |
| --- | --- | --- |
| المزايا | أمان تام — لا يمكن للتابع تعديل الأصل بالخطأ | لا نسخ إضافي، وأسرع مع السجلات الكبيرة، ويستطيع تعديل الأصل |
| العيوب | ينسخ كل الحقول في كل استدعاء، أبطأ مع السجلات الكبيرة | يجب الحذر لأن التابع يستطيع تعديل بيانات المستدعي |
| متى تختاره | عندما تريد فقط قراءة القيم (`print`) دون تعديلها | عندما تريد تعديل القيم الأصلية (`enter`) أو تفادي النسخ |

#### مهم للامتحان ⚠️:
> عند تمرير سجل عبر **مؤشر**، الوصول للحقول يكون بالسهم `->` وليس بالنقطة `.` — لأن السهم يقوم أولاً بعملية `dereferencing` (فك الإشارة) ثم الوصول للحقل.

### 4. السجلات والمؤشرات

#### النص الأصلي يقول:
> "يمكننا تعريف مؤشر على سجل `Complex * ptr` لكن سيحدث لدينا اختلاف في كيفية الوصول إلى حقول السجل. يمكننا الوصول إلى حقول سجل باستخدام المؤشرات بطريقتين: السهم `->` أو معامل `*` مثل `(*ptr).x`"

#### الشرح المبسّط:
عندما يكون لدينا مؤشر يشير إلى سجل، لدينا طريقتان متكافئتان تماماً للوصول إلى حقوله:
1. **السهم** `ptr->x` — الصيغة المختصرة والأكثر استخداماً
2. **فك الإشارة ثم النقطة** `(*ptr).x` — الصيغة الصريحة التي توضح ما يحدث فعلياً خطوة بخطوة

**لماذا الأقواس إلزامية حول `*ptr`؟** لأن عامل النقطة `.` له أولوية تنفيذ (`precedence`) أعلى من عامل فك الإشارة `*`. فلو كتبنا `*ptr.x` بدون أقواس، سينفذ المترجم `ptr.x` أولاً (وهذا خطأ لأن `ptr` مؤشر وليس سجلاً مباشراً)، ثم يحاول فك إشارة النتيجة — وهذا يسبب خطأ ترجمة. الأقواس `(*ptr)` تجبر المترجم على فك الإشارة أولاً، أي "احصل على السجل الذي يشير إليه `ptr`"، **ثم** الوصول لحقله `x`.

```cpp
Complex a = {1, 2};
Complex* ptr = &a;

cout << ptr->x << endl;    // form 1: arrow operator
cout << (*ptr).x << endl;  // form 2: dereference then dot (both print 1)
```

#### جدول مقارنة سريع:
| الصيغة | المعنى | ملاحظة |
| --- | --- | --- |
| `ptr->x` | "اذهب إلى ما يشير إليه `ptr` وأحضر الحقل `x`" | الأشيع استخداماً في الكود الحقيقي |
| `(*ptr).x` | نفس المعنى تماماً، لكن مكتوبة صراحة | الأقواس إلزامية بسبب أولوية `.` على `*` |

#### 💡 التشبيه:
> تخيل `ptr` كورقة عليها عنوان منزل مكتوب عليها. `ptr->x` يعني "اذهب إلى العنوان المكتوب، وادخل الغرفة `x`" مباشرة بخطوة واحدة، بينما `(*ptr).x` هو نفس الشيء لكن موضّح كخطوتين: أولاً اذهب للمنزل `(*ptr)`، ثم ادخل الغرفة `.x`.
> **وجه الشبه:** الورقة التي تحمل العنوان = المؤشر `ptr` / الدخول إلى الغرفة = الوصول للحقل

### 5. التحميل الزائد للعمليات (Operator Overloading)

#### النص الأصلي يقول:
> "نحن نعلم أن `Complex` هو عدد عقدي يمكن تنفيذ العمليات الرياضية عليه، وبناءً على ذلك يفترض أن تعطي العملية التالية نتيجة... إلا أن المترجم يعطي أخطاءً لما؟ بكل بساطة، نحن من عرف العدد العقدي وليس المترجم وبالتالي هو لا يعرف كيف سينفذ عملية الجمع عليه"

#### الشرح المبسّط:
عندما نكتب `int a=1, b=2; cout << a+b;` يعمل الأمر لأن عامل `+` **معرَّف مسبقاً** في اللغة للأنواع الأساسية (`int`, `double`, ...). لكن `Complex` نوع بيانات **من صنعنا نحن**، والمترجم لا يملك أي فكرة فطرية عن معنى "جمع سجلين"؛ فمحاولة `c1+c2` أو `cout << c1` تعطي خطأ ترجمة لأن هذه العمليات **غير معرَّفة** لهذا النوع الجديد.

**الحل:** التحميل الزائد للعمليات (`operator overloading`)، أي أن نكتب نحن أنفسنا تابعاً خاصاً يعلّم المترجم كيف ينفذ عامل معين (`+`, `<<`, ...) عندما يُطبَّق على نوعنا الجديد.

```cpp
int main(){
    Complex c1={1,2};
    Complex c2={3,4};
    cout << c1+c2; // ERROR: compiler doesn't know how to add two Complex
}
```

#### الفهم الخاطئ الشائع ❌:
المترجم "يخمّن" تلقائياً أن جمع سجلين يعني جمع حقولهما المتناظرة

#### الفهم الصحيح ✅:
المترجم لا يخمّن شيئاً؛ كل عملية على نوع بيانات مخصص يجب أن يُعرّفها المبرمج صراحة عبر `operator overloading`

#### 💡 التشبيه:
> لو اخترعت لعبة جديدة بقطع شطرنج جديدة، لا يمكنك أن تتوقع من الحكم أن "يعرف" تلقائياً قواعد حركة هذه القطع الجديدة — يجب أن تكتب له القواعد أولاً.
> **وجه الشبه:** القطعة الجديدة = نوع البيانات `Complex` / كتابة القواعد للحكم = تعريف `operator overloading`

### 5.1. الشكل الأول: تابع حر (Free Function) بمعامل بالمرجع

#### النص الأصلي يقول:
> "أن نوضح لـ C++ كيفية تنفيذ هذه العمليات على السجلات — الشكل الأول"

#### الشرح المبسّط:
نكتب تابعاً عادياً خارج السجل، اسمه `operator+`، يستقبل مرجعين للسجلين المطلوب جمعهما ويعيد النتيجة. الكلمة المفتاحية `operator` متبوعة بالرمز المطلوب تحميله (`+`) تخبر المترجم أن هذا التابع يجب استدعاؤه كلما ظهر الرمز `+` بين نوعين من `Complex`.

```cpp
struct Complex
{
    double x;
    double y;
};

Complex& operator+(Complex& n1, Complex n2){ // free function, overloads +
    n1.x += n2.x;
    n1.y += n2.y;
    return n1;
}

ostream& operator<<(ostream& cout, Complex& n){ // overloads << for printing
    cout << n.x << ((n.y >= 0)? " + " : " - ") << n.y << "i";
    return cout;
}
```

#### شرح كل سطر:
1. `Complex& operator+(Complex& n1, Complex n2)` → تعريف تابع باسم خاص `operator+` يستقبل `n1` بالمرجع (سيُعدَّل مباشرة) و`n2` بالقيمة، ويعيد مرجعاً لنفس `n1` بعد التعديل
2. `n1.x += n2.x;` و `n1.y += n2.y;` → جمع الحقول المتناظرة، تماماً كما نتوقع من الجمع الرياضي للأعداد العقدية
3. `return n1;` → إعادة `n1` بعد أن أصبحت تحمل الناتج
4. `ostream& operator<<(ostream& cout, Complex& n)` → تحميل عامل `<<` لكي يفهم `cout` كيفية طباعة `Complex` مباشرة بصيغة `x + yi`

#### نقطة مهمة ⚠️:
> لاحظ أن هذا الشكل يُعدّل السجل الأول `n1` نفسه (لأنه بالمرجع)! أي أن `c1+c2` تُغيّر قيمة `c1` الأصلية أيضاً — سلوك مختلف عن الجمع الرياضي العادي الذي لا يُغيّر أياً من الطرفين. هذا تصميم مبسّط تعليمي، وليس بالضرورة الأسلوب "الصحيح" في برنامج حقيقي.

### 5.2. الشكل الثاني: تابع عضو (Member Function) باستخدام `this`

#### النص الأصلي يقول:
> "الشكل الثاني"

#### الشرح المبسّط:
بدل تابع حر خارج السجل، نضع `operator+` **داخل** السجل نفسه كتابع عضو (`member function`). هنا لسنا بحاجة لتمرير `n1` كمعامل — لأن `n1` هي السجل الذي استُدعي التابع من خلاله، ويُشار إليها ضمنياً بالمؤشر الخاص `this`.

```cpp
struct Complex
{
    double x;
    double y;
    Complex& operator+(Complex n2){ // member function: only takes the "other" operand
        this->x += n2.x;
        this->y += n2.y;
        return *this; // will talk about this later
    }
};
```

#### شرح كل سطر:
1. `Complex& operator+(Complex n2)` → تابع عضو يستقبل معاملاً واحداً فقط `n2` (الطرف الثاني)؛ الطرف الأول هو السجل نفسه الذي استُدعي التابع من خلاله
2. `this->x += n2.x;` → `this` هو مؤشر ضمني يشير إلى السجل الحالي (نفس فكرة `c1` في `c1+c2`)؛ نصل لحقوله بالسهم `->` لأنه مؤشر
3. `return *this;` → إعادة **قيمة** السجل الحالي (`fك إشارة this`)، أي `c1` نفسه بعد التعديل

#### 💡 التشبيه:
> `this` يشبه أن تكتب "أنا" داخل رسالة بدل كتابة اسمك بالكامل في كل جملة — إنه إشارة ضمنية للكائن الذي تنفّذ العملية بداخله حالياً.
> **وجه الشبه:** كلمة "أنا" في الرسالة = `this` داخل التابع العضو

### 5.3. الشكل الثالث: تابع صديق (Friend Function) داخل السجل

#### النص الأصلي يقول:
> "الشكل الثالث"

#### الشرح المبسّط:
عامل `<<` لا يمكن أن يكون تابعاً عضواً عادياً بسهولة لأن الطرف الأيسر ليس من نوع `Complex` بل من نوع `ostream` (أي `cout`). لذلك نستخدم مفهوم التابع الصديق (`friend`)، الذي يُكتب **داخل** تعريف السجل (لذلك يستطيع الوصول لحقوله الخاصة لاحقاً) لكنه من الناحية الفنية **ليس** تابعاً عضواً في السجل.

```cpp
struct Complex
{
    double x;
    double y;
    friend ostream& operator<<(ostream& cout, Complex& n){ // friend function
        cout << n.x << ((n.y >= 0)? " + " : " - ") << n.y << "i";
        return cout;
    }
    // we will talk about friends later (if you have any)
};

Complex& operator+(Complex& n1, Complex n2){ // + stays a free function here
    n1.x += n2.x;
    n1.y += n2.y;
    return n1;
}
```

#### شرح كل سطر:
1. `friend ostream& operator<<(...)` → الكلمة المفتاحية `friend` تسمح لهذا التابع بالوصول إلى الحقول الخاصة للسجل رغم أنه غير مرتبط به كتابع عضو حقيقي
2. باقي الأسطر → نفس منطق طباعة العدد العقدي كما سبق

#### ملاحظة:
> في هذا الشكل، عامل `+` بقي **تابعاً حراً خارج السجل** كما في الشكل الأول، بينما عامل `<<` أصبح تابعاً صديقاً **داخل** السجل. الشكلان يمكن مزجهما.

### 5.4. الشكل الرابع: دمج التابع الصديق مع التابع العضو

#### النص الأصلي يقول:
> "الشكل الرابع"

#### الشرح المبسّط:
هذا الشكل يجمع بين الحلين السابقين داخل نفس تعريف السجل: `operator<<` كتابع صديق (لأن طرفه الأيسر `ostream`)، و`operator+` كتابع عضو يستخدم `this` (لأن طرفيه من نوع `Complex`). هذا هو الشكل الأكثر اكتمالاً واستخداماً عملياً.

```cpp
struct Complex
{
    double x;
    double y;
    friend ostream& operator<<(ostream& cout, Complex& n){ // friend: for cout
        cout << n.x << ((n.y >= 0)? " + " : " - ") << n.y << "i";
        return cout;
    }
    // we will talk about friends later (if you have any)
    Complex& operator+(Complex n2){ // member: for Complex + Complex
        this->x += n2.x;
        this->y += n2.y;
        return *this; // will talk about this later
    }
};
```

#### ⚙️ الخطوات / الخوارزمية: اختيار شكل التحميل الزائد المناسب

> ما هدف هذه العملية؟ تحديد أي نمط (تابع حر / تابع عضو / تابع صديق) نستخدم عند تحميل عامل معين

```algorithm
1 | تحقق من نوع الطرف الأيسر للعامل | فحص المعاملات | إذا كان الطرف الأيسر من نوعنا (Complex) يمكن استخدام تابع عضو
2 | تحقق إذا كان الطرف الأيسر نوعاً خارجياً | فحص المعاملات | مثل ostream في حالة << — لا يمكن جعله تابعاً عضواً في Complex
3 | استخدم friend عند الحاجة للوصول لحقول خاصة من خارج السجل | friend keyword | يسمح لتابع حر بالوصول لحقول السجل الخاصة
4 | اكتب جسم العملية | operator keyword + رمز العامل | نفّذ المنطق الرياضي/المنطقي المطلوب
```

#### نقاط التنفيذ:
- عامل `<<` دائماً تقريباً يُكتب كـ `friend` أو تابع حر، لأن الطرف الأيسر (`ostream`) خارج عن سيطرتنا
- عامل `+` بين نوعين من نفس السجل يمكن كتابته كتابع عضو (`this` يمثل الطرف الأيسر) أو كتابع حر عادي

### 6. التحميل الزائد للعمليات — تمرين مفتوح

#### النص الأصلي يقول:
> "عليكم الآن تحقيق الطرح والقسمة والضرب"

#### الشرح المبسّط:
المحاضرة تنتهي بتكليف الطالب بتطبيق نفس المبدأ (`operator overloading`) على بقية العمليات الحسابية للعدد العقدي: الطرح `-`، القسمة `/`، والضرب `*`. سنغطي حلاً كاملاً لهذا التكليف في قسم "الكود الكامل" وقسم "تمارين إضافية" أدناه، باستخدام قواعد الرياضيات للأعداد العقدية:
- الطرح: `(a+bi) - (c+di) = (a-c) + (b-d)i`
- الضرب: `(a+bi) * (c+di) = (ac-bd) + (ad+bc)i`
- القسمة: `(a+bi) / (c+di) = [(ac+bd) + (bc-ad)i] / (c²+d²)`

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** لماذا تحتاج القسمة إلى صيغة أعقد من الجمع والطرح؟
> **لماذا هذا مهم؟** لأن قسمة عددين عقديين رياضياً تتطلب "ضرب المقام بمرافقه" لجعل المقام حقيقياً — وهذا يوضح أن التحميل الزائد لا يعني فقط "طبّق العامل على كل حقل على حدة"، بل يجب اتباع المنطق الرياضي/المفاهيمي الصحيح لكل عملية.

## الجزء الثاني: ملخص منظم

### أهم التعاريف والمفاهيم
| المصطلح | التعريف | مثال/ملاحظة |
| --- | --- | --- |
| `struct` | كلمة مفتاحية لتعريف نمط بيانات جديد يجمع عدة حقول | `struct Complex {...};` |
| `field` (حقل) | متحول معرَّف داخل السجل | `double x;` |
| `dot operator .` | الوصول لحقل من متحول سجل عادي | `number.x` |
| `arrow operator ->` | الوصول لحقل من خلال مؤشر على سجل | `ptr->x` |
| `dereference *` | فك إشارة المؤشر للوصول للسجل ذاته | `(*ptr).x` |
| `this` | مؤشر ضمني داخل التابع العضو يشير إلى الكائن الحالي | `this->x` |
| `operator overloading` | إعادة تعريف سلوك عامل (`+`, `<<`, ...) لنوع بيانات مخصص | `operator+(Complex n2)` |
| `friend` | كلمة مفتاحية تمنح تابعاً حراً حق الوصول للحقول الخاصة للسجل | `friend ostream& operator<<(...)` |

### المكونات الرئيسية (مرجع سريع)
| الأداة | الوظيفة | ملاحظة |
| --- | --- | --- |
| `{}` بعد اسم السجل | تحديد بداية ونهاية جسم الحقول | يُغلق دائماً بـ `};` |
| `{val1, val2, ...}` | تهيئة سجل بقيم ابتدائية بترتيب الحقول | الحقول الناقصة تُهيَّأ بـ `0` |
| `new StructName` | تخصيص سجل ديناميكياً على `heap` | يعيد مؤشراً؛ يحتاج `delete` لاحقاً |
| `&` | أخذ عنوان متحول أو تعريف مرجع | `&a` أو `Complex& ref = a;` |

### جداول مقارنات سريعة
| المقارنة | الأول | الثاني | الفرق |
| --- | --- | --- | --- |
| الوصول للحقول | `number.x` (سجل مباشر) | `ptr->x` (عبر مؤشر) | النقطة تُستخدم مع الكائن مباشرة، السهم يفك الإشارة أولاً ثم يصل للحقل |
| تمرير السجل للتابع | `by value` | `by reference` / `by pointer` | بالقيمة ينسخ كامل السجل ولا يعدّل الأصل؛ بالمرجع/المؤشر لا نسخ ويمكن التعديل على الأصل |
| تعريف حقل من نفس نوع السجل | `Complex xc;` (مباشر) | `Complex* xc;` (مؤشر) | المباشر **خطأ ترجمة** (حجم غير محدد)، المؤشر **صحيح** (حجمه ثابت) |
| موقع `operator overloading` | تابع حر (`free function`) خارج السجل | تابع عضو (`member function`) داخل السجل | التابع الحر يستقبل كل المعاملات صراحة؛ التابع العضو يستخدم `this` كطرف أول ضمني |

### قاموس المصطلحات
| الفئة | المصطلحات |
| --- | --- |
| تعريف السجل | `struct`, `field`, `initializer list` |
| الوصول للبيانات | `dot operator .`, `arrow operator ->`, `dereference *` |
| التوابع | `pass by value`, `pass by reference`, `pass by pointer`, `return by value` |
| التحميل الزائد | `operator overloading`, `friend`, `this`, `member function`, `free function` |

### أبرز النقاط الذهبية
1. السجل (`struct`) هو نمط بيانات مخصص يجمّع عدة حقول تحت اسم واحد
2. تعريف السجل يجب أن ينتهي بفاصلة منقوطة بعد `}`
3. الحقول غير المذكورة في تهيئة `{}` تأخذ القيمة الافتراضية `0`
4. لا يمكن تضمين السجل داخل نفسه مباشرة، لكن يمكن تضمين مؤشر عليه
5. الوصول لحقول السجل: `.` للكائن المباشر، `->` أو `(*ptr).` للمؤشر
6. السجلات تتعامل مع التوابع تماماً كباقي الأنواع (معامل، قيمة معادة)
7. المترجم لا يعرف تلقائياً كيف ينفذ عوامل رياضية (`+`, `<<`, ...) على أنواع مخصصة — يجب تحميلها زائداً (`operator overloading`)
8. عامل `<<` عادة يُكتب كتابع صديق (`friend`) أو حر، بينما عوامل الحساب (`+`, `-`, ...) يمكن كتابتها كتابع عضو باستخدام `this`

### الأخطاء الشائعة عند الطلاب ⚠️
| الخطأ | التصحيح |
| --- | --- |
| نسيان الفاصلة المنقوطة بعد `};` عند تعريف السجل | تذكّر أن تعريف السجل جملة تعريف نوع، تحتاج `;` في النهاية دائماً |
| استخدام `.` بدل `->` عند الوصول لحقل من مؤشر | استخدم السهم `->` أو `(*ptr).x` مع الأقواس الإلزامية |
| تضمين السجل داخل نفسه مباشرة `Complex xc;` | استخدم مؤشراً بدلاً منه `Complex* xc;` |
| افتراض أن `c1+c2` تعمل تلقائياً على سجل مخصص | يجب تعريف `operator+` صراحة لهذا النوع |
| كتابة `*ptr.x` بدون أقواس | اكتب `(*ptr).x` لأن `.` أعلى أولوية من `*` |

### خطوات وإجراءات المحاضرة
> كل عملية أو إجراء ورد في المحاضرة — مكتوب كـ `algorithm` block مستقل بالترتيب.

#### ⚙️ الخطوات / الخوارزمية: تعريف سجل جديد

> ما هدف هذه العملية؟ إنشاء نمط بيانات مخصص يجمع عدة حقول مرتبطة

```algorithm
1 | اكتب الكلمة المفتاحية | struct | يُعلم المترجم ببدء تعريف نوع مركّب جديد
2 | اكتب اسم النمط | اسم مخصص (مثلاً Complex) | يصبح هذا الاسم نوع بيانات قابل للاستخدام لاحقاً
3 | افتح القوس | { | بداية جسم الحقول
4 | عرّف الحقول | نوع + اسم لكل حقل | كل حقل ينتهي بفاصلة منقوطة
5 | أغلق القوس مع فاصلة منقوطة | }; | إنهاء تعريف السجل — إلزامي
```

#### نقاط التنفيذ:
- لا تنسَ الفاصلة المنقوطة بعد `}`
- لا يمكن تضمين نوع السجل نفسه كحقل مباشر داخله

#### ⚙️ الخطوات / الخوارزمية: تهيئة سجل بقيم ابتدائية

> ما هدف هذه العملية؟ إسناد قيم للحقول مباشرة عند تعريف المتحول دون استخدام `cin` أو إسناد منفصل لكل حقل

```algorithm
1 | عرّف المتحول من نوع السجل | اسم النوع | مثال: Complex number1
2 | ضع علامة المساواة والقوس المعقوف | = { | بداية قائمة التهيئة
3 | اكتب القيم مفصولة بفواصل | val1, val2, ... | بنفس ترتيب تعريف الحقول داخل struct
4 | أغلق القائمة | } | القيم غير المذكورة تُهيَّأ تلقائياً بـ 0
```

#### نقاط التنفيذ:
- ترتيب القيم يجب أن يطابق ترتيب تعريف الحقول داخل `struct`
- يمكن ترك قيم متأخرة دون ذكر؛ ستأخذ القيمة الافتراضية `0`

#### ⚙️ الخطوات / الخوارزمية: الوصول لحقل سجل عبر مؤشر

> ما هدف هذه العملية؟ قراءة أو تعديل حقل داخل سجل تمت الإشارة إليه بمؤشر

```algorithm
1 | تحقق من وجود مؤشر صالح يشير إلى سجل | pointer variable | مثل Complex* ptr = &a;
2 | اختر إحدى الصيغتين | -> أو (*ptr). | كلا الصيغتين متكافئتان تماماً
3 | نفّذ الوصول للحقل | ptr->x أو (*ptr).x | يعيد قيمة الحقل أو يسمح بتعديله
```

#### نقاط التنفيذ:
- عند استخدام `*` مباشرة يجب وضع الأقواس حول `*ptr` لأن `.` له أولوية أعلى
- السهم `->` هو الصيغة الأكثر شيوعاً واختصاراً في الكود الحقيقي

#### ⚙️ الخطوات / الخوارزمية: تحميل عامل زائد (`operator overloading`) لسجل

> ما هدف هذه العملية؟ تعليم المترجم كيف ينفذ عاملاً رياضياً/منطقياً معيناً على نوع سجل مخصص

```algorithm
1 | حدد العامل المطلوب تحميله | +, -, *, /, << ... | كل عامل يحتاج تعريفاً منفصلاً
2 | حدد شكل التابع المناسب | free function / member function / friend | حسب نوع الطرف الأيسر للعامل
3 | اكتب رأس التابع بالصيغة operator{symbol} | operator+ , operator<< ... | اسم خاص يتعرف عليه المترجم
4 | نفّذ منطق العملية على الحقول | += , -= ... | طبّق القاعدة الرياضية/المنطقية الصحيحة لكل عملية
5 | أعد النتيجة المناسبة | return | حسب توقيع التابع (مرجع أو قيمة)
```

#### نقاط التنفيذ:
- عوامل مثل `<<` تحتاج غالباً `friend` أو تابع حر لأن الطرف الأيسر (`ostream`) خارج نوعنا
- عوامل الحساب بين نفس النوع يمكن كتابتها كتابع عضو باستخدام `this` بدل المعامل الأول

### أنماط الأكواد والبنى المتكررة
> كل بنية كود ظهرت أكثر من مرة أو تمثّل أسلوباً يجب أن يُحفظ.

| النمط | البنية الأساسية | متى تستخدمه |
| --- | --- | --- |
| تعريف سجل | `struct Name{ type field; ... };` | عند تجميع بيانات مرتبطة تحت نوع واحد |
| تهيئة سجل | `Type var = {v1, v2, ...};` | عند معرفة القيم الابتدائية وقت التعريف |
| قراءة/طباعة حقول العدد العقدي | `cin >> n.x >> n.y;` / `cout << n.x << (sign) << n.y << "i";` | نمط متكرر لكل سجل يمثل عدداً عقدياً |
| تابع عضو للتحميل الزائد | `Complex& operatorX(Complex n2){ this->field OP= n2.field; return *this; }` | عند تحميل عامل حسابي بين سجلين من نفس النوع |
| تابع صديق للطباعة | `friend ostream& operator<<(ostream& cout, Complex& n){ ...; return cout; }` | عند تحميل `<<` لطباعة سجل مباشرة عبر `cout` |

### أنماط التعامل والسلوك
> كيف يتصرف النظام / الخوارزمية في سيناريوهات مختلفة.

| السيناريو | التعامل الصحيح | لماذا؟ |
| --- | --- | --- |
| تهيئة سجل بقيم أقل من عدد الحقول | الحقول المتبقية تُهيَّأ تلقائياً بـ `0` | نفس سلوك تهيئة المصفوفات الجزئية في C++ |
| محاولة تعريف حقل من نفس نوع السجل مباشرة | خطأ ترجمة (`compile error`) | نوع غير مكتمل الحجم (`incomplete type`) |
| استخدام عامل غير مُحمَّل زائداً (مثل `+`) على سجل مخصص | خطأ ترجمة | المترجم لا يملك تعريفاً افتراضياً لعوامل الأنواع المخصصة |
| تمرير سجل بالقيمة إلى تابع وتعديله داخل التابع | لا يتأثر السجل الأصلي في المستدعي | لأن التابع يعمل على نسخة منفصلة تماماً |
| تمرير سجل بالمرجع أو بالمؤشر وتعديله داخل التابع | يتأثر السجل الأصلي في المستدعي | لا يوجد نسخ؛ التابع يعمل مباشرة على نفس الذاكرة |

### الأفكار الرئيسية الشاملة
> أي فكرة محورية في المحاضرة لم تُغطَّ في الأقسام أعلاه.

- الفكرة المحورية للمحاضرة كاملة هي أن `struct` هو خطوة تمهيدية نحو مفهوم `class` (الذي سيُشرح لاحقاً في المادة)؛ فكل ما تعلمناه هنا (الحقول، التوابع الأعضاء، `this`, `friend`, `operator overloading`) هو أساس مباشر للبرمجة كائنية التوجه (`Object-Oriented Programming`).
- ملاحظة الشريحة الأخيرة "عليكم الآن تحقيق الطرح والقسمة والضرب" تعني أن المحاضرة تعليمية تفاعلية، وتترك تطبيق بقية العمليات كتمرين — وقد غطينا حلاً كاملاً لها في قسم الكود الكامل أدناه.

## الجزء الثالث: أسئلة اختيار من متعدد (MCQ)

> **16 سؤالاً** — مستوى: متوسط/صعب. التوزيع: مقارنات 25%، سيناريو كود 35%، تطبيق 30%، تتبع خوارزمية 10%.

### السؤال 1 (سهل)
ما الرمز الذي يجب أن يتبع `}` مباشرة عند إنهاء تعريف `struct`؟
أ) `.`  ب) `,`  ج) `;`  د) لا شيء
**الإجابة الصحيحة: ج**
**التعليل:** تعريف السجل هو جملة تعريف نوع بيانات، ويجب أن تنتهي بفاصلة منقوطة `;` مثل أي جملة أخرى في C++. الخيار (أ) و(ب) غير صالحين نحوياً هنا، و(د) يسبب خطأ ترجمة لأن الجملة تبقى غير منتهية.

### السؤال 2 (متوسط)
ماذا يطبع الكود التالي؟
```cpp
struct Point{ double x; double y; };
int main(){
    Point p={5};
    cout << p.x << " " << p.y;
}
```
أ) `5 5`  ب) `5 0`  ج) خطأ ترجمة  د) `0 5`
**الإجابة الصحيحة: ب**
**التعليل:** القيمة `5` تُسند للحقل الأول `x` حسب ترتيب التعريف، والحقل الثاني `y` غير المذكور يأخذ القيمة الافتراضية `0`. الخيار (أ) يفترض خطأً أن القيمة الواحدة تنطبق على كل الحقول، و(د) يعكس الترتيب خطأً، و(ج) خطأ لأن هذه التهيئة الجزئية صحيحة نحوياً.

### السؤال 3 (متوسط)
أي الأسطر التالية **خاطئة** إذا كان `Complex` نوع `struct` معرَّفاً مسبقاً؟
أ) `Complex* p = new Complex;`  ب) `Complex a; Complex& r = a;`  ج) `struct Complex{ double x; Complex xc; };`  د) `Complex arr[5];`
**الإجابة الصحيحة: ج**
**التعليل:** لا يمكن أن يحوي السجل حقلاً من نفس نوعه مباشرة لأن حجمه يصبح غير محدد (تناقض منطقي)؛ الحل الصحيح هو استخدام مؤشر `Complex* xc;`. باقي الخيارات (أ، ب، د) كلها صيغ صحيحة لتعريف متحولات من نوع سجل كما وضّحت المحاضرة.

### السؤال 4 (صعب)
ما ناتج الكود التالي؟
```cpp
struct Complex{ double x; double y; };
void print(Complex n){ n.x = 100; }
int main(){
    Complex c={1,2};
    print(c);
    cout << c.x;
}
```
أ) `100`  ب) `1`  ج) `0`  د) خطأ ترجمة
**الإجابة الصحيحة: ب**
**التعليل:** التابع `print` يستقبل `n` بالقيمة (`by value`)، أي نسخة منفصلة عن `c`؛ تعديل `n.x` داخل التابع لا يؤثر إطلاقاً على `c` الأصلي في `main`. الخيار (أ) يفترض خطأً أن التمرير كان بالمرجع.

### السؤال 5 (متوسط)
أي عامل من التالي يُستخدم للوصول لحقل سجل من خلال **مؤشر** عليه؟
أ) `.`  ب) `->`  ج) `::`  د) `&`
**الإجابة الصحيحة: ب**
**التعليل:** السهم `->` هو العامل المخصص للوصول إلى حقل عبر مؤشر لأنه يقوم أولاً بفك الإشارة (`dereference`) ثم يصل للحقل. النقطة `.` (أ) تُستخدم مع الكائن المباشر فقط وليس المؤشر.

### السؤال 6 (صعب)
ما الصيغة **الصحيحة نحوياً** للوصول لحقل `x` عبر المؤشر `ptr` باستخدام عامل فك الإشارة `*` بدلاً من السهم؟
أ) `*ptr.x`  ب) `(*ptr).x`  ج) `ptr.*x`  د) `*(ptr.x)`
**الإجابة الصحيحة: ب**
**التعليل:** الأقواس حول `*ptr` إلزامية لأن عامل النقطة `.` له أولوية تنفيذ أعلى من عامل فك الإشارة `*`؛ بدون الأقواس (كما في أ) سيحاول المترجم تنفيذ `ptr.x` أولاً وهذا خطأ لأن `ptr` مؤشر. (ج) و(د) صيغتان غير موجودتين في اللغة.

### السؤال 7 (متوسط)
لماذا يفشل المترجم عند كتابة `cout << c1;` حيث `c1` من نوع `Complex` مخصص، دون أي تحميل زائد سابق؟
أ) لأن `Complex` نوع غير قابل للطباعة إطلاقاً  ب) لأن المترجم لا يملك تعريفاً افتراضياً لعامل `<<` مع الأنواع المخصصة  ج) لأن `cout` لا يقبل السجلات  د) لأن `c1` غير مهيأ
**الإجابة الصحيحة: ب**

### السؤال 8 (صعب)
في الشكل التالي لتحميل عامل الجمع كتابع عضو، ما الذي يمثله `this`؟

أ) الطرف الثاني `n2`  ب) مؤشر ضمني يشير إلى الكائن الذي استُدعي التابع من خلاله (الطرف الأول)  ج) متحول عام في البرنامج  د) نسخة من `n2`
**الإجابة الصحيحة: ب**
**التعليل:** `this` هو مؤشر ضمني متوفر داخل كل تابع عضو يشير إلى الكائن الذي استُدعي التابع باسمه، أي الطرف الأيسر في `c1+c2` (أي `c1`)؛ التابع العضو يستقبل الطرف الثاني فقط كمعامل صريح.

**التعليل:** كما وضّحت المحاضرة، تُعالَج تهيئة السجل مثل تهيئة المصفوفة الجزئية — الحقول غير المذكورة تُهيَّأ تلقائياً بالقيمة `0`.

### السؤال 11 (صعب)
ماذا يطبع الكود التالي (بافتراض `operator+` مُعرَّف كتابع عضو يُعدّل `this` ويعيد `*this`)؟
```cpp

Complex c3 = c1 + c2;
```

أ) `1`  ب) `4`  ج) `0`  د) خطأ ترجمة

## الجزء الرابع: أسئلة تصحيح الكود

> غطِّ أنواع الأخطاء: ترجمة، منطقية، فحص إرجاع، dead code، سوء فهم.

### سؤال تصحيح 1 — misconception

**الكود (يحتوي خطأ):**
```cpp
struct Complex
{
    double x;
    double y;
}
```
**اكتشف الخطأ:** نسيان الفاصلة المنقوطة `;` بعد القوس المغلق `}` لتعريف السجل.

**التصحيح:**
```cpp
struct Complex
{
    double x;
    double y;
}; // added missing semicolon
```
**شرح الحل:**
1. تعريف `struct` هو جملة تعريف نوع، ويجب أن تُنهى بفاصلة منقوطة تماماً مثل أي جملة أخرى
2. بدون `;` يعتقد المترجم أن الجملة لم تنتهِ بعد، فيحاول دمج ما يليها ضمن نفس التعريف، مما يسبب سلسلة أخطاء ترجمة غامضة
3. القاعدة العملية: كل `struct {...}` ينتهي دائماً بـ `};`

### سؤال تصحيح 2 — logic

**الكود (يحتوي خطأ):**
```cpp
struct Point{ double x; double y; };
int main(){
    Point p;
    p->x = 5; // access field of a normal (non-pointer) variable
    p->y = 10;
    cout << p.x << " " << p.y;
}
```
**اكتشف الخطأ:** استخدام السهم `->` مع متحول عادي `p` (ليس مؤشراً)، بينما `p` سجل مباشر وليس مؤشراً على سجل.

**التصحيح:**
```cpp
struct Point{ double x; double y; };
int main(){
    Point p;
    p.x = 5; // use dot operator for a direct object
    p.y = 10;
    cout << p.x << " " << p.y;
}
```
**شرح الحل:**
1. عامل `->` مخصص فقط للوصول لحقول عبر **مؤشر** على سجل، لأنه يقوم أولاً بفك الإشارة ضمنياً
2. `p` هنا متحول عادي (`Point`) وليس مؤشراً (`Point*`)، لذا يجب استخدام النقطة `.` مباشرة
3. القاعدة: `.` مع الكائن المباشر، `->` مع المؤشر عليه

### سؤال تصحيح 3 — return_check

**الكود (يحتوي خطأ):**
```cpp
struct Complex{ double x; double y; };
Complex create(){
    Complex c={5,10};
    // missing return statement
}
int main(){
    Complex a = create();
    cout << a.x;
}
```
**اكتشف الخطأ:** التابع `create()` من نوع إرجاع `Complex` لكنه لا يحوي جملة `return` إطلاقاً — سلوك غير معرَّف/خطأ ترجمة في المترجمات الصارمة.

**التصحيح:**
```cpp
struct Complex{ double x; double y; };
Complex create(){
    Complex c={5,10};
    return c; // added the missing return statement
}
int main(){
    Complex a = create();
    cout << a.x;
}
```
**شرح الحل:**
1. أي تابع يُعلن عن نوع إرجاع غير `void` يجب أن يحوي `return` تعيد قيمة من ذلك النوع في كل مسار تنفيذ ممكن
2. غياب `return` هنا يعني أن قيمة `a` بعد الاستدعاء غير محددة (سلوك غير معرَّف)
3. الحل: إضافة `return c;` لإعادة السجل المحلي بالقيمة (تُنسخ قيمه إلى المستدعي)

### سؤال تصحيح 4 — off_by_one / misconception

**الكود (يحتوي خطأ):**
```cpp
struct Complex{ double x; double y; };
struct ComplexContainer
{
    double x;
    double y;
    Complex self; // trying to embed itself
};
```
**اكتشف الخطأ:** الحقل `self` من نوع `Complex` **وهذا التركيب سليم فعلياً** إذا كان `Complex` نوعاً مختلفاً عن `ComplexContainer` — **لكن** لو كان القصد أصلاً تضمين `ComplexContainer` نفسه، فهذا خطأ. لنفترض أن الخطأ المقصود تعليمياً هو الحالة العامة لتضمين نوع السجل داخل نفسه، أي:
```cpp
struct Complex
{
    double x;
    double y;
    Complex self; // embedding the SAME struct type inside itself — invalid
};
```

**التصحيح:**
```cpp
struct Complex
{
    double x;
    double y;
    Complex* self; // use a pointer to the same type instead
};
```
**شرح الحل:**
1. تضمين نوع السجل مباشرة داخل نفسه (`Complex self;` داخل `struct Complex`) يجعل حجم السجل يعتمد على حجم نفسه — تناقض منطقي، فيرفضه المترجم (`incomplete type`)
2. الحل الوحيد المتوافق مع قواعد اللغة هو استخدام **مؤشر** لنفس النوع (`Complex* self;`)، لأن حجم المؤشر ثابت ومعروف مسبقاً بغض النظر عن حجم ما يشير إليه
3. هذا هو بالضبط الأساس الذي تُبنى عليه لاحقاً القوائم المترابطة (`Linked List`)

### سؤال تصحيح 5 — logic (operator overloading)

**الكود (يحتوي خطأ):**
```cpp
struct Complex
{
    double x, y;
    Complex& operator+(Complex n2){
        x + n2.x; // computed but not assigned/used, and result discarded
        y + n2.y;
        return *this;
    }
};
```
**اكتشف الخطأ:** العبارتان `x + n2.x;` و `y + n2.y;` تحسبان قيمة الجمع لكن **لا تُسندانها** إلى أي مكان (لا `x = ...` ولا `x += ...`)؛ النتيجة تُحسب ثم تُرمى فوراً (`dead code` من ناحية الأثر الفعلي)، فيبقى `this` دون تغيير حقيقي.

**التصحيح:**
```cpp
struct Complex
{
    double x, y;
    Complex& operator+(Complex n2){
        x += n2.x; // properly accumulate into the field
        y += n2.y;
        return *this;
    }
};
```
**شرح الحل:**
1. عملية حسابية بلا إسناد (`x + n2.x;` بمفردها) لا تُغيّر أي متحول — قيمتها تُحسب ثم تُهمَل
2. لتحديث الحقل فعلياً يجب استخدام إسناد صريح مثل `x = x + n2.x;` أو اختصاره `x += n2.x;`
3. هذا النوع من الأخطاء صامت تماماً (لا يعطي خطأ ترجمة)، ويُكتشف فقط بتتبع الناتج الفعلي — لذلك من الأخطاء الخطيرة الشائعة

## الجزء الرابع: بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ما تعريف السجل (`struct`) بجملة واحدة؟
A: نمط بيانات مخصص يجمّع مجموعة من الحقول المرتبطة تحت اسم واحد.

**Q2:** ما الرمز الإلزامي بعد `}` عند تعريف `struct`؟
A: الفاصلة المنقوطة `;`.

**Q3:** كيف نصل لحقل من سجل عادي (غير مؤشر)؟
A: باستخدام عامل النقطة `.`، مثل `number.x`.

**Q4:** كيف نصل لحقل سجل من خلال مؤشر عليه؟
A: باستخدام السهم `->`، أو `(*ptr).x` مع الأقواس الإلزامية.

**Q5:** ماذا يحدث للحقول غير المذكورة في تهيئة سجل بـ `{}`؟
A: تأخذ القيمة الافتراضية `0` تلقائياً.

**Q6:** هل يمكن تضمين نوع السجل مباشرة كحقل داخل نفسه؟
A: لا؛ يُسمح فقط بتضمين مؤشر على نفس النوع (`Type* field;`).

**Q7:** لماذا تفشل عملية `c1+c2` على نوع `struct` مخصص دون تعريف مسبق؟
A: لأن المترجم لا يملك تعريفاً افتراضياً لعامل `+` للأنواع المخصصة؛ يجب تحميله زائداً (`operator overloading`).

**Q8:** ما الفرق بين تمرير سجل بالقيمة وبالمرجع لتابع؟
A: بالقيمة يُنسخ السجل بالكامل ولا يتأثر الأصل بأي تعديل؛ بالمرجع لا نسخ ويتأثر الأصل مباشرة بأي تعديل.

**Q9:** ما دور الكلمة المفتاحية `this` داخل تابع عضو؟
A: مؤشر ضمني يشير إلى الكائن الذي استُدعي التابع من خلاله (الطرف الأيسر ضمنياً).

**Q10:** لماذا يُكتب `operator<<` غالباً كتابع صديق (`friend`) وليس عضواً؟
A: لأن الطرف الأيسر لعامل `<<` هو `ostream` وليس نوع السجل نفسه، فلا يمكن جعله تابعاً عضواً في السجل.

**Q11:** ما الفرق بين `Complex* ptrN = new Complex;` و `Complex a;`؟
A: الأول يخصص السجل ديناميكياً على `heap` ويحتاج `delete` لاحقاً، بينما الثاني متحول عادي على `stack` يُحرَّر تلقائياً.

**Q12:** لماذا يجب وضع الأقواس في `(*ptr).x` ولا يمكن كتابة `*ptr.x`؟
A: لأن عامل النقطة `.` له أولوية تنفيذ أعلى من عامل فك الإشارة `*`؛ بدون الأقواس سيُنفَّذ `ptr.x` أولاً وهو تعبير خاطئ لأن `ptr` مؤشر.

**Q13:** ما الفرق بين `Complex& ref = a;` و `Complex* ptr = &a;`؟
A: `ref` مرجع، أي اسم بديل مباشر لنفس المتحول `a` (لا يحتاج فك إشارة)؛ `ptr` مؤشر يخزّن عنوان `a` ويحتاج فك إشارة أو سهم للوصول لحقولها.

**Q14:** بأي ترتيب تُسند القيم في تهيئة سجل بقائمة `{v1, v2, ...}`؟
A: بنفس ترتيب تعريف الحقول داخل `struct`.

## الجزء الخامس: كتابة الكود الكامل (مرجع شامل)

```cpp
#include <iostream>
using namespace std;

// ===== Struct definition =====
struct Complex
{
    double x; // real part
    double y; // imaginary part

    // Member operator for addition (modifies this, mirrors slide's design)
    Complex& operator+(Complex n2){
        this->x += n2.x;
        this->y += n2.y;
        return *this;
    }

    // Member operator for subtraction
    Complex& operator-(Complex n2){
        this->x -= n2.x;
        this->y -= n2.y;
        return *this;
    }

    // Member operator for multiplication: (a+bi)(c+di) = (ac-bd)+(ad+bc)i
    Complex operator*(Complex n2){
        Complex result;
        result.x = this->x * n2.x - this->y * n2.y;
        result.y = this->x * n2.y + this->y * n2.x;
        return result;
    }

    // Member operator for division: (a+bi)/(c+di) = [(ac+bd)+(bc-ad)i]/(c^2+d^2)
    Complex operator/(Complex n2){
        Complex result;
        double denom = n2.x * n2.x + n2.y * n2.y;
        result.x = (this->x * n2.x + this->y * n2.y) / denom;
        result.y = (this->y * n2.x - this->x * n2.y) / denom;
        return result;
    }

    // Friend function for printing via cout
    friend ostream& operator<<(ostream& cout, Complex& n){
        cout << n.x << ((n.y >= 0) ? " + " : " - ") << (n.y >= 0 ? n.y : -n.y) << "i";
        return cout;
    }
};

// Free function to read a Complex by reference
void enter(Complex& number){
    cin >> number.x >> number.y;
}

// Free function to read a Complex by pointer
void enter(Complex* number){
    cin >> number->x >> number->y;
}

// Free function to print a Complex by value
void print(Complex number){
    cout << number.x <<
    ((number.y >= 0) ? " + " : " - ")
    << (number.y >= 0 ? number.y : -number.y) << "i" << endl;
}

// Free function returning a Complex by value
Complex create(){
    Complex c = {0};
    return c;
}

int main(){
    // Basic usage: define, read, print
    Complex number;
    cin >> number.x >> number.y;
    print(number);

    // Initialization with initializer lists
    Complex number1 = {1, 2};
    Complex number2 = {1}; // y defaults to 0
    print(number1);
    print(number2);

    // Pointer access to fields
    Complex a = {5, -3};
    Complex* ptr = &a;
    cout << ptr->x << " " << (*ptr).y << endl;

    // Operator overloading usage
    Complex c1 = {1, 2};
    Complex c2 = {3, 4};
    cout << (c1 + c2) << endl;
    cout << (c1 - c2) << endl;
    cout << (c1 * c2) << endl;
    cout << (c1 / c2) << endl;

    return 0;
}
```

## الجزء الخامس: أسئلة نظرية متوقعة بالامتحان

### سؤال 1: عرّف السجل (`struct`) واشرح الغرض منه
**نموذج الإجابة:**
1. التعريف: نمط بيانات مركّب يجمع عدة حقول (`fields`) من أنواع مختلفة أو متماثلة تحت اسم واحد
2. المكونات/الشروط: كلمة `struct` + اسم النمط + جسم الحقول بين `{}` + فاصلة منقوطة إلزامية بعد `}`
3. مثال: `struct Complex{ double x; double y; };`
4. متى نستخدم: عند الحاجة لتمثيل بيانات مركّبة مرتبطة منطقياً كوحدة واحدة (نقطة، طالب، عدد عقدي)

### سؤال 2: ما الفرق بين الوصول لحقل سجل بالنقطة والسهم؟
**نموذج الإجابة:**
1. التعريف: كلا العاملين يُستخدمان للوصول لحقل داخل سجل، لكن يختلفان حسب نوع المتحول المستخدَم
2. المكونات/الشروط: النقطة `.` تُستخدم مع الكائن المباشر؛ السهم `->` يُستخدم مع مؤشر على الكائن (يتضمن فك إشارة ضمني)
3. مثال: `number.x` مقابل `ptr->x` (أو `(*ptr).x`)
4. متى نستخدم: النقطة عندما يكون لدينا السجل نفسه؛ السهم عندما يكون لدينا مؤشر إليه

### سؤال 3: اشرح لماذا لا يمكن تضمين نوع السجل داخل نفسه مباشرة
**نموذج الإجابة:**
1. التعريف: تضمين `Complex xc;` داخل `struct Complex` يعني أن حجم السجل يعتمد على حجم نفسه — تناقض دائري
2. المكونات/الشروط: المترجم يحتاج معرفة الحجم الكامل للنوع أثناء تعريفه (`incomplete type error` إن لم يعرفه)
3. مثال: الحل الصحيح `Complex* xc;` لأن حجم المؤشر ثابت بغض النظر عن حجم ما يشير إليه
4. متى نستخدم: هذه الفكرة أساس لاحقاً في بناء بنى مثل `Linked List` حيث كل عقدة تحوي مؤشراً على العقدة التالية من نفس النوع

### سؤال 4: ما الفرق بين تمرير السجل للتابع بالقيمة وبالمرجع وبالمؤشر؟
**نموذج الإجابة:**
1. التعريف: ثلاث طرق لتمرير متحول سجل كمعامل لتابع
2. المكونات/الشروط: بالقيمة (`Type param`) ينسخ كامل السجل؛ بالمرجع (`Type& param`) يعمل مباشرة على الأصل بدون نسخ؛ بالمؤشر (`Type* param`) يستقبل عنوان الأصل ويحتاج السهم `->` للوصول للحقول
3. مثال: `void print(Complex n)` مقابل `void enter(Complex& n)` مقابل `void enter(Complex* n)`
4. متى نستخدم: بالقيمة للقراءة فقط دون تعديل؛ بالمرجع/المؤشر عند الحاجة لتعديل الأصل أو تفادي تكلفة النسخ

### سؤال 5: ما المقصود بـ `operator overloading` ولماذا نحتاجه؟
**نموذج الإجابة:**
1. التعريف: آلية في C++ تسمح للمبرمج بتعريف سلوك مخصص لعامل (`+`, `<<`, ...) عند تطبيقه على نوع بيانات من صنعه
2. المكونات/الشروط: يُكتب بصيغة `operator{symbol}` كتابع حر، عضو، أو صديق حسب طبيعة العامل والأطراف
3. مثال: `Complex& operator+(Complex n2){ ... }`
4. متى نستخدم: عندما نريد أن تتصرف الأنواع المخصصة مثل الأنواع الأساسية عند استخدام عوامل اللغة القياسية

### سؤال 6: متى نستخدم `friend` عند تحميل عامل زائد؟
**نموذج الإجابة:**
1. التعريف: `friend` تمنح تابعاً حراً (ليس عضواً حقيقياً) صلاحية الوصول لحقول السجل الخاصة، رغم كتابته داخل تعريف السجل
2. المكونات/الشروط: يُستخدم غالباً عندما يكون الطرف الأيسر للعامل من نوع خارجي (مثل `ostream` في `<<`) فلا يمكن جعل العامل تابعاً عضواً في نوعنا
3. مثال: `friend ostream& operator<<(ostream& cout, Complex& n){ ... }`
4. متى نستخدم: عند الحاجة للوصول للحقول الخاصة من خارج السجل، خصوصاً عند تحميل عوامل الإدخال/الإخراج

### سؤال 7: اشرح دور `this` في تابع عضو يحمّل عامل جمع
**نموذج الإجابة:**
1. التعريف: `this` مؤشر ضمني متاح داخل أي تابع عضو، يشير إلى الكائن الذي استُدعي التابع باسمه
2. المكونات/الشروط: يُستخدم بالسهم `->` للوصول لحقول الكائن الحالي؛ `*this` يعيد قيمة الكائن نفسه (فك إشارة `this`)
3. مثال: `this->x += n2.x; return *this;`
4. متى نستخدم: عند كتابة تابع عضو يحتاج الإشارة صراحة إلى الكائن الحالي، خصوصاً عند إعادة الكائن نفسه من التابع

### سؤال 8: كيف تُعالَج تهيئة سجل بقيم أقل من عدد حقوله؟
**نموذج الإجابة:**
1. التعريف: تهيئة جزئية للسجل باستخدام `{}` بعدد قيم أقل من عدد الحقول المُعرَّفة
2. المكونات/الشروط: تُسند القيم المذكورة بالترتيب للحقول الأولى؛ أي حقل متبقٍّ يُهيَّأ تلقائياً بالقيمة الافتراضية `0`
3. مثال: `Complex number2={1};` تعطي `x=1, y=0`
4. متى نستخدم: عندما نعرف قيمة بعض الحقول فقط عند التعريف ونريد ترك الباقي بقيمة افتراضية آمنة

## الجزء السادس: قائمة فحص ذاتي قبل الامتحان ✅

- [ ] أستطيع تعريف `struct` جديد بحقول متعددة دون نسيان `;` في النهاية
- [ ] أفهم الفرق بين تهيئة سجل كامل التهيئة وسجل بتهيئة جزئية (والقيمة الافتراضية `0`)
- [ ] أعرف لماذا لا يمكن تضمين نوع السجل داخل نفسه مباشرة، وكيف نستخدم مؤشراً بدلاً منه
- [ ] أميّز بين الوصول للحقول بالنقطة `.` (كائن مباشر) والسهم `->` (مؤشر)
- [ ] أفهم لماذا `(*ptr).x` يحتاج أقواساً إلزامية
- [ ] أستطيع تمرير سجل لتابع بالقيمة، بالمرجع، وبالمؤشر، وأعرف الفرق العملي بينها
- [ ] أفهم لماذا يفشل المترجم مع `c1+c2` دون `operator overloading`
- [ ] أستطيع كتابة `operator overloading` كتابع عضو باستخدام `this`
- [ ] أفهم متى نحتاج `friend` عند تحميل عامل مثل `<<`
- [ ] أستطيع تطبيق نفس مبدأ `operator overloading` على `-`, `*`, `/` باتباع القواعد الرياضية الصحيحة لكل عملية
