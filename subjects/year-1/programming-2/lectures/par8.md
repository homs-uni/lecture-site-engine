## الجزء الأول: الشرح التفصيلي (سطر بسطر / فقرة بفقرة)

> هذه المحاضرة امتداد مباشر لمحاضرة الصفوف الأولى (`class`, `private`, `public`, الباني `constructor`, الهادم `destructor`, توابع `getter`/`setter`). قبل المتابعة يجب أن تكون هذه المفاهيم واضحة.

### 1. أثر الكلمة `const` على الصفوف والأغراض

#### النص الأصلي يقول:
> «عندما نعمل على تعريف صف class جديد نحن نعمل على تعريف نمط بيانات جديد Data Type. يمكن أن يستخدم هذا النمط مع الكلمة const... قمنا بتعريف غرض من الصف class1 وحددنا أنه ثابت. في C++ إذا كان الغرض ثابت يجب ألا يكون قابلًا للتعديل لا هو ولا محتوياته. بمعنى آخر يجب أن نحدد بعض التوابع على أنها const والبعض الآخر عادي.»

#### الشرح المبسّط:
عندما تكتب `class Time{...};` أنت لا تُعرّف غرضًا واحدًا، بل تُعرّف **نمط بيانات جديد** (Data Type) تمامًا كما يُعتبر `int` أو `double` نمط بيانات. وبما أن `int` يمكن أن يُسبَق بالكلمة `const` (مثل `const int x = 5;`)، فإن أي `class` معرَّف من قِبلك يمكن أيضًا أن يُسبَق بـ `const`:

```cpp
const class1 c;
```

**لماذا؟** لأن الهدف من `const` واحد في كل الحالات: منع أي تعديل على القيمة بعد إنشائها. الفرق أن `int` قيمة بسيطة، بينما الغرض (object) قد يحتوي عدة حقول (fields)، وبالتالي "ثبات الغرض" يعني ثبات **كل** الحقول الداخلية له معًا، لا حقلًا واحدًا فقط.

هذا يطرح مشكلة: إذا كان الغرض بأكمله `const`، فكيف يمكن استدعاء توابع مثل `getHour()` التي لا تُعدّل شيئًا؟ ألن يرفضها المترجم لمجرد أنها تنتمي لغرض ثابت؟ من هنا تأتي الحاجة لتمييز التوابع التي **لا تُعدّل** حالة الغرض (توابع القراءة فقط، مثل `getters`) عن التي **تُعدّل** حالته (مثل `setters`)، ووسم النوع الأول بالكلمة `const`.

#### 💡 التشبيه:
> تخيّل عقدًا قانونيًا موقّعًا (الغرض الثابت). يمكنك **قراءة** بنوده أي عدد من المرات، لكن لا يحق لأي طرف **تعديل** أي بند فيه بعد التوقيع.
> **وجه الشبه:** قراءة العقد = استدعاء تابع `const` (مسموح) | تعديل العقد = استدعاء تابع غير `const` (ممنوع).

#### مهم للامتحان ⚠️:
> `const` على غرض تعني أن جميع أعضاء البيانات (data members) فيه تصبح ثابتة، وليس عضوًا واحدًا فقط.

### 2. بنية الصف قبل تطبيق `const` على التوابع

#### النص الأصلي يقول:
> «بفرض لدينا الصف التالي» (صف `Time` الذي يحتوي `hour, minute, second` كحقول خاصة، وبانيًا، وتوابع `set` و`get` و`print`).

#### الشرح المبسّط:
```cpp
// Time class before marking functions as const
class Time{
private:
    int hour, minute, second;          // (1) private data members
public:
    Time(int=0, int=0, int=0);         // (2) constructor with default args
    void setHour(int);                 // (3) modifies state -> NOT const
    void setMinute(int);               // (4) modifies state -> NOT const
    void setSecond(int);               // (5) modifies state -> NOT const
    void setTime(int, int, int);       // (6) modifies state -> NOT const
    int getHour();                     // (7) only reads -> candidate for const
    int getMinute();                   // (8) only reads -> candidate for const
    int getSecond();                   // (9) only reads -> candidate for const
    void printMilitary();              // (10) only reads -> candidate for const
    void printStandard();              // (11) only reads -> candidate for const
};
```

#### شرح كل سطر:
1. `int hour, minute, second;` → حقول خاصة تخزّن حالة الغرض — لا يمكن الوصول لها من خارج الصف.
2. `Time(int=0,int=0,int=0);` → باني بقيم افتراضية، يستدعي `setTime` داخليًا لضبط القيم مع التحقق من صحتها.
3–6. توابع `set*` → **تُعدّل** الحقول، لذلك لا يصح وسمها `const`.
7–11. توابع `get*` و`print*` → **تقرأ فقط** ولا تُعدّل أي حقل، لذلك هي **مرشّحة** لتصبح `const`.

> **الناتج المتوقع:** لا ناتج تنفيذي بعد — هذا مجرد تعريف الصف (الإعلان `declaration`) دون تنفيذ التوابع بعد.

### 3. تنفيذ توابع الصف (Implementation)

#### النص الأصلي يقول:
> كود تنفيذ الباني والتوابع الأعضاء لصف `Time` (`setHour`, `setMinute`, `setSecond`, `setTime`, `getHour`, `getMinute`, `getSecond`, `printMilitary`, `printStandard`).

#### الشرح المبسّط:
```cpp
Time::Time(int h=0,int m=0,int s=0){ setTime(h,m,s); }        // (1)
void Time::setHour(int h){ hour=(h>=0 && h<24)? h:0; }         // (2)
void Time::setMinute(int m){ minute=(m>=0 && m<60)? m:0; }     // (3)
void Time::setSecond(int s){ second=(s>=0 && s<60)? s:0; }     // (4)
void Time::setTime(int h,int m,int s){                         // (5)
    setHour(h); setMinute(m); setSecond(s);
}
int Time::getHour(){ return hour; }                             // (6)
int Time::getMinute(){ return minute; }                         // (7)
int Time::getSecond(){ return second; }                         // (8)
void Time::printMilitary(){ cout<<hour<<minute<<second; }       // (9)
void Time::printStandard(){                                     // (10)
    cout<<((hour>12)?hour-12:hour)<<":"<<minute<<":"<<second
        <<" "<<((hour>=12)?"PM":"AM");
}
```

#### شرح كل سطر:
1. الباني يفوّض العمل إلى `setTime` بدلًا من تكرار كود التحقق من الصحة (Delegation).
2–4. كل تابع `set*` يستخدم عاملًا شرطيًا `?:` للتحقق من أن القيمة ضمن مجال صحيح، وإلا يضع `0`.
5. `setTime` يجمع الاستدعاءات الثلاثة السابقة في مكان واحد لتفادي التكرار.
6–8. توابع القراءة تُرجع القيمة كما هي بدون أي تعديل — هذا ما يجعلها آمنة لتصبح `const` لاحقًا.
9–10. توابع الطباعة تعرض القيم بصيغة عسكرية (24 ساعة) أو قياسية (12 ساعة + AM/PM) دون تعديل أي حقل.

#### الفهم الخاطئ ❌: أي تابع لا يأخذ وسيطًا يعتبر تلقائيًا آمنًا للاستخدام مع غرض ثابت.
#### الفهم الصحيح ✅: المعيار ليس وجود وسائط، بل هل يُعدّل التابع أي حقل من حقول الغرض أم لا — وهذا ما تحدده الكلمة `const` صراحة في التوقيع.

### 4. وسم التوابع بالكلمة `const`

#### النص الأصلي يقول:
> «في حال أردنا أن نعرف الغرض على أنه const يجب أن يسبق تعريف الغرض الكلمة المفتاحية const... نلاحظ وجود كلمة const في نهاية تعريف التابع، مما يسمح لنا باستدعاء هذا التابع في حال كان الغرض const.»

#### الشرح المبسّط:
لتعريف غرض ثابت:
```cpp
const bdate(1,1,1991);
```
الكلمة `const` **تسبق** تعريف الغرض.

أما لتحديد أن تابعًا عضوًا لا يُعدّل حالة الغرض، تُوضع الكلمة `const` **بعد** قوسي المعاملات مباشرة، في نهاية توقيع التابع:

```cpp
class Time{
private:
    int hour, minute, second;
public:
    Time(int=0, int=0, int=0);
    void setHour(int);
    void setMinute(int);
    void setSecond(int);
    void setTime(int, int, int);
    int getHour() const;          // (1) const function
    int getMinute() const;        // (2) const function
    int getSecond() const;        // (3) const function
    void printMilitary() const;   // (4) const function
    void printStandard();         // (5) NOT const (لاحظ الفرق!)
};
```

#### شرح كل سطر:
1–4. وُسمت هذه التوابع بـ `const` لأنها لا تُعدّل أي حقل — فقط تقرأ وتعرض.
5. لاحظ أن `printStandard` **لم تُوسم** بـ `const` في هذه النسخة من الشرح — وهذا مقصود في المحاضرة لتوضيح ما سيحدث لاحقًا عند استدعائها على غرض ثابت (راجع القسم 6).

#### مهم للامتحان ⚠️:
> موقع كلمة `const` يحدد معناها: **قبل الغرض** = الغرض نفسه ثابت. **بعد توقيع التابع** = التابع لا يُعدّل حالة الغرض.

### 5. لا تأثير لـ `const` على جسم الكود (body) — فقط على التوقيع

#### النص الأصلي يقول:
> «لا يؤثر إضافة كلمة const في تعريف التابع على كود التابع.»

#### الشرح المبسّط:
إضافة `const` لا تُغيّر شيئًا في تنفيذ التابع نفسه — فقط تُضاف الكلمة بعد الأقواس في تعريف التابع أينما ظهر (في الصف وفي التنفيذ):

```cpp
int Time::getHour() const{ return hour; }               // (1)
int Time::getMinute() const{ return minute; }           // (2)
int Time::getSecond() const{ return second; }           // (3)
void Time::printMilitary() const{ cout<<hour<<minute<<second; } // (4)
```

#### شرح كل سطر:
1–4. الكود داخل الأقواس `{}` مطابق تمامًا لما كان قبل إضافة `const`. الإضافة الوحيدة هي كلمة `const` بعد `()` في كل من الإعلان (داخل الصف) والتعريف (خارج الصف باستخدام `::`). إن نسيت إضافتها في أحد الموضعين فقط، سيرفض المترجم الكود لعدم تطابق التوقيع.

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** لو أضفت `const` في تعريف التابع خارج الصف فقط، ونسيتها داخل الصف — ماذا سيحدث؟
> **لماذا هذا مهم؟** لأن المترجم يعتبر التوقيعين مختلفين، فسيحدث خطأ ترجمة (لا يوجد تابع مطابق يمكن تعريفه).

### 6. التأثير في الاستدعاء (Calling Rules)

#### النص الأصلي يقول:
> «التابع الموسوم بكلمة const يمكن استدعاؤه في حال كان الغرض ثابت أو متحول. في حال كان الغرض ثابت const عندها لا يمكن سوى استدعاء التوابع الموسومة بـ const.»

#### الشرح المبسّط:
القاعدة الذهبية للاستدعاء:

| نوع الغرض | تابع `const` | تابع غير `const` |
| --- | --- | --- |
| غرض عادي (`variable`) | ✅ يمكن استدعاؤه | ✅ يمكن استدعاؤه |
| غرض ثابت (`constant`) | ✅ يمكن استدعاؤه | ❌ خطأ ترجمة |

**لماذا؟** لأن استدعاء تابع غير `const` على غرض `const` يعني أن المترجم **لا يستطيع ضمان** عدم تعديل الغرض — والتابع أصلًا قد يُعدّل حالته لأنه غير موسوم بـ `const`. فالمترجم يرفض الاحتمال بالكامل، حتى لو كان التابع في الواقع لا يُعدّل شيئًا.

```cpp
Time t(10,12,13);              // (1) غرض عادي variable
const Time c(10,12,13);        // (2) غرض ثابت constant

t.setHour(15);                 // (3) OK: غرض عادي + تابع غير const
cout<<t.getMinute()<<endl;     // (4) OK: غرض عادي + تابع const
t.printMilitary();             // (5) OK: غرض عادي + تابع const
t.printStandard();             // (6) OK: غرض عادي + تابع غير const

cout<<c.getMinute()<<endl;     // (7) OK: غرض ثابت + تابع const
c.printMilitary();             // (8) OK: غرض ثابت + تابع const
c.printStandard();             // (9) ❌ خطأ! غرض ثابت + تابع غير const
```

#### شرح كل سطر:
1. `t` غرض عادي (variable) — يمكن استدعاء أي تابع عليه.
2. `c` غرض ثابت (constant) — لا يمكن تعديله بعد الإنشاء.
3. `setHour` تابع غير `const`، لكن `t` عادي → مسموح.
4. `getMinute` تابع `const` على غرض عادي → مسموح دائمًا.
5. `printMilitary` تابع `const` على غرض عادي → مسموح.
6. `printStandard` تابع غير `const` على غرض عادي → مسموح لأن `t` ليس ثابتًا.
7. `getMinute` تابع `const` على غرض ثابت → مسموح لأن التابع يضمن عدم التعديل.
8. `printMilitary` تابع `const` على غرض ثابت → مسموح لنفس السبب.
9. `printStandard` **غير موسومة بـ `const`**، والغرض `c` ثابت → **خطأ ترجمة**، لأن المترجم لا يستطيع ضمان أن `printStandard` لن تُعدّل الغرض `c`.

#### 🛠️ استكشاف الأخطاء:
| الخطأ | السبب | الحل |
| --- | --- | --- |
| `passing 'const Time' as 'this' argument discards qualifiers` | استدعاء تابع غير `const` على غرض `const` | وسم التابع بـ `const` إذا كان لا يُعدّل أي حقل فعليًا |
| نسيان `const` في أحد الموضعين (الإعلان أو التعريف) | عدم تطابق التوقيع بين الصف والتنفيذ | إضافة `const` في كلا الموضعين معًا |

#### الدرس المستفاد:
> يجب وسم كل تابع لا يُعدّل حالة الغرض بـ `const` منذ البداية، حتى لو لم تكن تخطط لاستخدام أغراض ثابتة حاليًا — لأن ذلك يوسّع إمكانية استخدام الصف لاحقًا دون كسر أي كود.

### 7. حقل ثابت ضمن الغرض (`const` Data Member)

#### النص الأصلي يقول:
> «ماذا يمكن أن يحدث في حال كان حقل ضمن الغرض من نوع const؟ نحن نعرف أن أي قيمة تسبقها كلمة const تعني أنها قيمة ثابتة. في هذه الحالة يجب إعطاء هذا الثابت قيمة قبل استخدام الغرض. يتم تفعيل ذلك في الباني.»

#### الشرح المبسّط:
حتى الآن تحدثنا عن **الغرض بأكمله** كـ `const`. لكن ماذا لو أردنا أن يكون **حقل واحد فقط** داخل الصف ثابتًا (بينما بقية الحقول قابلة للتعديل)؟

```cpp
class Increment{
public:
    Increment(int c=0, int i=1);
    void addIncrement() { count += increment; }   // (1)
    void print() const {
        cout << "count = " << count
             << ", increment = " << increment << endl;
    }
private:
    int count;             // (2) حقل عادي قابل للتعديل
    const int increment;   // (3) حقل ثابت — لا يتغير بعد الإنشاء
};
```

#### شرح كل سطر:
1. `addIncrement` يُعدّل `count` فقط (باستخدام قيمة `increment` الثابتة) — لذلك لم توسم بـ `const` لأنها تُعدّل حالة الغرض.
2. `count` حقل عادي، يتغير مع كل استدعاء لـ `addIncrement`.
3. `increment` حقل `const` — بمجرد تحديد قيمته عند إنشاء الغرض، لا يمكن تغييره طوال عمر الغرض.

#### 💡 التشبيه:
> فكّر في `increment` كرقم تسلسلي محفور على بطاقة هوية عند إصدارها — لا يمكن تعديله لاحقًا مهما تغيّرت بيانات أخرى على البطاقة (مثل العنوان).
> **وجه الشبه:** الرقم التسلسلي المحفور = `const int increment` | العنوان القابل للتحديث = `int count`.

### 8. تهيئة الحقل الثابت عبر قائمة التهيئة (Initialization List)

#### النص الأصلي يقول:
> «المميز هنا ما يوجد بعد النقطتين في الباني وقبل أقواس الكود `increment(i)`... بناءً على ذلك يتم تهيئة قيمة الثابت increment عند بناء الغرض... هذه الطريقة تجبر المترجم على إسناد القيم قبل تنفيذ كود الباني... هذه الطريقة في إسناد القيم قابلة للاستخدام مع أي نوع من الحقول وليس فقط الثوابت، لكنها الطريقة الوحيدة للثوابت.»

#### الشرح المبسّط:
لأن الحقل `increment` ثابت (`const`)، **لا يمكن** إسناد قيمة له داخل جسم الباني بالطريقة العادية (`increment = i;`) — لأن ذلك يُعتبر "تعديلًا"، والحقل الثابت يجب أن يُهيَّأ (initialize) لا أن يُعدَّل (assign). لذلك نستخدم **قائمة التهيئة**:

```cpp
Increment::Increment(int c=0, int i=1) : increment(i){   // (1)
    count = c;                                             // (2)
}
```

#### شرح كل سطر:
1. `: increment(i)` هي قائمة التهيئة — تُنفَّذ **قبل** جسم الباني، وتُسند القيمة `i` إلى `increment` مباشرة عند إنشاء الحقل، لا بعده.
2. `count = c;` إسناد عادي داخل جسم الباني — مسموح لأن `count` ليس `const`.

#### لتهيئة عدة حقول معًا بنفس الطريقة:
```cpp
Increment::Increment(int c=0, int i=1) : increment(i), count(c) {}   // (1)
```

#### شرح كل سطر:
1. يمكن تهيئة عدة حقول في قائمة واحدة مفصولة بفواصل، وحينها يصبح جسم الباني فارغًا `{}` لأن كل الإسناد تم في قائمة التهيئة.

#### مهم للامتحان ⚠️:
> قائمة التهيئة (Initialization List) هي **الطريقة الوحيدة** لإسناد قيمة لحقل `const` (أو مرجع `reference`) داخل صف — لا يمكن استخدام `=` داخل جسم الباني لهذا النوع من الحقول.

#### ⚙️ الخطوات / الخوارزمية: تهيئة غرض يحتوي حقلًا ثابتًا

> ما هدف هذه العملية؟ ضمان أن كل حقل `const` يحصل على قيمة صحيحة قبل أن يصبح الغرض جاهزًا للاستخدام، ومنع أي محاولة لاحقة لتعديله.

```algorithm
1 | استقبال الوسائط | باني الصف (constructor) | يستقبل c و i من المستدعي
2 | تنفيذ قائمة التهيئة | Initialization List | يُسند i إلى increment مباشرة قبل بدء الجسم
3 | تنفيذ جسم الباني | Constructor Body | يُسند c إلى count بإسناد عادي
4 | الغرض جاهز | Compiler | لا يمكن تعديل increment بعد هذه اللحظة
```

#### نقاط التنفيذ:
- الترتيب الذي تُكتب فيه الحقول في قائمة التهيئة **لا** يحدد ترتيب التنفيذ الفعلي؛ الترتيب الفعلي هو ترتيب إعلان الحقول داخل الصف.
- إغفال تهيئة حقل `const` في قائمة التهيئة يُسبب خطأ ترجمة، لأنه سيبقى بلا قيمة إلى الأبد.

### 9. التوابع والصفوف الصديقة (`friend`)

#### النص الأصلي يقول:
> «التابع الصديق هو تابع لا ينتمي للصف ولكن يحق له الوصول للحقول من نوع private (خاص) و protected (محمي). الصف الصديق هو صف خارجي كليًا عن الصف الحالي (لا يرتبط بأي علاقة من نوع تراكب composition أي صف داخل صف أو علاقة وراثة).»

#### الشرح المبسّط:
عادة، الحقول `private` لا يمكن الوصول إليها إلا من داخل توابع الصف نفسه. لكن أحيانًا نحتاج **تابعًا خارجيًا** (لا ينتمي للصف) أو **صفًا خارجيًا بالكامل** أن يصل إلى تلك الحقول الخاصة — دون كسر التغليف (encapsulation) تمامًا، بل بمنح **إذن استثنائي صريح**. هذا ما توفره الكلمة `friend`.

**نقطة جوهرية:** الصف أو التابع الصديق **ليس** جزءًا من الصف نفسه، ولا يرث منه، ولا يُركَّب (composition) بداخله — هو كيان مستقل تمامًا، مُنح فقط إذن الوصول.

#### 💡 التشبيه:
> تخيل منزلك (الصف) بأبواب مقفلة (`private`). عادة لا يدخل أحد إلا أهل المنزل (توابع الصف). لكنك تعطي **مفتاحًا احتياطيًا** لصديق مقرّب (تابع/صف `friend`) يستطيع الدخول رغم أنه لا يسكن في المنزل ولا ينتمي للعائلة.
> **وجه الشبه:** المفتاح الاحتياطي = إعلان `friend` | الصديق الذي يدخل = التابع أو الصف الصديق.

### 10. ميزات علاقة الصداقة

#### النص الأصلي يقول:
> «العلاقة من طرف واحد، A صديق B لا يعني بالضرورة أن يكون B صديق A. العلاقة غير متعدية، A صديق B وB صديق C لا يعني بالضرورة أن يكون A صديق C. الصداقة تمنح ولا تؤخذ، أي A صديق B يعني أن B يمنح الصداقة لـ A ولا يمكن لـ A أن يكون صديق B إذا لم تحدد العلاقة ضمن B أولًا.»

#### الشرح المبسّط:
| الخاصية | الشرح | مثال |
| --- | --- | --- |
| علاقة من طرف واحد (One-directional) | كون A صديقًا لـ B لا يعني العكس | إذا أعلن `A` أن `B` صديق له، هذا لا يمنح `A` تلقائيًا حق الوصول لحقول `B` الخاصة |
| غير متعدية (Non-transitive) | صداقة A مع B وB مع C لا تعني صداقة A مع C | يجب أن يصرّح كل صف بأصدقائه بنفسه |
| تُمنح لا تُؤخذ (Granted, not taken) | فقط الصف الذي يريد منح إذن الوصول هو من يكتب إعلان `friend` داخل تعريفه | لا يمكن لصف خارجي أن "يفرض" نفسه صديقًا لصف آخر |

#### الفهم الخاطئ ❌: إذا كان الصف A صديقًا للصف B، فإن B تلقائيًا يعتبر صديقًا لـ A أيضًا.
#### الفهم الصحيح ✅: يجب أن يُعلن كل صف صداقته بشكل منفصل وصريح؛ الصداقة ليست تبادلية إلا إذا أُعلنت من الطرفين.

### 11. تعريف وتنفيذ التابع الصديق

#### النص الأصلي يقول:
> «يتم تعريف التابع الصديق كقالب ضمن الصف قبل أي كلمة public أو private (لأن التابع الصديق ليس جزءًا من التابع). ينبغي أن يتقبل هذا التابع غرضًا من الصف. كود التابع الصديق يكون كتلة منفصلة عن الصف.»

#### الشرح المبسّط:
```cpp
class Count{
    friend void setX(Count&, int);    // (1) إعلان تابع صديق
public:
    Count(){ x=0; }                   // (2)
    void print() const { cout<<x<<endl; }  // (3)
private:
    int x;                            // (4) حقل خاص
};

void setX(Count& c, int x){           // (5) تنفيذ التابع الصديق خارج الصف
    c.x = x;                          // (6) وصول مباشر للحقل الخاص!
}
```

#### شرح كل سطر:
1. إعلان الصداقة يُكتب **قبل** أي `public` أو `private` لأنه لا ينتمي لأي منهما — هو مجرد إذن وصول، وليس عضوًا في الصف.
2. الباني يهيّئ `x` بالقيمة `0`.
3. `print` تابع عضو عادي، موسوم بـ `const` لأنه لا يُعدّل شيئًا.
4. `x` حقل خاص — لا يمكن الوصول إليه من خارج الصف عادة.
5. `setX` يُعرَّف **خارج** الصف تمامًا، بدون `Count::` لأنه ليس تابعًا عضوًا، بل تابع عادي حصل على إذن خاص.
6. بفضل إعلان `friend`، يستطيع `setX` الوصول إلى `c.x` مباشرة رغم أن `x` خاص (`private`).

#### 🔄 قبل / بعد: الوصول لحقل خاص بدون وبمع friend

**قبل (بدون friend):**
```cpp
void setX(Count& c, int x){
    c.x = x;   // ❌ خطأ: x عضو خاص، لا يمكن الوصول إليه من خارج الصف
}
```

**بعد (مع friend):**
```cpp
class Count{
    friend void setX(Count&, int);
    // ...
};
void setX(Count& c, int x){
    c.x = x;   // ✅ مسموح بفضل إعلان friend
}
```

**ماذا تغيّر؟** إعلان `friend void setX(Count&, int);` داخل الصف منح التابع الخارجي `setX` إذن الوصول المباشر إلى الحقل الخاص `x`.

### 12. التعامل مع المؤشرات (Pointers to Objects)

#### النص الأصلي يقول:
> «في حال تعريف الغرض بالاعتماد على المؤشر يجب استخدام الكلمة new التي تستدعي باني الغرض. يجب ألا ننسى مقابل كل تعليمة new يجب علينا أن نستدعي تعليمة delete. كل استدعاء لعملية delete يستدعي الهادم الخاص بالصف.»

#### الشرح المبسّط:
حتى الآن أنشأنا الأغراض مباشرة على المكدس (Stack) مثل `Time t(10,12,13);`. لكن يمكن أيضًا إنشاء غرض على الكومة (Heap) باستخدام مؤشر و`new`:

```cpp
Class1 *x = new Class1();       // (1) استدعاء الباني الافتراضي
Class1 *y = new Class1(1,2);    // (2) استدعاء باني بوسيطين
Class1 *z = new Class1(x);      // (3) استدعاء باني نسخ أو باني يقبل مؤشر

delete x;                        // (4) استدعاء الهادم ثم تحرير الذاكرة
delete y;                        // (5)
delete z;                        // (6)
```

#### شرح كل سطر:
1. `new Class1()` يحجز ذاكرة على الكومة وينادي الباني الافتراضي، ثم يُعيد **مؤشرًا** إلى الغرض الجديد يُخزَّن في `x`.
2. مثل السطر السابق لكن باستدعاء باني يأخذ وسيطين.
3. باني يستقبل مؤشرًا آخر (`x`) كوسيط — قد يكون باني نسخ مخصص.
4–6. كل استدعاء لـ `delete` يستدعي **هادم** (destructor) الصف أولًا (لتحرير أي موارد داخلية)، ثم يُحرّر الذاكرة المحجوزة على الكومة.

#### مهم للامتحان ⚠️:
> **كل** `new` يجب أن يقابله `delete` واحد بالضبط. نسيان `delete` يُسبب **تسرّب ذاكرة (Memory Leak)**؛ استدعاء `delete` أكثر من مرة على نفس المؤشر يُسبب سلوكًا غير معرَّف (Undefined Behavior).

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** ماذا يحدث إذا كتبت `Class1 x = Class1();` (بدون مؤشر ولا `new`) ثم لم تستدعِ `delete`؟
> **لماذا هذا مهم؟** لأن الأغراض المُنشأة على المكدس (بدون `new`) تُهدَم تلقائيًا عند نهاية نطاقها (scope) — لا حاجة لـ `delete` أصلًا، والمشكلة تظهر فقط مع أغراض `new`/الكومة.

### 13. العناصر الساكنة `Static`

#### النص الأصلي يقول:
> «العناصر الساكنة هي عناصر (حقول وتوابع) معرَّفة على أنها ساكنة أي مسبوقة بالكلمة static. تكون القيمة للحقل مشتركة بين كل الأغراض من الصف. في حال كان تابعًا ساكنًا لن يتعامل مع مؤشر this. لا يمكن للتابع الساكن أن يصل لعناصر غير ساكنة ضمن الغرض.»

#### الشرح المبسّط:
كل غرض عادي يملك **نسخته الخاصة** من الحقول (مثل `hour`, `minute` لكل غرض `Time`). لكن أحيانًا نحتاج قيمة **واحدة مشتركة** بين كل الأغراض من نفس الصف — مثلًا: "عدد الموظفين الحاليين" في صف `Employee`. هذه هي وظيفة الحقل الساكن `static`.

**الفروق الجوهرية بين العنصر العادي والساكن:**

| الخاصية | عنصر عادي (Non-static) | عنصر ساكن (`static`) |
| --- | --- | --- |
| مكان التخزين | نسخة منفصلة لكل غرض | نسخة واحدة مشتركة بين كل الأغراض |
| الوصول من التابع | عبر `this` ضمنيًا | لا يوجد `this` لأنه لا ينتمي لغرض واحد |
| متى تُهيَّأ (الحقول) | في الباني، لكل غرض | مرة واحدة خارج الصف قبل أي غرض |
| الوصول لعناصر غير ساكنة | مسموح (عبر `this`) | **ممنوع** — لا يوجد غرض محدد يشير إليه `this` |

#### 💡 التشبيه:
> تخيّل صندوق بريد مشترك في مبنى سكني (`static`) يستخدمه كل السكان (كل الأغراض)، مقابل صندوق بريد خاص بكل شقة (حقل عادي). عداد الرسائل في الصندوق المشترك يتغيّر بغض النظر عن أي شقة أرسلت الرسالة.
> **وجه الشبه:** الصندوق المشترك = حقل `static` | صندوق كل شقة = حقل عادي غير ساكن.

### 14. تعريف حقل ساكن وتوابع مرتبطة به

#### النص الأصلي يقول:
> كود صف `Employee` مع حقل `static int count;` وتابع `static int getCount();`، حيث يزداد `count` عند بناء كل موظف، وينقص عند هدمه.

#### الشرح المبسّط:
```cpp
class Employee{
public:
    static int getCount();                              // (1) تابع ساكن
    Employee(string firstName, string LastName){         // (2) الباني
        this->firstName = firstName;
        this->lastName = lastName;
        ++count;                                          // (3) زيادة العداد المشترك
    }
    ~Employee(){ --count; }                               // (4) الهادم ينقص العداد
    // getters and setters
private:
    string firstName, lastName;                           // (5) حقول عادية
    static int count;                                      // (6) حقل ساكن
};
```

#### شرح كل سطر:
1. `getCount` تابع ساكن — يمكن استدعاؤه دون الحاجة لأي غرض من الصف.
2. الباني يهيّئ الحقول العادية الخاصة بكل غرض.
3. `++count;` يزيد **الحقل المشترك** بمقدار واحد مع كل غرض جديد — التغيير يظهر لكل الأغراض لأن النسخة واحدة فقط.
4. الهادم يُنقص العداد المشترك عند تدمير أي غرض من الصف.
5. `firstName`, `lastName` حقول عادية — لكل غرض نسخته الخاصة.
6. `static int count;` هذا مجرد **إعلان** ضمن الصف — التعريف والتهيئة الفعلية تتم خارج الصف (راجع القسم التالي).

### 15. تهيئة الحقل الساكن وتعريف التابع الساكن خارج الصف

#### النص الأصلي يقول:
> «تهيئة عداد الموظفين بالقيمة 0 عند بدء تنفيذ الملف. تعريف تابع getCount الساكن.»

#### الشرح المبسّط:
```cpp
int Employee::count = 0;                       // (1) تهيئة الحقل الساكن مرة واحدة فقط

int Employee::getCount(){                       // (2) تعريف التابع الساكن
    return count;                                // (3) وصول مباشر للحقل الساكن بدون this
}
```

#### شرح كل سطر:
1. حقول `static` **يجب** تهيئتها خارج الصف باستخدام `اسم_الصف::اسم_الحقل`، وذلك **مرة واحدة فقط** في كامل البرنامج، عادة قبل بداية `main`.
2. تعريف التابع الساكن يشبه أي تابع عضو عادي (`اسم_الصف::اسم_التابع`) لكن دون الحاجة لكلمة `static` هنا (تُكتب فقط في الإعلان داخل الصف).
3. `return count;` — لاحظ عدم وجود `this->count`؛ التابع الساكن لا يملك `this` لأنه لا يرتبط بغرض واحد بعينه.

#### مهم للامتحان ⚠️:
> كلمة `static` تُكتب فقط في **إعلان** الحقل/التابع داخل الصف. عند **تعريف** التابع أو تهيئة الحقل خارج الصف، **لا تُكرَّر** كلمة `static`.

### 16. استدعاء العناصر الساكنة

#### النص الأصلي يقول:
> كود `main` يبني غرضين `Employee`، يطبع `Employee::getCount()` قبل وبعد حذفهما.

#### الشرح المبسّط:
```cpp
int main(){
    Employee *e1 = new Employee("James","Kirck");   // (1)
    Employee *e2 = new Employee("Han","Solo");        // (2)
    cout << Employee::getCount();                     // (3) طباعة 2
    delete e1;                                          // (4)
    delete e2;                                          // (5)
    cout << Employee::getCount();                     // (6) طباعة 0
}
```

#### شرح كل سطر:
1. إنشاء غرض أول → الباني يزيد `count` من 0 إلى 1.
2. إنشاء غرض ثانٍ → `count` يصبح 2.
3. استدعاء `getCount` **باستخدام اسم الصف مباشرة** (`Employee::getCount()`) وليس عبر غرض — هذا ممكن لأن التابع ساكن ولا يحتاج غرضًا محددًا. الناتج: `2`.
4–5. حذف الغرضين → الهادم يُنقص `count` مرتين، فيعود إلى `0`.
6. طباعة `getCount()` مجددًا → الناتج: `0`.

> **الناتج المتوقع:**
> ```
> 20
> ```
> (طباعة `2` ثم `0` متتاليتين بدون فاصل لأن الكود لا يستخدم `endl` أو `" "` بين النداءين)

#### 🔍 تتبع التنفيذ: عداد الموظفين الساكن

**المدخل:** إنشاء غرضين ثم حذفهما بالترتيب أعلاه

| الخطوة | العملية | قيمة `Employee::count` |
| --- | --- | --- |
| 1 | `new Employee("James","Kirck")` | 1 |
| 2 | `new Employee("Han","Solo")` | 2 |
| 3 | `cout << getCount()` | يطبع 2 (القيمة تبقى 2) |
| 4 | `delete e1` | 1 |
| 5 | `delete e2` | 0 |
| 6 | `cout << getCount()` | يطبع 0 |

**النتيجة:** `20`

### 17. التحميل الزائد للعمليات (`Operator Overloading`)

#### النص الأصلي يقول:
> «مثل السجلات، نمتلك القدرة على تنفيذ التحميل الزائد للعمليات على الصفوف. سنطبق ذلك على صف (عوضًا عن سجل) يعبر عن الأعداد العقدية.»

#### الشرح المبسّط:
تمامًا كما رأينا في السجلات (`struct`) سابقًا، يمكن للصفوف (`class`) أيضًا أن "تُحمّل" عوامل التشغيل مثل `+`, `-`, `<<` لتعمل بشكل طبيعي مع أنماط بيانات جديدة عرّفناها بأنفسنا — هنا صف `Complex` يمثّل الأعداد العقدية (Complex Numbers).

```cpp
class Complex{
private:
    double real;                                        // (1)
    double imaginary;                                    // (2)
public:
    Complex(double real=0, double imaginary=0){          // (3)
        this->real = real;
        this->imaginary = imaginary;
    }
    // getters and setters
    Complex& operator+(Complex& other){                  // (4) تحميل زائد للعملية +
        this->real += other.real;                         // (5)
        this->imaginary += other.imaginary;                // (6)
        return *this;                                      // (7)
    }
};
```

#### شرح كل سطر:
1–2. الجزء الحقيقي (`real`) والجزء التخيلي (`imaginary`) لعدد عقدي.
3. الباني يهيّئ الجزأين بقيم افتراضية `0`.
4. `operator+` هو **تحميل زائد** لعملية الجمع `+` كتابع عضو — يستقبل غرضًا آخر `other` من نفس الصف، ويُعيد مرجعًا لغرض من نفس الصف.
5–6. تُجمَع الأجزاء الحقيقية والتخيلية مع نظيرتها في `other`، ويُعدَّل الغرض الحالي مباشرة (جمع تراكمي داخل نفس الغرض).
7. `return *this;` يُعيد **مرجعًا** إلى الغرض الحالي بعد التعديل، ليتيح تسلسل العمليات مثل `a+b+c`.

#### الفهم الخاطئ ❌: عملية `operator+` تُعيد دائمًا غرضًا جديدًا مختلفًا عن الطرفين.
#### الفهم الصحيح ✅: في هذا التصميم بالذات، `operator+` تُعدّل الغرض الذي استُدعيت عليه (`this`) ثم تُعيد مرجعًا له — أي أنها أقرب لسلوك `+=` من `+` الرياضي الخالص. يجب قراءة جسم التابع بدقة لمعرفة السلوك الفعلي، لا الاعتماد على التسمية فقط.

### 18. عندما لا تُعيد العملية غرضًا من نفس الصف: التابع الصديق

#### النص الأصلي يقول:
> «قمنا بتنفيذ تحميل زائد للعملية + لتتقبل الأعداد العقدية (بشكل مشابه يمكننا تنفيذ التحميل الزائد لعمليات الطرح والضرب والقسمة...إلخ). في حال كانت العملية لا تعيد غرضًا من نفس الصف، نعرّفه كتابع صديق.»

#### الشرح المبسّط:
عملية `<<` (إخراج إلى الشاشة عبر `cout`) خاصة: الطرف الأيسر لها ليس غرضًا من صف `Complex`، بل غرض من `ostream` (مثل `cout`). لذلك **لا يمكن** تعريفها كتابع عضو عادي داخل `Complex` (لأن التابع العضو يفترض أن `this` هو الطرف الأيسر دائمًا). الحل: تعريفها كـ **تابع صديق**.

```cpp
class Complex{
    friend ostream& operator<<(ostream& out, Complex& c){   // (1)
        out << c.real << " "
            << ((c.imaginary >= 0) ? "+" : "")
            << c.imaginary << "i" << endl;                    // (2)
        return out;                                            // (3) — يُفترض إعادة out (انظر ملاحظة)
    }
    // the others (operator-, operator*, operator/, ...)
};
```

#### شرح كل سطر:
1. `friend ostream& operator<<(ostream& out, Complex& c)` — تعريف تابع صديق يستقبل مجرى الإخراج `out` (الطرف الأيسر، مثل `cout`) والغرض العقدي `c` (الطرف الأيمن)؛ وهذا ما يسمح بكتابة `cout << complexObject;`.
2. طباعة الجزء الحقيقي، ثم إشارة `+` إن كان الجزء التخيلي موجبًا أو صفرًا (وإلا لا تُطبع إشارة لأن السالب يحمل إشارته الخاصة)، ثم الجزء التخيلي متبوعًا بالحرف `i` (رمز الوحدة التخيلية).
3. من الضروري إعادة `out` (المرجع لمجرى الإخراج) للسماح بتسلسل عمليات `<<` المتعددة مثل `cout << c1 << c2;`.

#### ⚖️ المقايضة: تابع عضو مقابل تابع صديق للتحميل الزائد

| | تابع عضو (Member Function) | تابع صديق (Friend Function) |
| --- | --- | --- |
| المزايا | وصول مباشر لحقول `this` بدون وسائط إضافية | يسمح بأن يكون الطرف الأيسر من نوع مختلف تمامًا عن الصف |
| العيوب | يفرض أن يكون الطرف الأيسر دائمًا غرضًا من نفس الصف | يحتاج إعلان صريح داخل الصف؛ لا يملك `this` |
| متى تختاره | عندما تكون العملية تُعدّل أو تعتمد أساسًا على الغرض الحالي (`this`) وتُعيد غرضًا من نفس الصف | عندما يكون الطرف الأيسر من نوع خارجي (مثل `ostream&` في `<<`) أو عندما تريد تناظرًا (symmetry) بين الطرفين |

## الجزء الثاني: ملخص منظم

### أهم التعاريف والمفاهيم
| المصطلح | التعريف | مثال/ملاحظة |
| --- | --- | --- |
| `const object` | غرض لا يمكن تعديل أي من حقوله بعد إنشائه | `const Time c(10,12,13);` |
| `const member function` | تابع عضو موسوم بـ `const` يضمن عدم تعديل حالة الغرض | `int getHour() const;` |
| `const data member` | حقل ثابت داخل الصف، يُهيَّأ مرة واحدة فقط | `const int increment;` |
| `Initialization List` | قائمة تهيئة تُنفَّذ قبل جسم الباني، وحيدة الوسيلة لتهيئة حقول `const` | `Increment(int i):increment(i){}` |
| `friend function` | تابع خارجي (لا ينتمي للصف) له إذن وصول للحقول الخاصة/المحمية | `friend void setX(Count&, int);` |
| `friend class` | صف خارجي كليًا يملك إذن وصول لحقول صف آخر الخاصة | صف صديق لصف آخر بدون علاقة وراثة أو تراكب |
| `static data member` | حقل مشترك بين كل أغراض الصف، نسخة واحدة فقط | `static int count;` |
| `static member function` | تابع لا يرتبط بغرض محدد، لا يملك `this` | `static int getCount();` |
| `operator overloading` | إعادة تعريف سلوك عامل تشغيل ليعمل مع نمط بيانات معرَّف من المستخدم | `Complex& operator+(Complex&);` |

### المكونات الرئيسية (مرجع سريع)
| الأداة | الوظيفة | ملاحظة |
| --- | --- | --- |
| `const` قبل الغرض | يجعل كل حقول الغرض ثابتة | يسبق تعريف الغرض مباشرة |
| `const` بعد توقيع التابع | يضمن عدم تعديل التابع لحالة الغرض | يُكتب في الإعلان والتعريف معًا |
| `:` قبل جسم الباني | يبدأ قائمة التهيئة | الطريقة الوحيدة لتهيئة حقول `const` |
| `friend` | يمنح إذن وصول لتابع/صف خارجي | يُكتب قبل `public`/`private` |
| `new` / `delete` | حجز/تحرير ذاكرة الغرض على الكومة | كل `new` يقابله `delete` واحد بالضبط |
| `static` | يُشرك القيمة/التابع بين كل الأغراض | تُهيَّأ الحقول خارج الصف مرة واحدة |
| `operator{symbol}` | يُعيد تعريف سلوك عامل تشغيل | تابع عضو أو صديق حسب موقع الطرف الأيسر |

### جداول مقارنات سريعة
| المقارنة | الأول | الثاني | الفرق |
| --- | --- | --- | --- |
| غرض عادي مقابل غرض ثابت | `Time t;` | `const Time c;` | الأول يقبل استدعاء كل التوابع؛ الثاني يقبل فقط توابع `const` |
| تابع صديق مقابل تابع عضو | `friend void f(Count&,int);` | `void f(int)` داخل الصف | الصديق لا يملك `this` ولا ينتمي للصف؛ العضو يملك `this` ضمنيًا |
| حقل عادي مقابل حقل ساكن | `int x;` | `static int x;` | الأول نسخة لكل غرض؛ الثاني نسخة واحدة مشتركة لكل الأغراض |
| `operator+` كعضو مقابل كصديق | `Complex& operator+(Complex&)` | `friend ostream& operator<<(ostream&, Complex&)` | العضو: الطرف الأيسر غرض من نفس الصف؛ الصديق: الطرف الأيسر نوع خارجي |

### قاموس المصطلحات
| الفئة | المصطلحات |
| --- | --- |
| ثبات | `const object`, `const function`, `const data member`, `Initialization List` |
| الصداقة | `friend function`, `friend class`, `encapsulation` |
| الذاكرة والمؤشرات | `new`, `delete`, `pointer`, `heap`, `memory leak` |
| العناصر المشتركة | `static field`, `static function`, `this` |
| التحميل الزائد | `operator overloading`, `operator+`, `operator<<`, `ostream&` |

### أبرز النقاط الذهبية
1. `const` قبل الغرض تُجمّد كل حقوله؛ `const` بعد توقيع التابع تضمن أنه لا يُعدّل شيئًا.
2. الحقول الثابتة (`const data members`) لا يمكن تهيئتها إلا عبر قائمة التهيئة، لا داخل جسم الباني.
3. الصداقة (`friend`) تُمنح صراحة من الصف المالك، وليست تلقائية ولا تبادلية ولا متعدية.
4. لكل استدعاء `new` يجب أن يقابله استدعاء `delete` واحد بالضبط، وإلا يحدث تسرّب ذاكرة.
5. الحقول والتوابع الساكنة (`static`) مشتركة بين كل أغراض الصف، ولا تملك التوابع الساكنة مؤشر `this`.
6. التحميل الزائد للعمليات (`operator overloading`) يُكتب كتابع عضو إن كان الطرف الأيسر غرضًا من نفس الصف، وكتابع صديق إن لم يكن كذلك (مثل `<<`).

### الأخطاء الشائعة عند الطلاب ⚠️
| الخطأ | التصحيح |
| --- | --- |
| استدعاء تابع غير `const` على غرض `const` | وسم التابع بـ `const` إذا كان لا يُعدّل أي حقل، أو استخدام غرض غير ثابت |
| محاولة إسناد قيمة لحقل `const` داخل جسم الباني بـ `=` | استخدام قائمة التهيئة `: field(value)` بدلًا من الإسناد العادي |
| كتابة كلمة `static` عند تعريف تابع/تهيئة حقل ساكن خارج الصف | إزالة `static` من التعريف الخارجي؛ تُكتب فقط في إعلان الصف |
| نسيان `delete` مقابل `new` | التحقق من أن كل `new` له `delete` مطابق قبل انتهاء البرنامج |
| كتابة عامل `<<` كتابع عضو داخل الصف | كتابته كتابع `friend` لأن الطرف الأيسر (`ostream`) ليس غرضًا من الصف |
| اعتقاد أن الصداقة علاقة تبادلية تلقائية | كل صف يجب أن يُعلن أصدقاءه بنفسه صراحة |

### خطوات وإجراءات المحاضرة

#### ⚙️ الخطوات / الخوارزمية: تحويل تابع عضو إلى `const`

> ما هدفه؟ التأكد من أن التابع لا يُعدّل حالة الغرض، والسماح باستدعائه على أغراض ثابتة.

```algorithm
1 | فحص جسم التابع | مراجعة يدوية | التحقق من عدم وجود إسناد لأي حقل عضو
2 | إضافة const بعد الأقواس في الإعلان | تعديل الصف | int getHour() const;
3 | إضافة const بعد الأقواس في التعريف | تعديل ملف التنفيذ | int Time::getHour() const{...}
4 | إعادة الترجمة | Compiler | التأكد من عدم وجود خطأ عدم تطابق توقيع
```

#### نقاط التنفيذ:
- يجب إضافة `const` في الإعلان والتعريف معًا وإلا يفشل الربط (linking) أو الترجمة.

#### ⚙️ الخطوات / الخوارزمية: تهيئة حقل ثابت عبر قائمة التهيئة

> ما هدفه؟ إعطاء الحقل الثابت قيمته الوحيدة قبل أن يصبح الغرض قابلًا للاستخدام.

```algorithm
1 | كتابة النقطتين بعد قائمة وسائط الباني | Constructor signature | Increment(int c, int i) :
2 | كتابة اسم الحقل والقيمة بين قوسين | Initialization List | increment(i)
3 | فصل عدة حقول بفواصل إن وجدت | Initialization List | increment(i), count(c)
4 | كتابة جسم الباني (قد يكون فارغًا) | Constructor Body | {} أو إسناد حقول غير ثابتة
```

#### نقاط التنفيذ:
- الحقول `const` والمراجع `reference` يجب أن تظهر في قائمة التهيئة دائمًا؛ لا خيار بديل لها.

#### ⚙️ الخطوات / الخوارزمية: منح صداقة لتابع خارجي

> ما هدفه؟ السماح لتابع لا ينتمي للصف بالوصول إلى حقوله الخاصة/المحمية.

```algorithm
1 | كتابة توقيع التابع مسبوقًا بـ friend | داخل تعريف الصف | friend void setX(Count&, int);
2 | وضع الإعلان قبل أي public/private | داخل تعريف الصف | لأن friend ليس عضوًا
3 | تعريف التابع خارج الصف بدون :: | خارج الصف | void setX(Count& c, int x){ c.x = x; }
4 | استخدام معامل من نوع الصف (مرجع أو مؤشر) | توقيع التابع | Count& c
```

#### نقاط التنفيذ:
- تعريف التابع الصديق لا يستخدم عامل النطاق `ClassName::` لأنه ليس عضوًا في الصف.

#### ⚙️ الخطوات / الخوارزمية: إدارة غرض عبر مؤشر (new/delete)

> ما هدفه؟ حجز غرض على الكومة عند الحاجة، وتحريره بأمان بعد الاستخدام لمنع تسرّب الذاكرة.

```algorithm
1 | إعلان مؤشر من نوع الصف | تعريف المتغير | Class1 *x;
2 | استدعاء new مع الباني المناسب | new expression | x = new Class1(1,2);
3 | استخدام الغرض عبر المؤشر | Runtime | x->someMethod();
4 | استدعاء delete عند الانتهاء | delete statement | delete x;
```

#### نقاط التنفيذ:
- استدعاء `delete` يستدعي الهادم أولًا، ثم يُحرّر الذاكرة — لا يجب استخدام المؤشر بعد ذلك.

#### ⚙️ الخطوات / الخوارزمية: إضافة حقل وتابع ساكنين للصف

> ما هدفه؟ مشاركة قيمة واحدة (مثل عداد) بين كل أغراض الصف.

```algorithm
1 | إعلان الحقل الساكن داخل الصف | Class declaration | static int count;
2 | إعلان التابع الساكن داخل الصف | Class declaration | static int getCount();
3 | تهيئة الحقل الساكن خارج الصف مرة واحدة | خارج الصف، قبل main | int Employee::count = 0;
4 | تعريف التابع الساكن خارج الصف بدون static | خارج الصف | int Employee::getCount(){ return count; }
5 | تحديث الحقل الساكن داخل الباني/الهادم | Constructor/Destructor | ++count; / --count;
6 | استدعاء التابع الساكن عبر اسم الصف | Client code | Employee::getCount();
```

#### نقاط التنفيذ:
- التابع الساكن لا يستطيع الوصول إلى الحقول غير الساكنة لأنه لا يملك `this` يشير إلى غرض محدد.

#### ⚙️ الخطوات / الخوارزمية: تحميل زائد لعملية إخراج `<<` كتابع صديق

> ما هدفه؟ السماح بطباعة غرض من صف مخصص مباشرة عبر `cout`.

```algorithm
1 | تحديد أن الطرف الأيسر ليس غرضًا من الصف | تحليل | الطرف الأيسر هو ostream (مثل cout)
2 | إعلان التابع كـ friend داخل الصف | Class declaration | friend ostream& operator<<(ostream&, Complex&);
3 | كتابة الجسم باستخدام out << ... | Function body | out << c.real << ...
4 | إعادة out بمرجع | return statement | return out;
```

#### نقاط التنفيذ:
- إعادة `ostream&` تسمح بتسلسل عمليات `<<` عدة مرات في تعليمة واحدة.

### أنماط الأكواد والبنى المتكررة

| النمط | البنية الأساسية | متى تستخدمه |
| --- | --- | --- |
| تابع عضو ثابت (const) | `ReturnType name(...) const { return field; }` | عند تابع لا يُعدّل حالة الغرض (getters, print) |
| تهيئة حقل ثابت | `Class(Type v) : field(v) { }` | عند وجود حقل `const` أو `reference` داخل الصف |
| تابع صديق يصل لحقل خاص | `friend RetType f(Class&, ...);` ثم تعريف عادي خارج الصف | عند حاجة تابع خارجي للوصول لحقل خاص/محمي |
| حقل وتابع ساكنان لعداد مشترك | `static int count;` + `++count` في الباني + `--count` في الهادم | عند حاجة قيمة واحدة مشتركة بين كل الأغراض (مثل عدّاد) |
| تحميل زائد كعضو معدِّل ومُعيد لمرجع | `Class& operator+(Class& other){ ...; return *this; }` | عند عملية تُعدّل الغرض الحالي وتحتاج تسلسلًا (chaining) |
| تحميل زائد كصديق لعملية إخراج | `friend ostream& operator<<(ostream& out, Class& c){ ...; return out; }` | عند حاجة طباعة الغرض عبر `cout` مباشرة |

### أنماط التعامل والسلوك

| السيناريو | التعامل الصحيح | لماذا؟ |
| --- | --- | --- |
| استدعاء تابع على غرض `const` | التحقق أولًا أن التابع موسوم بـ `const` | وإلا يرفض المترجم الاستدعاء تمامًا |
| حقل يجب ألا يتغير بعد الإنشاء | تعريفه `const` وتهيئته في قائمة التهيئة فقط | لضمان الثبات طوال عمر الغرض |
| تابع خارجي يحتاج وصولًا خاصًا محدودًا | استخدام `friend` بدلًا من جعل الحقل `public` بالكامل | يحافظ على التغليف (encapsulation) قدر الإمكان مع منح استثناء دقيق |
| الحاجة لقيمة مشتركة بين كل الأغراض | استخدام حقل `static` بدل حقل عادي في كل غرض | لتفادي تكرار وتضارب النسخ، ويضمن التزامن التلقائي |
| عملية تشغيل طرفها الأيسر ليس غرضًا من الصف | تعريفها كـ `friend` وليس كتابع عضو | لأن التابع العضو يفترض أن `this` هو الطرف الأيسر دائمًا |

### الأفكار الرئيسية الشاملة
> ربط كل مفاهيم المحاضرة معًا: كل هذه الأدوات (`const`, `friend`, `static`, `operator overloading`) هي وسائل للتحكم الدقيق في **من يستطيع الوصول لماذا، ومتى يمكن التعديل، وكيف تتصرف الأغراض مع العمليات القياسية**. الهدف المشترك هو صف أكثر أمانًا (`const`) ومرونة (`friend`, `operator overloading`) وكفاءة في مشاركة البيانات (`static`) دون التضحية بمبدأ التغليف الأساسي للبرمجة كائنية التوجه.

## الجزء الثالث: أسئلة اختيار من متعدد (MCQ)

> **16 سؤالاً** — مستوى: متوسط/صعب. التوزيع: مقارنات 25%، سيناريو كود 35%، تطبيق 30%، تتبع خوارزمية 10%.

### السؤال 1 (متوسط)
أي مما يلي صحيح حول الكلمة `const` عند وضعها **قبل** تعريف غرض؟
أ) تجعل التابع الأول فقط ثابتًا
ب) تجعل كل حقول الغرض ثابتة وغير قابلة للتعديل
ج) لا تؤثر على الغرض إلا إذا كان يحتوي مؤشرات
د) تمنع استدعاء أي تابع على الغرض إطلاقًا
**الإجابة الصحيحة: ب**
**التعليل:** `const` قبل الغرض تجعل كل الحقول ثابتة (ب). (أ) خاطئة لأنها لا تخص تابعًا واحدًا. (ج) خاطئة لأنها تنطبق على كل أنواع الحقول. (د) خاطئة لأنه يمكن استدعاء التوابع الموسومة بـ `const`.

### السؤال 2 (سهل)
أين تُكتب الكلمة `const` لتحديد أن تابعًا عضوًا لا يُعدّل حالة الغرض؟
أ) قبل اسم التابع مباشرة
ب) بعد قوسي المعاملات `()` في نهاية التوقيع
ج) داخل جسم التابع فقط
د) قبل نوع القيمة المُعادة فقط
**الإجابة الصحيحة: ب**
**التعليل:** الصيغة الصحيحة `ReturnType name() const;` (ب). (أ) و(د) مواقع خاطئة نحويًا لهذا الغرض. (ج) لا تؤثر لأن `const` جزء من التوقيع لا الجسم.

### السؤال 3 (صعب)
```cpp
class Time{
public:
    int getHour() const;
    void setHour(int);
};
```
أيّ سطر التالي يسبب خطأ ترجمة إذا كان `const Time c(10,0,0);`؟
أ) `c.getHour();`
ب) `int h = c.getHour();`
ج) `c.setHour(5);`
د) لا يوجد خطأ في أي سطر
**الإجابة الصحيحة: ج**
**التعليل:** `setHour` غير موسوم بـ `const`، واستدعاؤه على غرض `const` مثل `c` يُسبب خطأ ترجمة (ج). (أ) و(ب) مسموحتان لأن `getHour` موسوم بـ `const`. (د) خاطئة لوجود خطأ فعلي.

### السؤال 4 (متوسط)
ما الطريقة **الوحيدة** لتهيئة حقل `const int increment;` داخل باني الصف؟
أ) `increment = value;` داخل جسم الباني
ب) استخدام قائمة التهيئة `: increment(value)`
ج) استخدام تابع `setIncrement`
د) تهيئته مباشرة في `main`
**الإجابة الصحيحة: ب**
**التعليل:** قائمة التهيئة هي الطريقة الوحيدة لتهيئة حقول `const` (ب). (أ) خاطئة لأنها إسناد بعد الإنشاء، غير مسموح للحقول الثابتة. (ج) خاطئة لنفس السبب — أي تابع `setter` سيحاول إسناد قيمة لاحقًا. (د) غير ممكنة لأن الحقل خاص وثابت.

## الجزء الرابع: أسئلة تصحيح الكود

> غطِّ أنواع الأخطاء: ترجمة، منطقية، فحص إرجاع، dead code، سوء فهم.

### سؤال تصحيح 1 — misconception

**الكود (يحتوي خطأ):**
```cpp
class Time{
public:
    int getHour();          // (1)
private:
    int hour;
};
int main(){
    const Time c(10,0,0);
    cout << c.getHour();    // (2)
}
```
**اكتشف الخطأ:** استدعاء `getHour()` على الغرض الثابت `c` رغم أن `getHour` غير موسومة بـ `const`، مما يُسبب خطأ ترجمة.

**التصحيح:**
```cpp
class Time{
public:
    int getHour() const;    // (1) أضيفت const
private:
    int hour;
};
int main(){
    const Time c(10,0,0);
    cout << c.getHour();    // (2) أصبح صحيحًا الآن
}
```
**شرح الحل:**
1. أُضيفت `const` بعد `()` في إعلان `getHour` لأن التابع لا يُعدّل أي حقل.
2. الآن يمكن استدعاء `getHour` على الغرض `c` الثابت دون أي خطأ.
3. القاعدة العامة: أي تابع للقراءة فقط يجب وسمه `const` منذ البداية.

### سؤال تصحيح 2 — logic

**الكود (يحتوي خطأ):**
```cpp
class Increment{
public:
    Increment(int c=0, int i=1){    // (1)
        count = c;
        increment = i;              // (2)
    }
private:
    int count;
    const int increment;
};
```
**اكتشف الخطأ:** محاولة إسناد قيمة للحقل الثابت `increment` داخل جسم الباني بـ `=`، وهو غير مسموح لحقول `const`.

**التصحيح:**
```cpp
class Increment{
public:
    Increment(int c=0, int i=1) : increment(i){   // (1) قائمة تهيئة
        count = c;                                  // (2) إسناد عادي مسموح لحقل غير ثابت
    }
private:
    int count;
    const int increment;
};
```
**شرح الحل:**
1. نُقلت تهيئة `increment` إلى قائمة التهيئة `: increment(i)` لأنها الطريقة الوحيدة المسموحة لحقل `const`.
2. `count` بقي إسنادًا عاديًا في جسم الباني لأنه ليس ثابتًا.
3. لو تُرك الكود كما هو، سيرفضه المترجم برسالة خطأ حول محاولة إسناد قيمة لحقل `const` بعد بنائه.

### سؤال تصحيح 3 — misconception (friend)

**الكود (يحتوي خطأ):**
```cpp
class Count{
public:
    friend void setX(Count&, int);   // (1)
    Count(){ x=0; }
private:
    int x;
};
void setX(Count& c, int x){
    c.x = x;
}
int main(){
    Count c1;
    setX::c1(5);   // (2)
}
```
**اكتشف الخطأ:** استدعاء `setX::c1(5);` خاطئ نحويًا — `setX` تابع صديق عادي (غير عضو)، فيجب استدعاؤه كدالة عادية تُمرَّر إليها الغرض كوسيط، لا باستخدام `::`.

**التصحيح:**
```cpp
int main(){
    Count c1;
    setX(c1, 5);   // (2) الاستدعاء الصحيح لتابع صديق
}
```
**شرح الحل:**
1. التابع الصديق ليس عضوًا في الصف، لذلك لا يُستدعى بصيغة `object.method()` ولا `Class::method()`.
2. يُستدعى كأي دالة عادية: `setX(c1, 5);` — مع تمرير الغرض كوسيط أول.
3. الفرق بين موقع إعلان `friend` (داخل الصف) وطريقة الاستدعاء (خارج الصف كدالة عادية) هو نقطة الالتباس الأكثر شيوعًا.

### سؤال تصحيح 4 — return_check (memory)

**الكود (يحتوي خطأ):**
```cpp
void createEmployees(){
    Employee *e1 = new Employee("Ali","Hassan");
    Employee *e2 = new Employee("Sara","Nour");
    cout << Employee::getCount();
    delete e1;
    // نسي المبرمج حذف e2
}
```
**اكتشف الخطأ:** لم يُستدعَ `delete e2;`، مما يُسبب تسرّب ذاكرة (memory leak) — الذاكرة المحجوزة لـ `e2` تبقى محجوزة دون تحرير، ولن يُستدعى هادمه أبدًا.

**التصحيح:**
```cpp
void createEmployees(){
    Employee *e1 = new Employee("Ali","Hassan");
    Employee *e2 = new Employee("Sara","Nour");
    cout << Employee::getCount();
    delete e1;
    delete e2;   // أُضيف الحذف المفقود
}
```
**شرح الحل:**
1. كل `new` يجب أن يُقابله `delete` واحد بالضبط قبل خروج المؤشر من نطاق استخدامه.
2. نسيان `delete e2` يعني بقاء `Employee::count` أكبر من قيمته الحقيقية بعد الخروج من الدالة، لأن الهادم لم يُستدعَ لتصحيح العداد.
3. تسرّبات الذاكرة تتراكم مع كل استدعاء متكرر لهذه الدالة دون تصحيح.

### سؤال تصحيح 5 — dead_code (static access)

**الكود (يحتوي خطأ):**
```cpp
class Employee{
public:
    static int getCount(){
        if(count > 0){
            return count;
        }
        return -1;
        cout << "unreachable";   // (1) شيفرة ميتة
    }
private:
    static int count;
};
int Employee::count = 0;
```
**اكتشف الخطأ:** السطر `cout << "unreachable";` بعد `return -1;` هو شيفرة ميتة (Dead Code) لن تُنفَّذ أبدًا لأن `return` تُنهي التابع فورًا.

**التصحيح:**
```cpp
class Employee{
public:
    static int getCount(){
        if(count > 0){
            return count;
        }
        return -1;
    }
private:
    static int count;
};
int Employee::count = 0;
```
**شرح الحل:**
1. أي كود يأتي بعد جملة `return` مباشرة في نفس المسار التنفيذي لا يمكن الوصول إليه إطلاقًا.
2. تمت إزالة السطر الميت لأنه لا فائدة منه ويسبب تحذير ترجمة (unreachable code warning) في أغلب المترجمات.
3. لو أراد المبرمج فعلًا تنفيذ طباعة تحذيرية، وجب وضعها **قبل** `return -1;` لا بعدها.

## الجزء الرابع: بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ما الفرق بين وضع `const` قبل الغرض ووضعها بعد توقيع التابع؟
A: قبل الغرض تعني أن الغرض بأكمله ثابت؛ بعد توقيع التابع تعني أن هذا التابع تحديدًا لا يُعدّل حالة الغرض.

**Q2:** لماذا لا يمكن استدعاء تابع غير `const` على غرض `const`؟
A: لأن المترجم لا يستطيع ضمان أن التابع لن يُعدّل الغرض، فيرفض الاستدعاء كليًا احتياطًا.

**Q3:** ما الطريقة الوحيدة لتهيئة حقل `const` داخل صف؟
A: قائمة التهيئة (Initialization List) بعد النقطتين `:` وقبل جسم الباني.

**Q4:** ما هو التابع الصديق (`friend function`)؟
A: تابع لا ينتمي للصف لكنه يملك إذن وصول لحقوله الخاصة والمحمية.

**Q5:** هل علاقة الصداقة تبادلية؟
A: لا، إذا كان A صديقًا لـ B فهذا لا يعني تلقائيًا أن B صديق لـ A.

**Q6:** هل علاقة الصداقة متعدية؟
A: لا، صداقة A مع B وB مع C لا تعني بالضرورة صداقة A مع C.

**Q7:** ماذا يجب أن يقابل كل استدعاء لـ `new`؟
A: استدعاء واحد بالضبط لـ `delete` لتحرير الذاكرة المحجوزة ومنع تسرّب الذاكرة.

**Q8:** ماذا يحدث عند استدعاء `delete` على مؤشر لغرض؟
A: يُستدعى هادم الصف (destructor) أولًا، ثم تُحرَّر الذاكرة المحجوزة.

**Q9:** ما الفرق بين الحقل العادي والحقل `static`؟
A: الحقل العادي له نسخة منفصلة لكل غرض؛ الحقل `static` له نسخة واحدة مشتركة بين كل أغراض الصف.

**Q10:** لماذا لا يملك التابع الساكن مؤشر `this`؟
A: لأنه لا يرتبط بغرض محدد بل بالصف ككل، فلا معنى لوجود مؤشر يشير لغرض واحد.

**Q11:** أين تُهيَّأ الحقول الساكنة غير الثابتة؟
A: خارج الصف، مرة واحدة فقط، باستخدام الصيغة `اسم_الصف::اسم_الحقل = قيمة;`.

**Q12:** متى نستخدم تابعًا صديقًا بدلًا من تابع عضو عند التحميل الزائد لعملية؟
A: عندما يكون الطرف الأيسر للعملية من نوع خارجي وليس غرضًا من نفس الصف، كما في حالة `operator<<` مع `ostream`.

**Q13:** ماذا يجب أن يُعيد `operator<<` عند تعريفه كتابع صديق؟
A: يجب أن يُعيد مرجعًا لمجرى الإخراج (`ostream&`) للسماح بتسلسل عمليات `<<` متعددة.

**Q14:** هل يمكن استدعاء تابع `const` على غرض عادي (غير `const`)؟
A: نعم، تابع `const` يمكن استدعاؤه دائمًا سواء كان الغرض ثابتًا أو عاديًا.

## الجزء الخامس: كتابة الكود الكامل (مرجع شامل)

> مرجع واحد يجمع كل أمثلة المحاضرة (`Time`, `Increment`, `Count`, `Employee`, `Complex`) في مكان واحد، بدون شرح إضافي.

```cpp
#include <iostream>
#include <string>
using namespace std;

// ===== 1) Time: const objects and const member functions =====
class Time{
private:
    int hour, minute, second;
public:
    Time(int h=0, int m=0, int s=0){ setTime(h,m,s); }
    void setHour(int h){ hour=(h>=0 && h<24)? h:0; }
    void setMinute(int m){ minute=(m>=0 && m<60)? m:0; }
    void setSecond(int s){ second=(s>=0 && s<60)? s:0; }
    void setTime(int h,int m,int s){ setHour(h); setMinute(m); setSecond(s); }
    int getHour() const { return hour; }
    int getMinute() const { return minute; }
    int getSecond() const { return second; }
    void printMilitary() const { cout<<hour<<minute<<second; }
    void printStandard(){
        cout<<((hour>12)?hour-12:hour)<<":"<<minute<<":"<<second
            <<" "<<((hour>=12)?"PM":"AM");
    }
};

// ===== 2) Increment: const data member with initialization list =====
class Increment{
public:
    Increment(int c=0, int i=1): increment(i), count(c) {}
    void addIncrement(){ count += increment; }
    void print() const {
        cout << "count = " << count << ", increment = " << increment << endl;
    }
private:
    int count;
    const int increment;
};

// ===== 3) Count: friend function =====
class Count{
    friend void setX(Count&, int);
public:
    Count(){ x=0; }
    void print() const { cout<<x<<endl; }
private:
    int x;
};
void setX(Count& c, int x){
    c.x = x;
}

// ===== 4) Employee: static field and static member function =====
class Employee{
public:
    static int getCount();
    Employee(string firstName, string lastName){
        this->firstName = firstName;
        this->lastName = lastName;
        ++count;
    }
    ~Employee(){ --count; }
private:
    string firstName, lastName;
    static int count;
};
int Employee::count = 0;
int Employee::getCount(){ return count; }

// ===== 5) Complex: operator overloading (member + friend) =====
class Complex{
private:
    double real;
    double imaginary;
public:
    Complex(double real=0, double imaginary=0){
        this->real = real;
        this->imaginary = imaginary;
    }
    Complex& operator+(Complex& other){
        this->real += other.real;
        this->imaginary += other.imaginary;
        return *this;
    }
    friend ostream& operator<<(ostream& out, Complex& c){
        out << c.real << " "
            << ((c.imaginary >= 0) ? "+" : "")
            << c.imaginary << "i" << endl;
        return out;
    }
};

int main(){
    // Time
    Time t(10,12,13);
    const Time c(10,12,13);
    t.setHour(15);
    cout << t.getMinute() << endl;
    c.printMilitary();

    // Increment
    Increment inc(0, 5);
    inc.addIncrement();
    inc.print();

    // Count (friend function)
    Count cnt;
    setX(cnt, 42);
    cnt.print();

    // Employee (static)
    Employee *e1 = new Employee("James","Kirck");
    Employee *e2 = new Employee("Han","Solo");
    cout << Employee::getCount();
    delete e1;
    delete e2;
    cout << Employee::getCount();

    // Complex (operator overloading)
    Complex a(1,2), b(3,4);
    a + b;
    cout << a;

    return 0;
}
```

> **الناتج المتوقع (لقطة الشاشة):**
> مخرجات مختلطة تشمل: قيمة `getMinute()`، طباعة عسكرية للوقت، طباعة `count` و`increment`، طباعة `42` من صف `Count`، طباعة `2` ثم `0` من عداد الموظفين، وطباعة عدد عقدي بصيغة `real +imaginaryi`.

## الجزء الخامس: أسئلة نظرية متوقعة بالامتحان

### سؤال 1: لماذا نحتاج إلى وسم بعض التوابع الأعضاء بـ `const`؟
**نموذج الإجابة:**
1. التعريف: `const` بعد توقيع تابع عضو تعني أن هذا التابع لا يُعدّل أي حقل في الغرض.
2. المكونات/الشروط: يجب ألا يحتوي جسم التابع أي إسناد لحقول الغرض (إلا حقولًا معرَّفة بـ `mutable`، غير مطروحة في هذه المحاضرة).
3. مثال: `int getHour() const { return hour; }`.
4. متى نستخدم: عند تعريف أي تابع للقراءة فقط، خصوصًا إذا كان من المتوقع استخدام الصف مع أغراض `const` لاحقًا.

### سؤال 2: ما وظيفة قائمة التهيئة (Initialization List) في الباني؟
**نموذج الإجابة:**
1. التعريف: قسم يُكتب بعد النقطتين `:` وقبل جسم الباني، يُهيّئ فيه حقول محددة بقيم مباشرة قبل تنفيذ أي كود داخل الباني.
2. المكونات/الشروط: صيغتها `field(value)` مفصولة بفواصل عند تعدد الحقول.
3. مثال: `Increment(int i): increment(i) {}`.
4. متى نستخدم: إلزامي لحقول `const` والمراجع `reference`؛ مفيد أيضًا لتحسين الأداء مع الحقول من نوع صف آخر (تجنب التهيئة المزدوجة).

### سؤال 3: ما الفرق بين التابع العضو والتابع الصديق؟
**نموذج الإجابة:**
1. التعريف: التابع العضو ينتمي للصف ويملك `this` ضمنيًا؛ التابع الصديق خارجي عن الصف لكن له إذن وصول لحقوله الخاصة/المحمية.
2. المكونات/الشروط: التابع الصديق يُعلَن داخل الصف بكلمة `friend` لكنه يُعرَّف ويُستدعى خارج الصف كدالة عادية.
3. مثال: `friend void setX(Count&, int);` مقارنة بتابع عضو عادي مثل `void print() const;`.
4. متى نستخدم: التابع الصديق عند الحاجة لوصول خارجي محدود دون كسر التغليف بالكامل، أو عند عدم توافق التابع مع نمط "الطرف الأيسر دائمًا غرض من الصف".

### سؤال 4: هل الصداقة (`friend`) علاقة وراثة أو تراكب؟ وضّح.
**نموذج الإجابة:**
1. التعريف: لا، الصداقة علاقة مستقلة تمامًا لا ترتبط بالوراثة (`inheritance`) ولا بالتراكب (`composition`).
2. المكونات/الشروط: الصف/التابع الصديق كيان خارجي كليًا، لا يرث من الصف المانح ولا يحتويه ولا يُحتوى فيه.
3. مثال: `friend class Bank;` داخل `Wallet` لا يعني أن `Bank` صف مشتق من `Wallet` أو يحتوي غرضًا منه.
4. متى نستخدم: عند الحاجة لمنح إذن وصول محدد دون إنشاء أي علاقة هيكلية (structural) أخرى بين الصفين.

### سؤال 5: ما الفرق بين حقل عادي وحقل `static` من حيث مكان التخزين؟
**نموذج الإجابة:**
1. التعريف: الحقل العادي يُخزَّن ضمن كل غرض على حدة؛ الحقل الساكن يُخزَّن مرة واحدة، مشتركًا بين كل الأغراض من نفس الصف.
2. المكونات/الشروط: الحقل الساكن يُهيَّأ خارج الصف مرة واحدة قبل بدء استخدام أي غرض.
3. مثال: `static int count;` في `Employee` مقارنة بـ `string firstName;` العادي.
4. متى نستخدم: عند الحاجة لقيمة مشتركة ملازمة للصف ككل، مثل عداد الأغراض أو إعداد ثابت مشترك.

### سؤال 6: لماذا لا يستطيع تابع ساكن الوصول إلى حقل غير ساكن؟
**نموذج الإجابة:**
1. التعريف: التابع الساكن لا يرتبط بغرض محدد، فلا يملك مؤشر `this` يمكّنه من معرفة "أي غرض" يقصد.
2. المكونات/الشروط: الحقول غير الساكنة تحتاج غرضًا محددًا لمعرفة قيمتها (كل غرض له نسخته الخاصة).
3. مثال: تابع `static int getCount()` لا يستطيع الوصول لحقل `salary` غير الساكن مباشرة.
4. متى نستخدم: هذه قاعدة صارمة دائمة الصحة في C++، لا استثناء لها.

### سؤال 7: متى يُعرَّف التحميل الزائد لعملية كتابع صديق بدلًا من تابع عضو؟
**نموذج الإجابة:**
1. التعريف: عندما لا يكون الطرف الأيسر للعملية غرضًا من نفس الصف الذي نُحمّل عليه العملية.
2. المكونات/الشروط: التابع الصديق يُعلَن داخل الصف لكنه لا يستقبل `this` ضمنيًا؛ بدلًا من ذلك يستقبل كل الأطراف كوسائط صريحة.
3. مثال: `friend ostream& operator<<(ostream& out, Complex& c)` لأن الطرف الأيسر هو `ostream` وليس `Complex`.
4. متى نستخدم: أي عملية يكون طرفها الأيسر نوعًا خارجيًا (كـ `ostream`, `istream`) أو عند الرغبة بتناظر كامل بين الطرفين.

### سؤال 8: ما الفرق بين `new`/`delete` وإنشاء غرض عادي على المكدس؟
**نموذج الإجابة:**
1. التعريف: `new` تحجز الغرض على الكومة (Heap) وتُعيد مؤشرًا إليه، ويجب تحريره يدويًا بـ `delete`؛ الغرض العادي على المكدس (Stack) يُهدَم تلقائيًا عند خروجه من نطاقه.
2. المكونات/الشروط: كل `new` يقابله `delete` واحد بالضبط، وإلا يحدث تسرّب ذاكرة أو سلوك غير معرَّف (عند تكرار `delete`).
3. مثال: `Class1 *x = new Class1(); ... delete x;` مقارنة بـ `Class1 x;` التي تُهدَم تلقائيًا.
4. متى نستخدم: `new` عندما نحتاج عمر الغرض ممتدًا يتجاوز نطاقه المحلي، أو عند الحاجة لحجم/عدد أغراض يتحدد وقت التشغيل.

## الجزء السادس: قائمة فحص ذاتي قبل الامتحان ✅

- [ ] أستطيع شرح الفرق بين `const` قبل الغرض و`const` بعد توقيع التابع.
- [ ] أعرف متى يُسمح باستدعاء تابع على غرض `const` ومتى يُرفض.
- [ ] أستطيع تهيئة حقل `const` باستخدام قائمة التهيئة بشكل صحيح.
- [ ] أفهم لماذا لا يمكن إسناد قيمة لحقل `const` داخل جسم الباني.
- [ ] أعرف الفرق بين التابع الصديق والصف الصديق ومكان كتابة إعلانهما.
- [ ] أدرك أن الصداقة علاقة من طرف واحد، غير متعدية، وتُمنح لا تُؤخذ.
- [ ] أستطيع كتابة كود يستخدم `new` و`delete` بشكل صحيح دون تسرّب ذاكرة.
- [ ] أعرف أن `delete` يستدعي الهادم قبل تحرير الذاكرة.
- [ ] أفهم الفرق بين الحقل العادي والحقل `static` من حيث التخزين.
- [ ] أعرف أن التابع الساكن لا يملك `this` ولا يصل للحقول غير الساكنة.
- [ ] أستطيع تهيئة حقل `static` خارج الصف بالصيغة الصحيحة (بدون تكرار كلمة `static`).
- [ ] أفرّق بين متى أُحمّل عملية كتابع عضو ومتى أُحمّلها كتابع صديق.
- [ ] أعرف أن `operator<<` يجب أن يُعيد مرجعًا لـ `ostream&` للسماح بالتسلسل.
- [ ] راجعت جميع أسئلة MCQ والتصحيح وتمارين التتبع في هذا الدليل.

### 🔑 أهم النقاط الذهبية
| الموضوع | النقاط |
| --- | --- |
| `const` | قبل الغرض = الغرض بأكمله ثابت. بعد توقيع التابع = التابع لا يُعدّل الحالة. حقل `const` يُهيَّأ فقط عبر قائمة التهيئة. |
| `friend` | يُمنح لا يُؤخذ. غير تبادلي وغير متعدٍ. يُكتب داخل الصف، يُستدعى ككود عادي خارجه. |
| المؤشرات | كل `new` يقابله `delete` واحد بالضبط. `delete` يستدعي الهادم أولًا. |
| `static` | نسخة واحدة مشتركة بين كل الأغراض. لا يوجد `this` للتوابع الساكنة. تُهيَّأ الحقول خارج الصف مرة واحدة. |
| `operator overloading` | تابع عضو إذا كان الطرف الأيسر غرضًا من الصف؛ تابع صديق إذا لم يكن كذلك (مثل `<<`). |

### 🔑 مرجع سريع
| الرمز/المصطلح | المعنى | يُستخدم في |
| --- | --- | --- |
| `const` (قبل الغرض) | تجميد كل حقول الغرض | `const Time c(10,0,0);` |
| `const` (بعد التابع) | التابع لا يُعدّل الحالة | `int getHour() const;` |
| `:` بعد توقيع الباني | بداية قائمة التهيئة | `Increment(int i): increment(i)` |
| `friend` | إذن وصول لتابع/صف خارجي | `friend void setX(Count&, int);` |
| `new` / `delete` | حجز/تحرير ذاكرة على الكومة | `Class1 *x = new Class1(); delete x;` |
| `static` | مشاركة قيمة/تابع بين كل الأغراض | `static int count;` |
| `operator{symbol}` | إعادة تعريف سلوك عامل تشغيل | `Complex& operator+(Complex&);` |

### 🔑 قواعد ذهبية لا تُنسى
| # | القاعدة |
| --- | --- |
| 1 | لا يمكن استدعاء تابع غير `const` على غرض `const` مهما كان بسيطًا. |
| 2 | حقول `const` تُهيَّأ فقط عبر قائمة التهيئة — لا يوجد بديل. |
| 3 | الصداقة تُصرَّح صراحة في الصف المالك، وليست تلقائية أو تبادلية. |
| 4 | لكل `new` يجب `delete` واحد بالضبط — لا أكثر ولا أقل. |
| 5 | الحقول الساكنة تُهيَّأ خارج الصف مرة واحدة فقط بدون تكرار كلمة `static`. |
| 6 | التوابع الساكنة لا تصل لحقول غير ساكنة لعدم امتلاكها `this`. |
| 7 | `operator<<` يُكتب كتابع صديق لأن الطرف الأيسر (`ostream`) ليس غرضًا من صفك. |
